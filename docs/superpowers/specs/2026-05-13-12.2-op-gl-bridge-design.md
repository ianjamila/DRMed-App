# 12.2 — Operational → GL Bridge

**Status:** Approved 2026-05-13. Implementation plan pending.
**Parent project:** [`2026-05-11-accounting-project-overview.md`](./2026-05-11-accounting-project-overview.md)
**Depends on:** 12.1 (GL foundation), live on `main` and tagged `v1.2.0`.
**Migration ID:** `0030_op_gl_bridge.sql`
**Effort:** ~6 working days · 1.5 weeks calendar
**Smoke test artifact:** `supabase/tests/0030_op_gl_bridge_smoke.sql`

## Goal

Wire every operational write that has economic meaning to emit a balanced journal entry, automatically and atomically. After 12.2, the GL is a true mirror of operational reality — no application code needs to know about JEs; no Server Action can forget to post one; replay of historical data (12.A, 12.B) becomes "insert into operational table."

## Scope

**In:**

- Postgres triggers on `payments` (insert / update / delete) and `test_requests` (status flip)
- Bridge SQL functions that resolve CoA accounts, construct balanced JEs, and post them
- Idempotency via partial unique index on `journal_entries (source_kind, source_id)` for posted non-reversal entries
- Soft-void semantics for payments (`voided_at`, `voided_by`, `void_reason` columns)
- `voidPaymentAction` Server Action with gift-code reset
- Admin page `/staff/admin/accounting/payment-routing` for editing the payment-method → cash-account mapping
- Fallback chain to `9999 Suspense` for missing mappings or stale CoA lookups
- Inspection helper `bridge_replay_summary(start, end)` for batch verification
- Four new guard triggers (P0004 / P0005 / P0006 / P0007) — defense in depth
- New CoA seed row: `1090 Cash — HMO Settlements Pending` (clearing account)

**Out (handled by other sub-projects):**

- Bridge wiring for `expenses`, `doctor_payouts`, `payroll_runs`, `doctor_rent_invoices` (each future phase ships its own bridge function in the same pattern)
- Staging tables and import UI (12.A and 12.B own these)
- Per-row SAVEPOINT pattern for dry-run resilience (12.A pattern; documented here but not implemented)
- True refund primitive — refunds operationally treated as voids; cash-back refunds via manual JE in 12.7
- Visit-level void (would touch multiple payments + test_requests — out of 12.2 scope)

## Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│                  Operational writes (today)                        │
│                                                                    │
│   payments  INSERT / UPDATE / DELETE                               │
│   test_requests  UPDATE (status transitions)                       │
└──────────────────────────────┬─────────────────────────────────────┘
                               │  (Postgres triggers, in-transaction)
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                  Bridge layer (12.2 — NEW)                         │
│                                                                    │
│   One SECURITY DEFINER function per event kind:                    │
│     bridge_payment_insert()                                        │
│     bridge_payment_void()                                          │
│     bridge_payment_delete()                                        │
│     bridge_test_request_released()                                 │
│     bridge_test_request_cancelled()                                │
│                                                                    │
│   Each function:                                                   │
│     1. Reads context (visit's HMO status, service kind, etc.)      │
│     2. Resolves CoA accounts via resolve_*() helpers               │
│     3. Constructs balanced JE                                      │
│     4. INSERTs journal_entries + journal_lines                     │
│     5. Drops zero-amount lines for cleanliness                     │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                  GL spine (12.1 — existing)                        │
│                                                                    │
│   journal_entries + journal_lines                                  │
│     • P0001 balance trigger enforces debits = credits              │
│     • P0002 period-lock trigger blocks closed periods              │
│     • P0003 zero-line guard                                        │
│     • partial unique index on (source_kind, source_id)             │
│                                                                    │
│   New guard triggers (12.2):                                       │
│     • P0004 — block payment edits after JE                         │
│     • P0005 — block postings to inactive accounts                  │
│     • P0006 — block physical delete of CoA rows                    │
│     • P0007 — block un-voiding payments                            │
└────────────────────────────────────────────────────────────────────┘
```

**Key properties:**

- **One operational event → exactly one JE.** Partial unique index enforces this DB-side. Trigger functions ride on top with idempotency checks for the trigger-fires-twice edge.
- **Triggers fire inside the operational transaction.** JE failure rolls back the operational write atomically. Reception never sees a half-committed state.
- **No application code knows about JEs.** Server Actions don't change. Bridge is purely DB-side.
- **Replay = INSERT into operational table.** 12.A's HMO history import seeds historical rows; the bridge constructs backdated JEs from the row's date columns automatically.

### `journal_entries` — one new partial unique index

```sql
create unique index journal_entries_one_posted_per_source
  on public.journal_entries (source_kind, source_id)
  where status = 'posted'
    and source_kind != 'reversal'
    and source_id is not null;
```

### New CoA seed row: `1090 Cash — HMO Settlements Pending`

The `payments.method = 'hmo'` value records "this payment was an HMO settlement" but doesn't carry which bank received the wire. Without per-bank routing on the operational side (would require reception workflow change), the cleanest accounting is a **clearing account**: HMO settlements post here on receipt; admin moves the balance to BPI/BDO via a manual JE in 12.7 once the bank deposit is reconciled.

```sql
insert into public.chart_of_accounts (code, name, type, normal_balance, description)
values (
  '1090',
  'Cash — HMO Settlements Pending',
  'asset',
  'debit',
  'Clearing account for method=hmo payments. Admin reclassifies to BPI/BDO via manual JE in 12.7 once bank deposit identified.'
);
```

## Helper SQL functions

All `language sql stable security definer set search_path = public`:

- `resolve_cash_account(p_method text) returns uuid` — lookup in `payment_method_account_map`. Fallback: returns the uuid of `9999 Suspense`.
- `resolve_revenue_account(p_service_kind text) returns uuid` — CASE WHEN: `lab` → `4100`, `doctor_consult` → `4200`, `doctor_procedure` → `4500`. Fallback: Suspense.
- `resolve_discount_account(p_service_kind text) returns uuid` — `lab` → `4910`, anything else → `4920`. Fallback: Suspense.
- `resolve_ar_account(p_is_hmo boolean) returns uuid` — `true` → `1110`, `false` → `1100`.
- `coa_uuid_for_code(p_code text) returns uuid` — strict by-code lookup. Raises if not found. Used internally by the `resolve_*` helpers as the final lookup step.

## Bridge trigger functions

### `bridge_payment_insert()` — `AFTER INSERT ON payments`

1. Read `NEW.visit_id` → look up `visits.hmo_provider_id` to determine `v_is_hmo`.
2. Idempotency check: `select 1 from journal_entries where source_kind = 'payment' and source_id = NEW.id and status = 'posted' for update`. If exists, return silently.
3. Resolve `v_cash_account_id` via `resolve_cash_account(NEW.method)`.
4. Resolve `v_ar_account_id` via `resolve_ar_account(v_is_hmo)`.
5. If either lookup hit Suspense fallback, write `audit_log` with action `coa.suspense_post`.
6. Insert `journal_entries` header:
   - `posting_date = NEW.received_at::date`
   - `source_kind = 'payment'`, `source_id = NEW.id`
   - `status = 'posted'`, `created_by = NEW.received_by`
   - Description = `'Payment received via ' || NEW.method`.
7. Insert two `journal_lines`:
   - DR `v_cash_account_id`, `NEW.amount_php`
   - CR `v_ar_account_id`, `NEW.amount_php`
8. Return NEW.

### `bridge_payment_void()` — `AFTER UPDATE ON payments WHEN OLD.voided_at IS NULL AND NEW.voided_at IS NOT NULL`

1. Find original JE: `select id from journal_entries where source_kind = 'payment' and source_id = NEW.id and status = 'posted' for update`.
2. If no such JE (defensive — shouldn't happen since insert always posts one), audit and return silently.
3. Read original JE's lines.
4. Update original JE: `status = 'reversed'`.
5. Insert reversal JE header:
   - `posting_date = NEW.voided_at::date`
   - `source_kind = 'reversal'`, `source_id = NULL`, `reverses = original_je_id`
   - `status = 'posted'`, `created_by = NEW.voided_by`
   - Description = `'Reversal of ' || original_entry_number || ': ' || NEW.void_reason`.
6. Update original JE: `reversed_by = new_reversal_je_id`.
7. Insert mirrored journal_lines (debits ↔ credits).
8. Return NEW.

### `bridge_payment_delete()` — `BEFORE DELETE ON payments`

Same logic as `bridge_payment_void()` except:

- Triggered by DELETE, not UPDATE.
- `posting_date = now()::date` (no `voided_at` available since the row is about to vanish).
- `created_by = NULL` (no actor available in DELETE context; caller should write audit_log separately if appropriate).
- Description = `'Reversal of ' || original_entry_number || ': payment row deleted'`.

This handles the existing gift-code-redemption rollback path and any future hard-delete scenarios defensively.

### `bridge_test_request_released()` — `AFTER UPDATE ON test_requests WHEN OLD.status != 'released' AND NEW.status = 'released'`

1. Read `NEW.visit_id` → `visits.hmo_provider_id` → `v_is_hmo`.
2. Read `NEW.service_id` → `services.kind` → `v_service_kind`.
3. Idempotency check (as in payment insert).
4. Compute amounts:
   - `v_base = NEW.base_price_php`
   - `v_discount = NEW.discount_amount_php`
   - `v_final = NEW.final_price_php`
   - `v_hmo_approved = COALESCE(NEW.hmo_approved_amount_php, 0)` (only relevant when `v_is_hmo`)
   - `v_patient_share = v_final - v_hmo_approved`
5. Resolve accounts: `v_revenue_id`, `v_discount_id`, `v_ar_hmo_id` (if HMO), `v_ar_patient_id`.
6. Insert JE header with `posting_date = NEW.released_at::date`, `source_kind = 'test_request'`, `source_id = NEW.id`, `created_by = NEW.released_by`.
7. Insert lines (drop zero-amount lines):
   - DR `v_ar_hmo_id`, `v_hmo_approved` (if > 0)
   - DR `v_ar_patient_id`, `v_patient_share` (if > 0; for non-HMO this equals `v_final`)
   - DR `v_discount_id`, `v_discount` (if > 0)
   - CR `v_revenue_id`, `v_base`
8. The balance trigger (P0001) validates the JE. By construction it balances:
   `hmo_approved + patient_share + discount = hmo_approved + (final - hmo_approved) + (base - final) = base`.
9. Return NEW.

### `bridge_test_request_cancelled()` — `AFTER UPDATE ON test_requests WHEN OLD.status = 'released' AND NEW.status = 'cancelled'`

Same flip-and-reverse pattern as `bridge_payment_void()`:

1. Find original release JE.
2. Flip its `status` to `'reversed'`.
3. Post reversal JE with mirrored lines; `posting_date = now()::date` (today's open period, not the original); `created_by = NEW.released_by` (preserved on the test_request row even after cancel).

## Guard triggers

| Function | Fires | Raises |
|---|---|---|
| `payments_block_post_je_edits()` | BEFORE UPDATE on payments | P0004 when `amount_php`, `method`, `visit_id`, or `received_at` changes on a row with a posted JE. `voided_at`/`voided_by`/`void_reason` updates are exempt. |
| `payments_block_unvoid()` | BEFORE UPDATE on payments | P0007 when `OLD.voided_at IS NOT NULL AND NEW.voided_at IS NULL`. |
| `journal_lines_block_inactive_account()` | BEFORE INSERT on journal_lines | P0005 when `account_id` resolves to a CoA row with `is_active = false`. |
| `chart_of_accounts_block_delete()` | BEFORE DELETE on chart_of_accounts | P0006 unconditionally. CoA is append-only; soft-disable via `is_active = false`. |

`src/lib/accounting/pg-errors.ts` extended:

```ts
case "P0004": return err.message ?? "Cannot edit payment after JE has posted.";
case "P0005": return err.message ?? "Cannot post to inactive account.";
case "P0006": return err.message ?? "Accounts are append-only; deactivate instead.";
case "P0007": return err.message ?? "Cannot un-void a payment.";
```

## Server Actions

### `voidPaymentAction(paymentId, reason)`

New file: `src/app/(staff)/staff/(dashboard)/payments/[id]/void/actions.ts`.

```ts
"use server";

export type VoidResult = { ok: true } | { ok: false; error: string };

export async function voidPaymentAction(
  paymentId: string,
  reason: string,
): Promise<VoidResult> {
  const session = await requireActiveStaff();
  const parsed = VoidPaymentSchema.safeParse({ reason });
  if (!parsed.success) {
    return { ok: false, error: parsed.error.issues[0]?.message ?? "Reason required." };
  }

  const admin = createAdminClient();

  // 1. Read payment + check not already voided.
  const { data: payment, error: readErr } = await admin
    .from("payments")
    .select("id, voided_at, redeemed_payment_id:gift_codes!gift_codes_redeemed_payment_id_fkey(id, status)")
    .eq("id", paymentId)
    .maybeSingle();
  if (readErr || !payment) return { ok: false, error: "Payment not found." };
  if (payment.voided_at) return { ok: false, error: "Payment already voided." };

  // 2. Reset gift code if one was redeemed against this payment.
  const giftCode = payment.redeemed_payment_id?.[0];
  if (giftCode && giftCode.status === "redeemed") {
    const { error: gcErr } = await admin
      .from("gift_codes")
      .update({
        status: "purchased",
        redeemed_at: null,
        redeemed_by: null,
        redeemed_visit_id: null,
        redeemed_payment_id: null,
      })
      .eq("id", giftCode.id);
    if (gcErr) return { ok: false, error: translatePgError(gcErr) };
  }

  // 3. Flip voided_at — trigger fires reversal JE.
  const { error: voidErr } = await admin
    .from("payments")
    .update({
      voided_at: new Date().toISOString(),
      voided_by: session.user_id,
      void_reason: parsed.data.reason,
    })
    .eq("id", paymentId);
  if (voidErr) return { ok: false, error: translatePgError(voidErr) };

  // 4. Audit log.
  const h = await headers();
  await audit({
    actor_id: session.user_id,
    actor_type: "staff",
    action: "payment.voided",
    resource_type: "payment",
    resource_id: paymentId,
    metadata: { reason: parsed.data.reason },
    ip_address: h.get("x-forwarded-for")?.split(",")[0]?.trim() ?? null,
    user_agent: h.get("user-agent"),
  });

  revalidatePath(`/staff/visits`);
  return { ok: true };
}
```

## Inspection helper

### `bridge_replay_summary(p_start timestamptz, p_end timestamptz) returns jsonb`

Returns a single JSON document with:

- `window: { start, end }`
- `je_count`: posted JE count in window
- `suspense_postings`: array of `{ entry_number, source_kind, source_id, amount }` for any JEs that hit account `9999`
- `totals_by_account`: object mapping CoA code → `{ debit, credit }` sums in window
- `unbalanced_count`: defensive — should always be 0

Used by 12.A's import code, admin diagnostic queries, and the smoke test. See Section 3 of brainstorm for full SQL.

## Audit logging

| Bridge event | Audit action | Metadata |
|---|---|---|
| Any JE post with Suspense fallback | `coa.suspense_post` | `{ source_kind, source_id, reason, attempted_lookup }` |
| Payment voided | `payment.voided` (from Server Action) | `{ payment_id, reason }` |
| Bridge JE posted (general) | None — JE creation is the audit trail itself; metadata via `journal_entries.source_*` |
| Visit credit balance detected (AR-Patient < 0 after cancel) | `accounting.credit_balance_detected` | `{ visit_id, balance_php }` |

`accounting.credit_balance_detected` is fired by `bridge_test_request_cancelled` when, post-cancel, the visit's net AR-Patient goes negative.

## Edge cases (full table)

| # | Case | Handling |
|---|---|---|
| 1 | Payment for HMO claim (method='hmo') | Posts to `1090 Cash — HMO Settlements Pending` clearing account |
| 2 | Cash patient, no discount, no HMO | 2-line JE: DR Cash / CR AR-Patient (insert) + DR AR-Patient / CR Revenue (release) |
| 3 | Lab test with senior discount | 3-line release JE: DR AR-Patient + DR Discount / CR Revenue |
| 4 | HMO partial-approval with discount | 4-line release JE (see Section 4F in brainstorm) |
| 5 | Test released, never paid → patient owes | AR-Patient stays positive on the GL; visit shows unpaid |
| 6 | Test released, paid, then cancelled | Revenue de-recognized; cash stays in; visit shows credit balance → `accounting.credit_balance_detected` audit row |
| 7 | Payment voided, gift code was redeemed | `voidPaymentAction` resets gift_code to `purchased` before flipping `voided_at`; reversal JE fires automatically |
| 8 | Re-release after cancel | Original JE flipped to `reversed`; new release JE posts cleanly (partial index allows) |
| 9 | Out-of-order: payment before release | Both JEs post correctly; AR-Patient transiently negative, returns to zero on release |
| 10 | HMO claim ultimately rejected by HMO | Not a bridge event. Manual JE in 12.7: DR AR-Patient / CR AR-HMO |
| 11 | Cash refund to patient (overpayment) | Not a bridge event. Manual JE in 12.7: DR Refund-Liability / CR Cash |
| 12 | Admin deactivates a CoA account in use | New postings raise P0005; existing JEs unaffected; admin must reclassify or reactivate |
| 13 | Admin tries to DELETE a CoA row | P0006 raised; no rows deleted |
| 14 | Stale payment_method mapping (admin deletes a row) | Cash side posts to Suspense; audit log fires |
| 15 | Future quarter has no period rows seeded | Period_status_for returns 'unknown'; period_lock allows (consistent with 12.1) |
| 16 | Trigger fires twice on same event (rare retry) | Idempotency check inside trigger function (`select 1 from journal_entries... for update`) returns silently |

## Hard invariants (from this design)

1. **Every operational write that has economic meaning emits exactly one balanced JE**, enforced by trigger + partial unique index.
2. **No path bypasses the period lock**, including reversals (which post to today's date, not the original).
3. **No path posts to an inactive CoA account.** Admin's intent to retire an account is visible at the trigger level.
4. **No path physically deletes a CoA row.** Append-only is now a database property, not just a documented intent.
5. **Voiding is reversible only by creating a new payment.** Un-voiding doesn't exist as a primitive.
