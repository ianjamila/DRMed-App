**Date:** 2026-06-10
**Status:** Approved 2026-06-10 — ready for implementation plan
**Context:** B1.4 — the final piece of the operational-analytics dashboard pack
([[project-ops-analytics-dashboard]]). Follows **B1.1** (operational daily
report, PR #47/#48), **B1.2** (cash collected + credit card, PR #49), and
**B1.3** (expenses + net income + cash flow, PR #50). Parent spec:

## Goal

Reproduce the clinic's manual **`DAILY MONITORING`** sheet section
**"HMO LAB RECEIVABLES" (rows 92–124)**: a per-provider accounts-receivable
roll-forward — **IN (billed) → OUT (paid) → ENDING BALANCE** — for each HMO
provider, plus a grand **HMO RECEIVABLES BALANCE** (row 124). Rendered as a new
**"HMO receivables"** tab under `Admin → Operations`, in the same
collapsible-month idiom as B1.1–B1.3.

This is the AR side of the daily report. B1.2 already booked the single
aggregate **"HMO received"** line (cash-collected tab); B1.4 breaks HMO out
**per provider** and adds the **billed (IN)** and **running ending-balance**
dimensions the aggregate line lacks.

## The sheet block (dumped from `DAILY MONITORING`)

```
r92–101  HMO LAB RECEIVABLES | IN (FOR BILLING)  | <provider>   (10 rows)
r102     HMO LAB RECEIVABLES | TOTAL IN (Billed / For billing)
r103–112 HMO LAB RECEIVABLES | OUT (PAID)        | <provider>   (10 rows)
r113     HMO LAB RECEIVABLES | TOTAL OUT (Paid)
r114–123 HMO LAB RECEIVABLES | ENDING BALANCE    | <provider>   (10 rows)
r124     HMO RECEIVABLES BALANCE                  (grand-total ending balance)
```

Providers in the sheet (10): **Maxicare, Valucare, Cocolife, Med Asia,
Intellicare, Avega, Generali, Etiqa, iCare, Amaphil**. All ten appear verbatim
(title-cased) in the DB; the data also carries an **`(unknown HMO)`** bucket
which we render as an 11th row.

The sheet's *daily* HMO cells are **sparsely maintained** (most day columns are
blank); the rightmost column is a running **summary** carrying cumulative
ending balances (e.g. END Intellicare ₱583,453; HMO BAL ₱1,895,216.40). So the
sheet is **not** a per-day golden target for HMO — the DB roll-forward is the
source of truth, and the summary column is a **cumulative cross-check** (see
Reconciliation).

## The validated roll-forward model

Reverse-engineered against prod and tied to the sheet to the peso for several
providers. Per provider:

- **IN (billed)** — every billed claim, added to AR at its claim/service date.
- **OUT (paid)** — claims marked paid **that carry a settlement date**, drawn
  down at that date. **Paid claims with no settlement date stay in the ending
  balance** — exactly what the bookkeeper did (a payment with no date was never
  drawn down).
- **ENDING BALANCE** — cumulative `Σ IN − Σ OUT` from the beginning of time
  through the period end. Runs forward; the first visible month already carries
  the correct opening balance.

**Proof it ties (prod, lab claims):**

| Provider | DB roll-forward ending | Sheet col-869 ending |
|---|---|---|
| Intellicare | 1,156,588 − 573,135 = **583,453** | 583,453 ✓ |
| Maxicare | 929,148 − 520,540 = **408,608** | 408,608 ✓ |
| iCare | 153,988 − 40,360 = **113,628** | 113,628 ✓ |
| Amaphil | 49,247.20 − 42,114 = **7,133.20** | 7,133.20 ✓ |
| Avega | 180,954 − 136,522 = **44,432** | 44,432 ✓ |

Grand total DB ≈ **₱1.89–1.90M** vs sheet **₱1,895,216.40** (≤0.3% manual
noise). Intellicare's gap from GL-AR (5,418) is exactly its undated-paid claims
— confirming the "undated-paid stays in AR" rule.

## Source of record — sheet measure → DB

Lab HMO only (`source_tab = 'LAB SERVICE'` historic; `services.kind` lab-family
live), unioning the live 12.3 subledger with the 12.B historic import — the
same `live ∪ historic` pattern as B1.2's `v_ops_daily_hmo_received` and the
0078/0079 historic-inclusive views.

| Sheet measure | Live (12.3 subledger) | Historic (12.B import) |
|---|---|---|
| **IN (billed)**, by date | `Σ hmo_claim_items.billed_amount_php` by `(tr.released_at at tz 'Asia/Manila')::date` (batch not voided, lab-kind only) | `Σ final_amount_php` by `claim_date`, **all statuses** |
| **OUT (paid)**, by date | `Σ hmo_claim_items.paid_amount_php` where `hmo_response='paid'` by `hmo_response_date` (batch not voided, lab-kind only) | `Σ final_amount_php` where `status='paid' AND date_paid IS NOT NULL` by `date_paid` |

The live arm filters to lab-family services
(`services.kind in ('lab_test','lab_package','vaccine','home_service')` via
`test_requests → services`) to match the historic `source_tab='LAB SERVICE'`
scope. `claim_date` / `date_paid` / `hmo_response_date` are `date` columns (no
tz); only the live `released_at` (timestamptz) is Manila-localized — same
convention as B1.1's `business_date`.

Provider key:
- Historic arm groups by **`historic_hmo_claims.hmo_provider`** directly (no
  join) so the free-text **`(unknown HMO)`** bucket is preserved.
- Live arm groups by **`hmo_providers.name`** via `batch.provider_id`.
- Names are title-cased consistently on import, so the two arms union cleanly on
  the `provider_name` string. **No `is_active` gate** — a deactivated provider
  can still carry historic AR.

**Known limitation (documented, not faked):** historic OUT = the *billed*
amount of paid claims (HMO short-pays / partial settlements were not recorded in
the import). Live claims use true `paid_amount_php`, so short-pays settle
correctly going forward. Historic write-offs/patient-rebills are likewise not
modelled (they weren't tracked), so the roll-forward ending = billed − dated-paid,
which is precisely the sheet's definition.

## Data layer — one new view + reuse the aging view

### New view — `v_ops_daily_hmo_provider_ar` (migration `0097`)

`security_invoker = on`, **no** grant to `anon`/`authenticated` (admin reads via
the page's `requireAdminStaff` gate + RLS on base tables). Mirrors the 0093–0096
recipe. Emits one row per `(business_date, provider_name, source)`:

| Column | Type | Meaning |
|---|---|---|
| `business_date` | `date` | claim/service date (IN) or settlement date (OUT) |
| `provider_name` | `text` | title-cased provider, or `(unknown HMO)` |
| `source` | `text` | `'live'` \| `'historic'` |
| `billed_in_php` | `numeric(14,2)` | money added to AR that day |
| `paid_out_php` | `numeric(14,2)` | money collected that day |

Built as a `union all` of four `select`s (live-IN, live-OUT, historic-IN,
historic-OUT), each emitting one measure with the other as 0, then
`group by business_date, provider_name, source`. **No ending balance in the
view** — cumulative running balance is a UI/range concern, computed in the pure
core (a view can't cleanly express "all-prior cumulative through the visible
window").

The new view's `Row` type is **surgically inserted** into
`src/types/database.ts` (don't wholesale-regen — avoids pulling the known 0035
drift, per B1.1–B1.3).

### Reuse — `v_hmo_ar_aging` (migration 0079, historic-inclusive)

Used **as-is** for the current-AR aging panel. Columns: `provider_id`,
`provider_name`, `bucket` (one of `0-30` / `31-60` / `61-90` / `91-180` /
`180+`), `total_php`, `item_count`.

**Lab-vs-consult reconciliation (important):** `v_hmo_ar_aging`'s historic arm
includes **both** lab and consult claims (`status in ('pending','overdue')`,
amount > 0) — it does **not** filter to lab. So the aging panel's grand total
runs ~₱194k higher than the lab-only roll-forward TOTAL (the consult-HMO
overdue: Maxicare ₱124k, Cocolife ₱52.8k, Etiqa ₱10.1k, Generali ₱7.5k, Avega
₱4.6k, Valucare/Med Asia/others). Rather than rebuild the view or drop the
panel, the page shows an explicit reconciliation line:

> **Lab AR (roll-forward) ₱X + Consult HMO AR ₱Y = Total HMO AR ₱Z** (aging total)

The page loader computes the consult-AR scalar `Y` directly
(`Σ historic_hmo_claims.final_amount_php where source_tab='DOCTOR CONSULTATION'
and status in ('pending','overdue')`, plus any live consult AR). This turns the
view's lab+consult scope into a *feature* — it surfaces the otherwise-deferred
consult AR as a read-only snapshot — while keeping the roll-forward lab-clean
and tied to the sheet.

## Pure core — `src/lib/operations/hmo-ar-report.ts` (+ vitest)

No `import "server-only"`; fully unit-tested. Reuses `num`, `enumerateDays`,
`groupDaysByMonth` from B1.1–B1.3 (`daily-report.ts`).

- **`buildHmoArMatrix(rows, range)`** — collapses `v_ops_daily_hmo_provider_ar`
  rows (live + historic summed) into one row per provider; for each month/day
  column, `billedIn`, `paidOut`, and `endingBalance` = **cumulative
  (all-prior + this-period) billed − paid**. Critically, the cumulation must
  include movement **before** the visible range so the opening month carries the
  correct opening balance — the full row stream is folded chronologically, then
  windowed to the requested range for display. Providers sorted by **current
  ending balance desc**; `(unknown HMO)` forced last. Plus a `TOTAL` row
  (Σ providers per column) — its final ending ≈ ₱1.895M today.
- **`summarizeAging(agingRows)`** — pivots `v_hmo_ar_aging` rows into per-provider
  bucket columns (`0-30`…`180+`) + a per-provider total + a grand total.

The cumulative ending-balance logic is the only piece with real logic, so it
carries the bulk of the tests: opening-balance carry into a windowed range,
undated-paid staying in AR, multi-provider isolation, empty/zero months,
`(unknown HMO)` ordering, single-month auto-expand.

## Reconciliation plan (views-first, before UI)

`scripts/ops-daily/validate-hmo.sql`, run via the Supabase MCP on **prod**
before building UI (project `qhptbmafrosgibooelpp`):

1. **Per-provider all-time ending** (cumulative IN − dated-OUT) == sheet col-869
   endings — exact for Intellicare 583,453 / Maxicare 408,608 / iCare 113,628 /
   Amaphil 7,133.20 / Avega 44,432; **TOTAL ≈ ₱1,895,216** (≤0.3% manual noise).
2. **Documented variance to `v_hmo_provider_summary.total_unresolved_ar_php`**
   per provider = (undated-paid) + (unknown-status amounts) — list the deltas so
   "why doesn't this equal GL-AR?" is answered in writing.
3. **Aging buckets sum == aging grand total**, and **lab roll-forward TOTAL +
   consult-AR scalar == aging grand total** (the page's reconciliation line).
4. Spot-check a populated historic day (e.g. an early Intellicare/Maxicare lab
   claim) flows IN at `claim_date` and OUT at `date_paid`.

## Scope

**In:**
- Lab HMO receivables only (matches the sheet, ties to ₱1.895M).
- 10 live providers + `(unknown HMO)`, rendered **dynamically** (any provider
  with lab-claim activity), not a hardcoded list.
- `live ∪ historic` union; per-provider IN / OUT / cumulative ENDING.
- Current-AR aging panel (reuse `v_hmo_ar_aging`) with lab/consult reconciliation.
- CSV (3-block sheet shape).

**Out (follow-on suggestions, deferred):**
- **Consult HMO AR roll-forward** (~₱194k real overdue) — surfaced read-only in
  the aging panel here; a full per-provider consult roll-forward is its own pass.
- **Drill-down** provider/month → underlying claims list.
- **Trends** mini-chart of HMO AR balance over time (B2).
- **HMO short-pay / write-down** modelling for historic claims (data not captured).

## Non-goals / invariants

- **Read-layer only — no schema change** beyond the one `security_invoker` view.
  No writes, no triggers, no GL impact.
- Does **not** touch the live 12.3 subledger tables, the existing `v_hmo_*`
  snapshot views (reused, not modified), or B1.2's `v_ops_daily_hmo_received`.
- Admin-only (`requireAdminStaff`); no anon/authenticated grant on the new view.
