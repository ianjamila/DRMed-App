## 1. Problem

The v1.15.0 clinical backfill committed only **1,605** of the **7,996** rows on
the `DOCTOR CONSULTATION` tab. The gap is **not** ambiguous names (those are a
smaller, separately-held set). It is the consult **postability rule**: a consult
is "postable" only when `clinic_fee > 0` (`classifyRow` in
`scripts/clinical-backfill/lib/classify.ts`). **6,316** consult rows have
`clinic_fee = 0` and were dropped as `zero_amount`.

Those zero-clinic-fee rows are **by design**, not data errors. The app models how
each doctor is paid via `physicians.compensation_arrangement ∈ {pf_split,
rent_paying, shareholder}` (set in **Staff → Admin → Physicians → "Compensation
arrangement"**; `src/lib/visits/consultation-fee.ts` returns clinic_fee `0` for
`rent_paying`/`shareholder`). The dropped consults belong **exactly** to the
clinic's `rent_paying` / `shareholder` doctors, who keep 100% of the consult — the
clinic earns ₱0:

| Doctor (col-8 surname) | Arrangement | consult rows | clinic_fee = 0 |
|---|---|---|---|
| Dr. Katherine Gayo | shareholder | 2,269 | 2,266 |
| Dr. Robert Vicencio (R.VICENCIO) | shareholder | 2,267 | 2,264 |
| Dr. Angelica Lorenzo | rent_paying | 838 | 837 |
| Dr. Aurora Vicencio (A. VICENCIO) | shareholder | 282 | 282 |
| Dr. Alain Arcega | rent_paying | 156 | 156 |
| *(all pf_split doctors: Manuel, Brojas, Elleazar, Anglo, Antonio…)* | pf_split | — | well covered |

The match is exact: every clinic_fee=0 doctor is `shareholder`/`rent_paying`;
every well-covered doctor is `pf_split`.

The backfill **correctly** excluded these from the **books** (₱0 clinic revenue to
record — migrations 0064/0091 already note "shareholder doctors have
clinic_fee=0"). But the dashboards need the **operational record**: per-doctor
consult **counts** for the busiest doctors (Gayo + the two Vicencios are the
highest-volume in the clinic). Right now Part B's Doctor/Specialty pack would show
them with ~3 historical consults each — badly wrong.

## 2. Goal

Recover the zero-clinic-fee consults of `rent_paying` / `shareholder` doctors into
the **operational layer** (`visits` + `test_requests`), GL-silent, so per-doctor
productivity over history is complete — **without** changing the books and
**without** loosening the rule for genuinely empty/error rows.

## 3. Non-goals / out of scope

- **No GL / books change.** These rows post **zero** journal entries (legacy
  rows, guarded by 0091). The clinic genuinely earned ₱0; nothing to book.
- **Not pf_split zero rows.** A `pf_split` consult with `clinic_fee = 0` is a
  genuine zero/error (missing data) and stays excluded — only `rent_paying` /
  `shareholder` zeros are expected.
- **Not lab.** Lab postability (`final > 0 || base > 0`) is unchanged.
- **No discount-amount fix** for consults (the v1.15.0 PF-as-discount artifact is
  a separate concern).

## 4. Approach — arrangement-aware postability

A consult row with `clinic_fee = 0` becomes **postable** iff its attending doctor
(col 8 surname → physician) has `compensation_arrangement ∈ {rent_paying,
shareholder}`. Mechanics:

1. **`classify.ts`** — `classifyRow` gains a fourth arg `doctorKeepsFullFee:
   boolean` (default `false`). New consult branch: if `isConsult` and
   `clinic_fee <= 0`, return `"postable"` **only when** `doctorKeepsFullFee`,
   else `"zero_amount"` (today's behavior). `clinic_fee > 0` stays `"postable"`
   as before. Lab and the date/`bad_date` checks are untouched.
2. **`engine.ts`** — before the row loop, fetch
   `physicians(full_name, compensation_arrangement)` and build a set of
   **surnames** whose doctor keeps the full fee, by reusing
   `scripts/clinical-enrich/lib/physician-map.ts` `resolveSurname()` over the
   roster (full_name → arrangement). In the loop, compute
   `doctorKeepsFullFee = isConsult && keepsFullFeeSurnames.has(normSurname(r.service))`
   and pass it to `classifyRow`. (`r.service` is col 8 = the surname for consults.)
3. **Build rules unchanged.** These visits import with `collected = 0` (clinic
   earned nothing — same path as the already-committed HMO zero-collected
   visits), `doctor_pf_php` recorded from col 17, `status = released`, GL-silent.
   Idempotent on the existing visit/test refs, so the 1,605 already-committed
   consults are **skipped** (`vSkip`), only the new ~6,316 import.

## 5. Safety & flow (same discipline as v1.15.0)

Dry-run → review CSVs → operator-gated `--commit --prod`. The dry-run must report,
for the consult tab:
- postable visits / test lines (now including the shareholder/rent consults),
- **new** patients vs reused vs **ambiguous (held)** — quantifies patient-table
  growth + the manual-resolution batch before any commitment,
- per-doctor postable consult counts (should now ≈ the sheet col-8 counts for the
  five shareholder/rent doctors).

After commit: re-run **`enrich:clinical`** (idempotent, fill-NULL-only) to attach
`attending_physician_id` to the newly-committed consults.

**Verification (read-only):**

## 7. Part B implication

Per-doctor **counts** become accurate. For shareholder/rent doctors, clinic
**Sales / clinic-fee revenue = ₱0** is correct (the clinic earns nothing on their
consults) — Part B must present per-doctor *productivity* (counts, and PF that
went to the doctor) distinctly from *clinic revenue*, so a high-volume shareholder
doctor doesn't read as "₱0 / broken."
