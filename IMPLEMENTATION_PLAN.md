# drmed.ph — Implementation Plan

A unified Next.js + Supabase application for a Philippine medical laboratory: marketing site, patient portal, and staff portal. This plan is structured as sequential phases. Complete each phase fully before moving to the next. After every phase, run the verification checklist at the bottom of that phase before proceeding.

---

## Project Context

**Domain:** drmed.ph
**Compliance:** Philippine Data Privacy Act (RA 10173)
**Locale:** Philippines (en-PH, Asia/Manila timezone, PHP currency)
**Deployment:** One Vercel project, one Supabase project

### Three surfaces, one app

1. **Marketing site** (`/`) — public, SEO-optimized, services and schedule info
2. **Patient portal** (`/portal/*`) — patients view and download released results
3. **Staff portal** (`/staff/*`) — reception, medtechs, and admins manage operations

### Operational flow (the actual lab workflow)

```
Patient walks in
  → Reception creates patient + visit (issues DRM-ID + receipt PIN)
  → Reception requests test(s) [test_request.status = requested]
  → Medtech sees queue, claims test [in_progress]
  → Medtech uploads PDF result [result_uploaded]
  → (Optional) Pathologist signs off [ready_for_release]
  → Patient pays at reception [visit.payment_status = paid]
  → Reception releases result [released] — gated by payment
  → Patient receives SMS/email notification
  → Patient logs into portal with DRM-ID + PIN, downloads PDF
```

## Phase 1 — Database Schema & RLS

**Goal:** Complete database schema with row-level security policies. No app code yet.

Create as a Supabase migration: `supabase/migrations/0001_init.sql`.

### Tables

**`patients`**
- `id` uuid PK default `gen_random_uuid()`
- `drm_id` text unique not null — public ID, e.g. `DRM-0001` (auto-generated by sequence)
- `first_name`, `last_name`, `middle_name` text
- `birthdate` date not null
- `sex` text check in (`male`, `female`)
- `phone` text — PH format, validated app-side (`+639XXXXXXXXX`)
- `email` text
- `address` text
- `created_at`, `updated_at` timestamptz default `now()`
- `created_by` uuid references `auth.users(id)`

**`staff_profiles`**
- `id` uuid PK references `auth.users(id)` on delete cascade
- `full_name` text not null
- `role` text not null check in (`reception`, `medtech`, `pathologist`, `admin`)
- `is_active` boolean default true
- `created_at`, `updated_at` timestamptz

**`services`** — catalog of tests/services offered
- `id` uuid PK
- `code` text unique — e.g. `CHEM`, `CBC`, `URINALYSIS`
- `name` text — display name
- `description` text
- `price_php` numeric(10,2) not null
- `is_active` boolean default true
- `turnaround_hours` int — typical hours to result

**`visits`**
- `id` uuid PK
- `patient_id` uuid references `patients(id)` not null
- `visit_number` text unique not null — auto-generated, e.g. `1234`
- `visit_date` date default `current_date`
- `payment_status` text default `unpaid` check in (`unpaid`, `partial`, `paid`, `waived`)
- `total_php` numeric(10,2) default 0
- `paid_php` numeric(10,2) default 0
- `notes` text
- `created_at`, `updated_at` timestamptz
- `created_by` uuid references `auth.users(id)`

**`visit_pins`** — receipt PIN for patient portal access
- `id` uuid PK
- `visit_id` uuid references `visits(id)` unique not null
- `pin_hash` text not null — bcrypt hash of the PIN
- `expires_at` timestamptz not null — default `now() + interval '60 days'`
- `last_used_at` timestamptz
- `failed_attempts` int default 0
- `locked_until` timestamptz — populated on too many failed attempts

**`test_requests`**
- `id` uuid PK
- `visit_id` uuid references `visits(id)` not null
- `service_id` uuid references `services(id)` not null
- `status` text default `requested` check in (`requested`, `in_progress`, `result_uploaded`, `ready_for_release`, `released`, `cancelled`)
- `requested_by` uuid references `auth.users(id)` not null
- `requested_at` timestamptz default `now()`
- `assigned_to` uuid references `auth.users(id)`
- `started_at`, `completed_at`, `signed_off_at`, `released_at` timestamptz
- `signed_off_by`, `released_by` uuid references `auth.users(id)`
- `cancelled_reason` text

**`results`**
- `id` uuid PK
- `test_request_id` uuid references `test_requests(id)` unique not null
- `storage_path` text not null — path in `results` storage bucket
- `file_size_bytes` int
- `uploaded_by` uuid references `auth.users(id)` not null
- `uploaded_at` timestamptz default `now()`
- `notes` text — internal medtech notes, NOT visible to patient

**`payments`**
- `id` uuid PK
- `visit_id` uuid references `visits(id)` not null
- `amount_php` numeric(10,2) not null
- `method` text check in (`cash`, `gcash`, `maya`, `card`, `bank_transfer`)
- `reference_number` text — for non-cash methods
- `received_by` uuid references `auth.users(id)` not null
- `received_at` timestamptz default `now()`
- `notes` text

**`appointments`** — for scheduling
- `id` uuid PK
- `patient_id` uuid references `patients(id)` — nullable for new patients
- `walk_in_name` text — for non-registered bookings
- `walk_in_phone` text
- `service_id` uuid references `services(id)`
- `scheduled_at` timestamptz not null
- `status` text default `confirmed` check in (`confirmed`, `arrived`, `cancelled`, `no_show`, `completed`)
- `notes` text
- `created_at` timestamptz, `created_by` uuid references `auth.users(id)`

**`audit_log`** — RA 10173 access log
- `id` bigserial PK
- `actor_id` uuid — staff user id, or NULL if patient/anonymous
- `actor_type` text check in (`staff`, `patient`, `system`, `anonymous`)
- `patient_id` uuid — subject of the access, when applicable
- `action` text not null — e.g. `result.viewed`, `result.downloaded`, `pin.failed`, `payment.recorded`
- `resource_type` text, `resource_id` uuid
- `metadata` jsonb
- `ip_address` inet
- `user_agent` text
- `created_at` timestamptz default `now()`

### Sequences and helpers

- Sequence `drm_id_seq` and a function `generate_drm_id()` returning `'DRM-' || lpad(nextval('drm_id_seq')::text, 4, '0')`.
- Sequence `visit_number_seq` similarly for visit numbers.
- Function `is_staff()` returning bool based on `staff_profiles` row + `is_active`.
- Function `staff_role()` returning the current user's role.
- Function `has_role(text[])` returning bool if current user's role is in list.

### Triggers

- `updated_at` auto-update trigger on every table that has it.
- **Payment-gating trigger** on `test_requests`: when transitioning `status` to `released`, check the parent visit's `payment_status = 'paid'`. If not, raise an exception.
- Trigger on `payments` insert: recalculate `visits.paid_php` and `visits.payment_status`.
- Trigger on `results` insert: auto-flip parent `test_requests.status` from `in_progress` to `result_uploaded` (or to `ready_for_release` if no pathologist sign-off is configured for this service — default to `ready_for_release` initially).

### RLS policies

Enable RLS on every table. Then add:

**`patients`**
- Staff with role in (`reception`, `medtech`, `pathologist`, `admin`) → `SELECT`, `INSERT`, `UPDATE`
- Patients (via the patient-PIN session — see Phase 2) can `SELECT` only their own row

**`visits`** — same staff rules; patients see only their own visits

**`test_requests`**
- Reception → `INSERT`, and `UPDATE` only the `released_*` and status to `released` (with the payment-gating trigger enforcing the rule)
- Medtech → `SELECT` all, `UPDATE` to set `assigned_to`, `started_at`, transition `requested` → `in_progress`
- Pathologist → `UPDATE` to sign off
- Admin → all
- Patient → `SELECT` only their visit's test requests, only when `status = 'released'`

**`results`**
- Medtech, Pathologist, Admin → `INSERT`, `SELECT`
- Reception → `SELECT` (to confirm a result exists; cannot read storage without signed URL)
- Patient → `SELECT` only when their parent `test_request.status = 'released'`. Patients NEVER access storage directly — they get signed URLs from a Server Action that audits the access.

**`payments`**
- Reception, Admin → `INSERT`, `SELECT`
- Medtech, Pathologist → no access
- Patient → `SELECT` only their own visit's payments (so they can see what they paid)

**`audit_log`**
- Admin → `SELECT`
- Everyone else → no `SELECT`. Inserts happen via service-role from server actions.

**`services`** — public read; admin write

**`appointments`** — staff read/write; public can `INSERT` for booking (rate-limited via Edge Function in Phase 6)

### Storage

Create a private bucket `results`. Paths follow `patient_id/visit_id/test_request_id.pdf`. No public access. All downloads go through signed URLs created by a Server Action that:
1. Verifies the requester is the patient who owns the visit OR a permitted staff member.
2. Inserts an `audit_log` row.
3. Returns a signed URL valid for 5 minutes.

### Verification

- [ ] Migration runs cleanly on a fresh Supabase project
- [ ] Inserting a `payment` updates the `visits.paid_php` correctly
- [ ] Attempting to set a `test_request.status` to `released` while `visit.payment_status = 'unpaid'` raises an error
- [ ] A logged-in staff user with role `medtech` cannot `SELECT` from `payments`
- [ ] Generated TypeScript types exist at `src/types/database.ts` (`supabase gen types typescript --linked > src/types/database.ts`)

## Phase 2 — Authentication

**Goal:** Two distinct auth flows working: staff via Supabase Auth (email + password + optional MFA), patients via DRM-ID + receipt PIN.

### Staff auth

Standard Supabase Auth email/password. After a successful sign-in, middleware checks `staff_profiles.is_active = true`; if not, sign out immediately. MFA via Supabase's TOTP enrollment (deferred to Phase 5 — for now, password-only is acceptable but mark it as a TODO with a clear note).

Pages:
- `/staff/login` — email + password (mirrors screenshot 3 layout — STAFF GATE LOGIN). Drop the "Backend API Base" field — that was a placeholder for the multi-backend idea you scrapped. Keep MFA code field as optional.
- `/staff/logout` — handled by Server Action.

### Patient auth (custom — not Supabase Auth)

Patients do NOT have Supabase Auth accounts. Their credentials are `drm_id + pin` against the `visit_pins` table.

Implementation:

1. **Server Action `signInPatient(drm_id, pin)`**:
   - Look up the patient by `drm_id`.
   - Find the most recent `visit_pins` row for any of the patient's visits where `expires_at > now()` and `locked_until` is null or in the past.
   - `bcrypt.compare(pin, pin_hash)`. If fails, increment `failed_attempts`; if >= 5, set `locked_until = now() + interval '15 minutes'`. Audit-log the failure.
   - On success, mint a signed JWT (HS256, server secret) with payload `{ patient_id, drm_id, visit_id, exp: now + 30min }`, and set it as an HttpOnly, Secure, SameSite=Strict cookie named `drmed_patient_session`.
   - Audit-log the success.

2. **Patient session middleware** at `/portal/*`:
   - Reads the cookie, verifies signature and expiry.
   - On failure, redirects to `/portal/login`.
   - On success, makes `patient_id` available via a Server Component helper `getPatientSession()`.

3. **RLS bridge:** since patients don't authenticate to Postgres, the patient portal queries must run through Server Actions or Route Handlers using the service-role client BUT scoped manually by the `patient_id` from the verified session. Alternatively (cleaner): define a Postgres function `set_patient_context(patient_id uuid)` that the server calls before queries, and write RLS policies against `current_setting('app.current_patient_id')`. Use the function approach — it keeps RLS as the single source of truth.

4. **PIN generation** — server-side, when reception creates a visit:
   - 8 characters from a reduced alphabet (no `0`, `O`, `1`, `I`, `l`) for legibility on receipts.
   - Hash with bcrypt (cost 12). Store hash, never the plain PIN.
   - Return the plain PIN ONCE in the API response so reception can print the receipt.

### Pages

- `/portal/login` (mirrors screenshot 4 — "Your Results, Securely Accessible") — DRM-ID and PIN inputs.
- `/portal/logout` — clears cookie.
- `/staff/login` (mirrors screenshot 3 layout, simplified).

### Verification

- [ ] Staff can log in, hit `/staff`, get redirected to `/staff/login` after logout
- [ ] A staff user with `is_active = false` is rejected
- [ ] A patient with valid DRM-ID + PIN can reach `/portal/results`
- [ ] After 5 wrong PINs, the account is locked for 15 minutes (verify in DB)
- [ ] Patient cookie cannot be reused after expiry (manually expire the JWT)
- [ ] Patient session does NOT grant access to other patients' rows (manually try accessing another `patient_id` via the Server Action — should fail at RLS)

## Phase 5 — Patient Portal

**Goal:** Patients sign in and see/download their released results, matching screenshot 1.

### Pages

- `/portal/login` — DRM-ID + PIN form (already built in Phase 2).
- `/portal` (after login) — same UI as screenshot 1: patient name and DRM-ID header, table of visits with columns Visit #, Test, Test Date, Released, Status, Action (Download).

- `/portal/help` — FAQ and contact info.
- `/portal/logout` — clears the patient session cookie.

### Download flow

1. User clicks Download on a released test.
2. Hits a Server Action `getResultDownloadUrl(test_request_id)`.
3. Server verifies: patient session is valid AND owns the visit AND test status is `released`.
4. Inserts `audit_log` with `action = 'result.downloaded'`.
5. Returns a 5-minute signed URL.
6. Browser navigates to the signed URL (browser handles the actual download).

### Notifications (basic)

When `test_requests.status` flips to `released`, fire a Supabase Edge Function `notify_patient`:
1. Look up patient phone and email.
2. Send SMS via Semaphore: "Hi {first_name}, your lab results from drmed.ph are ready. Sign in at https://drmed.ph/portal with your DRM-ID and Secure PIN from your receipt. — drmed.ph"
3. Send email via Resend with a similar message.
4. Failed sends should be logged but should NOT block the release.

Add env vars: `RESEND_API_KEY`, `SEMAPHORE_API_KEY`, `SEMAPHORE_SENDER_NAME`.

### Verification

- [ ] Patient sees only their own visits and only released tests
- [ ] Download produces a working PDF and an audit log entry
- [ ] Releasing a result triggers SMS + email (test with real numbers)
- [ ] Cookie expires after 30 minutes of inactivity (or extend on activity — pick one and document)
- [ ] After expiry, patient is redirected to login

## Phase 7 — Accounting Sync (three Google Sheets)

**Goal:** Faithfully replicate reception's existing Google Sheets accounting (Lab Services, Doctor Consultations, Doctor Procedures HMO) so the accountant's workflow is preserved during beta. The full in-app accounting (HMO billing, doctor PF disbursement, financial statements) ships in a later phase. Until then, the Google Sheets remain the source of truth and the app is a faithful capture-and-export layer.

This phase has three sequential sub-phases. Do not start 7C before 7B is done — the export shape depends on the new schema.

### Real-world context (from reception's current sheets)

Three separate Google Sheets the accountant maintains today, each with its own row grain and column set:

1. **Lab Services (Reception)** — one row per **service line** (a 4-test patient gets 4 rows sharing a control number). Columns: Date, Control No, Patient Name, HMO block (Yes/No, Provider, Approval Date), Service, Base Price, Senior/PWD (20%), Discount (10%), Discount (5%), Actual Discount, Final Price, Payment Method (Cash / HMO / Card / GCash / BPI), Paid Ref, Result Release block (Preferred Medium = Physical / Email / Viber / GCash, Date Released), Receptionist Remarks (last name).
2. **Doctor Consultations** — one row per consultation. Columns: Control No, **Test No**, Patient Name, HMO block, **Doctor Consultant** (last name), **Base Price (variable — each doctor sets their own)**, Senior/PWD (20%), Other Discounts (20%), Final Price, **Clinic Fee** (separate from doctor PF), Paid block (Yes/No, Reference), Remarks (CASH / MEDCERT / CLEARANCE / PRE-EMPLOYMENT, etc.).
3. **Doctor Procedures HMO** — one row per procedure. Columns: Date, Patient Name, HMO Provider, Approval Date, Service Description (free-text procedure list, e.g. "NASAL ENDOSCOPY, NASAL DECONGESTION, SUCTIONING…"), Doctor Consultant, **Approved Amount** (HMO grants a specific peso figure post-approval).

HMO providers actively in use (admin maintains the list, but seed with these 11): **Maxicare, Valucare, Cocolife, Med Asia, Intellicare, Avega, Generali, Etiqa, Amaphil, iCare, Pacific Cross**. The historical sheets only show six of these — the rest are either new or low-volume; reception will start tagging them once the staff UI ships.

The April 2026 test list (≈150 items) has three price columns per test: **DRMed Price (cash)**, **HMO Price** (≈2.2× cash but not always — store as data, not a multiplier), **Senior Discount** (peso amount, not the final price). Five sections: Packages, Individual Tests, X-Ray, Ultrasound, Vaccines, Hi Precision send-outs (asterisked items routed to an external lab). Some entries (e.g. Hepatitis B Vaccine, Executive Packages) have only the DRMed price — HMO column is N/A.

1. Verifies `Authorization: Bearer ${CRON_SECRET}` header.
2. For each sheet key, reads `sync_state.last_synced_at` (default: 24h ago).

6. Updates `sync_state.last_synced_at` to the maximum row timestamp seen.
7. Audit-logs `accounting.sync.completed` with row counts per tab.

## Cross-Cutting Conventions

### Code style
- TypeScript strict mode on
- No `any` without a comment justifying it
- Server Components by default; mark client components with `'use client'` only when needed
- All Supabase calls typed with the generated `Database` type

### Error handling
- Never expose stack traces to users
- All Server Actions return `{ ok: true, data } | { ok: false, error }`
- User-facing errors are short and actionable; technical details go to Sentry

### Naming
- Routes: kebab-case
- DB columns: snake_case
- TypeScript: camelCase for variables, PascalCase for types/components
- Files: kebab-case for routes, PascalCase for components

### Commits
- Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`)
- Each phase ends with a tagged release (`v0.1.0` after Phase 1, etc.)

### What NOT to do
- Do NOT use Supabase Auth for patients — they auth via DRM-ID + PIN
- Do NOT expose the service-role key client-side
- Do NOT skip RLS — it is the single source of truth for access
- Do NOT log plain PINs anywhere, ever
- Do NOT hardcode service prices in the frontend — always read from `services` table
- Do NOT add a "Backend API Base" field to the staff login (legacy from a scrapped multi-backend design)

## Phase 9 — Physician schedules (full availability-driven booking)

**Goal:** Replace Phase 6.5's "reception assigns the doctor on day-of" with picker-driven physician selection based on actual recurring availability and one-off overrides. The marketing physicians page (`/physicians`) also moves from a static TS module to live DB data.

This is the proper version of doctor booking. Phase 6.5 ships a usable shortcut (specialty + manual assignment); this phase delivers what the original drmed.ph hinted at — patients pick a doctor and see only that doctor's open slots.

### Schema

Migration `0006_physicians_and_schedules.sql`:

```sql
-- Promote the static src/lib/marketing/physicians.ts roster into the DB.
create table public.physicians (
  id           uuid primary key default gen_random_uuid(),
  slug         text unique not null,
  full_name    text not null,
  specialty    text not null,
  group_label  text,
  is_active    boolean not null default true,
  photo_path   text,
  bio          text,
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now()
);

-- Recurring weekly availability blocks.
create table public.physician_schedules (
  id            uuid primary key default gen_random_uuid(),
  physician_id  uuid not null references public.physicians(id) on delete cascade,
  day_of_week   int not null check (day_of_week between 0 and 6),
  start_time    time not null,
  end_time      time not null check (end_time > start_time),
  valid_from    date not null default current_date,
  valid_until   date,
  notes         text,
  created_at    timestamptz not null default now()
);

create index idx_physician_schedules_physician on public.physician_schedules(physician_id);

-- One-off overrides (vacation, conference, swap).
create table public.physician_schedule_overrides (
  id            uuid primary key default gen_random_uuid(),
  physician_id  uuid not null references public.physicians(id) on delete cascade,
  override_on   date not null,
  start_time    time,    -- null = unavailable all day
  end_time      time,
  reason        text,
  created_at    timestamptz not null default now()
);

create index idx_physician_overrides_lookup
  on public.physician_schedule_overrides(physician_id, override_on);

-- Tag the booked doctor on appointments.
alter table public.appointments
  add column physician_id uuid references public.physicians(id);

-- Public read on physicians + schedules (the marketing page reads it);
-- admin manages.
alter table public.physicians enable row level security;
alter table public.physician_schedules enable row level security;
alter table public.physician_schedule_overrides enable row level security;

create policy "physicians: public read active" on public.physicians
  for select to anon, authenticated using (is_active = true);
create policy "physicians: admin manage" on public.physicians
  for all to authenticated
  using (public.has_role(array['admin'])) with check (public.has_role(array['admin']));

create policy "physician_schedules: public read" on public.physician_schedules
  for select to anon, authenticated using (true);
create policy "physician_schedules: admin manage" on public.physician_schedules
  for all to authenticated
  using (public.has_role(array['admin'])) with check (public.has_role(array['admin']));

create policy "physician_overrides: public read" on public.physician_schedule_overrides
  for select to anon, authenticated using (true);
create policy "physician_overrides: admin manage" on public.physician_schedule_overrides
  for all to authenticated
  using (public.has_role(array['admin'])) with check (public.has_role(array['admin']));
```

### Photo storage

Doctor photos move from `public/doctors/*.jpg` (static) to a new public Supabase Storage bucket `physician-photos`. Admin upload UI replaces ad-hoc commit-then-deploy. Marketing `/physicians` reads `photo_path` and renders signed-or-public URLs as appropriate.

### Booking flow (Doctor branch update)

Replaces the Phase 6.5 specialty-only flow:

1. Pick specialty → loads `physicians` filtered by that specialty.
2. Render physician cards (photo + name + schedule preview).
3. Pick physician → loads their `physician_schedules` (recurring) and the next 60 days of overrides + `clinic_closures`.
4. Slot picker now intersects: open day (not a clinic closure, weekday matches a recurring block, no full-day override) ∩ open time (within recurring start/end, minus any override window).
5. On submit, `appointments.physician_id` is set; reception sees the assignment without having to ask.
