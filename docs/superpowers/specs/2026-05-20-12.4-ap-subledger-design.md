## 1. Goal

Replace ad-hoc OPEX tracking in the source workbook with a real per-vendor Accounts Payable subledger inside drmed.ph. Three concrete deliverables:

1. **Per-vendor AP balance** answerable on demand ("how much do we owe Mercury Drug right now?").
2. **AP-aging report** (current / 1-30d / 31-60d / 60+d) with vendor breakdown and total outstanding.
3. **Captured supplier withholding tax** at bill-posting time, populating `2340 Withholding Tax Payable — Expanded` so BIR 1601-EQ filing (FP2) has clean source data from day one.

12.4 ships forward-looking AP only. Historical OPEX backfill (2023→present) is **12.B**, sequenced after 12.5 (Doctor PF/Rent) per the project overview. Doctor compensation is **12.5**, NOT here. Input VAT capture is **FP1**. BIR-form generation (2307 PDF, 1601-EQ summary) is **FP2** — the data captured here feeds it.

## 2. Scope boundaries

**In scope (v1.9.0):**

- `vendors` master with WT defaults, default expense account, soft-disable, free-form notes
- Multi-line bills with header + lines + WT (per-bill override of vendor defaults) + per-bill exempt flag
- Bill lifecycle: `draft → posted → partially_paid → paid → voided`. Drafts hard-deletable; posted bills only voidable
- `bill_payments` with M:N allocations across bills (same-vendor constraint per payment)
- Payment methods: cash / bank_transfer / gcash / cheque (with PDC support — JE posts on handover date, not face date)
- Bill attachments via private Supabase Storage bucket; 5-minute signed URLs for reads
- Recurring bill templates with daily Vercel cron to auto-create drafts
- Bridge JEs via the 12.2 reversal mechanism (no new void source_kinds)
- Generic JE detail page (covering all source kinds, not just AP)
- 10 admin surfaces across 14 routes
- ~72 SQL smoke assertions + browser smoke at 390×844 + 1280×800

**Out of scope (deferred — preserved in §13):**

Early-payment discounts, overpayment/underallocation, vendor advances/credit notes, vendor merge, bulk operations, bill duplication soft-warning, input VAT capture (FP1), BIR-form generation (FP2), bank reconciliation outstanding-cheque tracking (FP4), doctor PF/rent (12.5), historical OPEX backfill (12.B).

## 3. Decisions locked in (from brainstorm 2026-05-20)

| # | Decision | Why |
|---|---|---|
| 1 | **Hybrid bill model with paid-on-entry shortcut.** All purchases are Bills. "Paid on entry" creates bill + payment + both JEs in one atomic transaction. | Clean subledger (AP balance = unpaid bills) without forcing double-entry friction on petty-cash buys. |
| 2 | **`vendors` master table, required.** Inline "+ New vendor" on bill form. `default_account_id`, `default_wt_*` autofill per-bill. | Subledger needs stable identities; mirrors HMO providers pattern from 12.3. |
| 3 | **WT-Expanded captured at bill posting.** `wt_amount` frozen on bill row. Vendor-level + per-bill override of `wt_classification` / `wt_rate` / `wt_exempt`. | Backfilling later is painful; CoA seed `2340` already implies WT-on-suppliers in scope. 2307/1601-EQ generation stays FP2. |
| 4 | **Multi-line bills** (header + `bill_lines`). WT applies to gross-total. `bill_lines.account_id` accepts any active CoA account. | Real invoices span expense accounts. Asset debits (Prepaid Expenses, Equipment) and liability debits (loan principal) also need to work. |
| 5 | **M:N `bill_payment_allocations` with bidirectional status recompute.** WT recognized at bill posting, NOT payment. | Partial payments + one-cheque-many-bills are real workflows. Bidirectional flip-back (different from 12.3's one-way) is required for AP-aging to reflect reality after voids. |
| 6 | **Lifecycle `draft → posted → partially_paid → paid → voided`.** Two-button save form ("Save & post" primary; "Save as draft" secondary). Single admin (no maker/checker). Drafts hard-deletable; posted bills only voidable. Draft-aging widget for rot detection. | Draft state earns its keep on genuinely-incomplete bills + FP7 forward-compat. Mandatory ceremony Post-click is unjustified for one-admin shop. |
| 7 | **PDC doctrine A** — JE posts on cheque **handover date** (`payment_date`), not cheque **face date** (`cheque_date`). Cheques in scope from day one. | GAAP/PFRS correct given Option C depth. AP-aging would be wrong otherwise. Bank-rec outstanding-cheque listing (FP4) handles the timing diff. |
| 8 | **Recurring templates in scope.** Daily Vercel cron at 02:00 Manila auto-creates `draft` bills. Single-line templates only in v1. | ~60 monthly re-entries saved across ~5 templates; cost is one table + one cron handler. |
| 9 | **Attachments to private Supabase Storage bucket `bill-attachments`** + internal `BL-YYYY-NNNN` / `BP-YYYY-NNNN` numbering using year counters. | BIR audit needs proof of expense; stable IDs help cross-referencing in bank rec + audit. |
| 10 | **All admin surfaces in scope** (including the JE detail page, pulled in mid-brainstorm from the deferred list). 14 routes total, split across 7 dispatches. | All surfaces close real daily-workflow gaps. JE detail enables drill-from-bill, also required for 12.7 (period close). |

## 5. Data model

### 5.1 New tables

All in `public` schema. Soft-disable rather than delete for any row that has audit implications (vendors, recurring templates).

**`vendors`**
- `id uuid pk default gen_random_uuid()`
- `name text not null` — display name
- `tin text` — PH 9-12 digits with optional hyphens; nullable
- `email text`, `phone text` — nullable
- `default_account_id uuid` references `coa_accounts(id)` — autofills bill line account
- `default_wt_classification text` — e.g., 'WI160', 'WI100', 'WC010'; nullable
- `default_wt_rate numeric(5,4)` — e.g., 0.0200 for 2%; nullable
- `notes text` — free-form admin notes; nullable
- `is_active boolean not null default true`
- `created_at`, `created_by`, `updated_at`, `updated_by` — standard audit cols
- Indexes:
  - `unique (lower(name))` — case-insensitive uniqueness
  - `unique (tin) where tin is not null` — partial unique allowing multiple NULL TINs
  - `gin (name gin_trgm_ops)` for search

**`bills`**
- `id uuid pk default gen_random_uuid()`
- `bill_number text not null unique` — `BL-YYYY-NNNN`, trigger-generated using `bill_year_counters`, year = `EXTRACT(YEAR FROM created_at)`
- `vendor_id uuid not null` references `vendors(id)`
- `vendor_invoice_number text` — vendor's own invoice ref; nullable
- `bill_date date not null` — invoice date per the vendor's document
- `due_date date not null check (due_date >= bill_date)`
- `status text not null default 'draft' check (status in ('draft','posted','partially_paid','paid','voided'))`
- `description text` — bill-level note
- `gross_amount numeric(12,2) not null default 0` — denorm from `SUM(bill_lines.amount_php)`
- `wt_classification text` — defaults from vendor, per-bill editable; nullable
- `wt_rate numeric(5,4)` — defaults from vendor, per-bill editable; nullable
- `wt_exempt boolean not null default false`
- `wt_amount numeric(12,2) not null default 0 check (wt_amount between 0 and gross_amount)` — computed at posting time, frozen thereafter
- `net_payable numeric(12,2) generated always as (gross_amount - wt_amount) stored`
- `paid_amount numeric(12,2) not null default 0` — denorm from active allocations
- `outstanding_amount numeric(12,2) generated always as (net_payable - paid_amount) stored` — indexed for aging queries
- `template_id uuid` references `recurring_bill_templates(id) on delete set null` — populated when bill came from cron
- `posted_at timestamptz`, `posted_by uuid references auth.users(id)` — null until posted
- `voided_at timestamptz`, `voided_by uuid references auth.users(id)`, `void_reason text` — null until voided
- `created_at`, `created_by`, `updated_at`, `updated_by`
- Indexes:
  - `(vendor_id, status, due_date)` — AP-aging queries
  - `(outstanding_amount) where outstanding_amount > 0` — partial index for "what's still owed"
  - `gin (vendor_invoice_number gin_trgm_ops, description gin_trgm_ops)` — search

**`bill_lines`**
- `id uuid pk default gen_random_uuid()`
- `bill_id uuid not null references bills(id) on delete cascade`
- `line_no int not null check (line_no >= 1)`
- `description text`
- `amount_php numeric(12,2) not null check (amount_php > 0)`
- `account_id uuid not null references coa_accounts(id)` — any active account (asset/expense/liability all valid)
- `created_at`
- Indexes: `unique (bill_id, line_no)`, `(account_id)` for "what bills hit this account"

**`bill_payments`**
- `id uuid pk default gen_random_uuid()`
- `payment_number text not null unique` — `BP-YYYY-NNNN`, trigger-generated using `bill_payment_year_counters`, year = `EXTRACT(YEAR FROM created_at)`
- `vendor_id uuid not null references vendors(id)` — denorm for filtering; matches allocations
- `payment_date date not null check (payment_date <= (now() at time zone 'Asia/Manila')::date)` — handover date = JE posting_date; CHECK prevents future-dated postings
- `method text not null check (method in ('cash','bank_transfer','gcash','cheque'))`
- `cash_account_id uuid not null references coa_accounts(id)` — 10xx
- `amount_php numeric(12,2) not null check (amount_php > 0)`
- `reference text` — free-form (transfer ref, GCash ref, etc.)
- `cheque_number text` — required iff method='cheque'
- `cheque_date date` — required iff method='cheque'; informational, NOT JE posting date
- `voided_at timestamptz`, `voided_by uuid`, `void_reason text`
- `created_at`, `created_by`, `updated_at`, `updated_by`
- CHECK: `(method = 'cheque') = (cheque_number IS NOT NULL AND cheque_date IS NOT NULL)` — DB-enforced
- Indexes: `(vendor_id)`, `(method)` (cheque-register filter), `(payment_date)`

**`bill_payment_allocations`**
- `id uuid pk default gen_random_uuid()`
- `payment_id uuid not null references bill_payments(id) on delete cascade`
- `bill_id uuid not null references bills(id)`
- `allocated_amount numeric(12,2) not null check (allocated_amount > 0)`
- `voided_at timestamptz` — set by cascade when parent payment is voided
- `created_at`
- Indexes: `unique (payment_id, bill_id)`, `(bill_id) where voided_at is null` for paid_amount denorm

**`bill_attachments`**
- `id uuid pk default gen_random_uuid()`
- `bill_id uuid not null references bills(id) on delete cascade`
- `storage_path text not null` — `bills/{bill_id}/{uuid}-{sanitized_filename}` in `bill-attachments` bucket
- `filename text not null` — original sanitized
- `mime_type text not null check (mime_type in ('application/pdf','image/jpeg','image/png'))`
- `size_bytes int not null check (size_bytes > 0 and size_bytes <= 10485760)` — 10 MB cap
- `uploaded_at timestamptz not null default now()`
- `uploaded_by uuid not null references auth.users(id)`
- Indexes: `(bill_id)`

**`recurring_bill_templates`**
- `id uuid pk default gen_random_uuid()`
- `vendor_id uuid not null references vendors(id)`
- `description text not null`
- `cadence text not null default 'monthly' check (cadence = 'monthly')` — v1 supports monthly only; column kept for FP extension
- `due_day_of_month int not null check (due_day_of_month between 1 and 31)`
- `bill_date_offset_days int not null default 0 check (bill_date_offset_days between -30 and 0)` — e.g. -5 for "bill arrives 5 days before due"
- `amount_php numeric(12,2)` — nullable: NULL = variable, cron-created draft has empty amount for admin to fill
- `default_account_id uuid not null references coa_accounts(id)`
- `default_wt_classification text`, `default_wt_rate numeric(5,4)`, `default_wt_exempt boolean not null default false`
- `next_run_date date not null` — driven by cron loop
- `is_active boolean not null default true`
- `created_at`, `created_by`, `updated_at`, `updated_by`
- Indexes: `(is_active, next_run_date) where is_active = true` — cron query hot path

**`bill_year_counters`**
- `year int primary key`
- `next_n int not null default 1`

**`bill_payment_year_counters`**
- `year int primary key`
- `next_n int not null default 1`

```sql
ALTER TYPE public.je_source_kind ADD VALUE IF NOT EXISTS 'bill_post';

```

**Voids use the existing 12.2 reversal mechanism — no new void source_kinds.**

### 5.3 Denorm and recompute triggers

| Trigger | Fires on | Effect |
|---|---|---|
| `bill_lines_recompute_gross` | INSERT/UPDATE/DELETE on `bill_lines` | Recomputes `bills.gross_amount = SUM(amount_php)` |
| `bill_payment_allocations_recompute_paid` | INSERT/UPDATE/DELETE on `bill_payment_allocations` | Recomputes `bills.paid_amount = SUM(allocated_amount WHERE voided_at IS NULL)` AND flips `bills.status` **bidirectionally** (`posted ↔ partially_paid ↔ paid`) |
| `bill_payment_void_cascade` | UPDATE on `bill_payments` (status → voided) | Sets `voided_at` on all allocations belonging to that payment |

**Bidirectional flip is the key deviation from 12.3.** 12.3's `recompute_hmo_batch_status` is intentionally one-way (operational signal, not financial). For AP, status is purely financial — voiding a payment genuinely means we owe the vendor again; AP-aging must reflect this.

**`updated_at` triggers.** Every table with an `updated_at` column gets a `BEFORE UPDATE` trigger `set_updated_at` that sets `NEW.updated_at = now()`. Standard project convention; consolidated as a single trigger function reused across `vendors`, `bills`, `bill_payments`, `recurring_bill_templates`.

### 5.4 Constraint guards — new error codes

| Code | Trigger | Fires when |
|---|---|---|
| **P0013** | `BEFORE UPDATE ON bills WHEN (OLD.status != 'voided' AND NEW.status = 'voided')` | Bill has non-voided allocations pointing to non-voided bill_payments |
| **P0014** | Constraint trigger `validate_bill_payment_allocations` (DEFERRABLE INITIALLY DEFERRED) | `SUM(allocated_amount) WHERE voided_at IS NULL` per payment ≠ `bill_payments.amount_php` |
| **P0015** | Same constraint trigger | `SUM(allocated_amount) WHERE voided_at IS NULL` per bill > `bills.net_payable` |
| **P0016** | Same constraint trigger | `allocation.bill.vendor_id ≠ allocation.payment.vendor_id` |
| **P0017** | Same constraint trigger | `allocation.bill.status NOT IN ('posted', 'partially_paid')` |

All five new codes get user-facing translations in `src/lib/db/pg-errors.ts`. The constraint trigger fires once per transaction at COMMIT and checks all four invariants in one function.

### 5.5 Lifecycle and deletion semantics

| Status | Allowed transitions | Edit allowed? | Delete allowed? |
|---|---|---|---|
| `draft` | → `posted` (Post action), → deleted (hard) | Yes (any field, via update_bill_draft RPC) | Yes (cascade deletes lines + attachments, plus app-layer storage file removal) |
| `posted` | → `partially_paid` / `paid` (auto via recompute), → `voided` (Void action) | No (P0004 from 12.2 blocks edits-after-JE) | No (only Void) |
| `partially_paid` | → `paid` (auto), → `posted` (allocation void), → `voided` (Void w/ active-payment check P0013) | No | No |
| `paid` | → `partially_paid` (allocation void), → `posted` (all allocations voided), → `voided` (P0013 may block) | No | No |
| `voided` | (terminal) | No | No |

**Vendor deactivation nuance:** `createBillDraftAction` blocks against inactive vendors (Zod refinement, server-side query check). `postBillAction` does NOT re-check vendor active state — drafts created when vendor was active retain the right to post, since legitimate late-arriving invoices from a now-discontinued vendor must still be recordable.

## 6. Bridge JEs

Three trigger points fire JEs. Each `bill_line` maps to its own `journal_line` for traceability (no account-level consolidation).

### 6.1 Bill post — fires on `bills.status` transition into `posted`

`WHEN (OLD.status = 'draft' AND NEW.status = 'posted')` — precise condition to avoid re-fires on subsequent status flips.

**WT calculation happens BEFORE the trigger fires.** The atomic-op PG function (`create_bill_and_post` or `create_bill_paid_on_entry`) computes `wt_amount = ROUND(gross_amount × wt_rate, 2)` (PG default half-away-from-zero) and writes to `bills.wt_amount` BEFORE issuing the `UPDATE status='posted'`. Bridge trigger reads the already-frozen value.

**No-WT case** (`wt_exempt = true` OR `wt_rate IS NULL` OR `wt_rate = 0`):

| Account | DR | CR |
|---|---:|---:|
| `bill_lines[i].account_id` (one row per line) | `bill_lines[i].amount_php` | |
| `2100 AP — Trade` | | `bills.net_payable` (= gross) |

**WT case:**

| Account | DR | CR |
|---|---:|---:|
| `bill_lines[i].account_id` (one row per line) | `bill_lines[i].amount_php` | |
| `2340 WT Expanded Payable` | | `bills.wt_amount` |
| `2100 AP — Trade` | | `bills.net_payable` (= gross − wt) |

- `source_kind = 'bill_post'`
- `source_id = bills.id`
- `posting_date = bills.bill_date` (not `created_at`; matters for closed-period guard)
- Description: `"Bill BL-YYYY-NNNN — <vendor.name> — <bills.description>"`
- Idempotency: existing `journal_entries_one_posted_per_source` partial unique index (predicate from 0037) covers `(bill_post, bills.id)` naturally — 1:1 mapping. No predicate extension needed.

### 6.2 Bill payment — fires on `bill_payments` INSERT

Allocations don't fire their own JEs (denorm only). One `bill_payments` row = one JE, regardless of allocation count.

| Account | DR | CR |
|---|---:|---:|
| `2100 AP — Trade` | `bill_payments.amount_php` | |
| `cash_account_id` (10xx) | | `bill_payments.amount_php` |

- `source_kind = 'bill_payment'`
- `source_id = bill_payments.id`
- `posting_date = bill_payments.payment_date` (handover date; for cheques = the date admin gave it over, NOT the cheque face date)
- Description: `"Payment BP-YYYY-NNNN — <vendor.name> — <method> #<reference or cheque_number>"`
- `payment_date <= today_manila` enforced by CHECK constraint + Zod refinement using `todayManilaISODate()` from `src/lib/dates/manila.ts`

### 6.3 Voids — reuse 12.2's reversal mechanism

**No new source_kind.** When admin voids a bill:

1. P0013 guard fires if active payments exist → blocked
2. Original `bill_post` JE: status `posted → reversed`, `reversed_by` populated
3. New reversal JE: `source_kind = 'reversal'`, `reverses = <original_je_id>`, sign-flipped lines, `posting_date = today` (must be in current open period — strict accountant practice)
4. `bills.status = 'voided'`, void audit cols populated

When admin voids a `bill_payment`:

1. Original `bill_payment` JE: status `posted → reversed`
2. Reversal JE: `source_kind = 'reversal'`, sign-flipped, `posting_date = today`
3. Allocations: `voided_at` populated by cascade trigger (soft-mark, NOT delete)
4. Recompute trigger fires: `bills.paid_amount` decreases → `bills.status` flips bidirectionally

### 6.4 Paid-on-entry atomic flow

When admin saves a new bill with "paid in full" ticked, the Server Action calls `create_bill_paid_on_entry(p_input jsonb, p_actor_id uuid)`:

1. INSERT `bills` with `status = 'draft'`
2. INSERT `bill_lines`
3. Compute `wt_amount = ROUND(gross_amount × wt_rate, 2)`; write to `bills.wt_amount`
4. UPDATE `bills.status = 'posted'` → `bill_post_bridge` trigger fires JE #1
5. INSERT `bill_payments` → `bill_payment_bridge` trigger fires JE #2
6. INSERT `bill_payment_allocations` (single row, `allocated_amount = bills.net_payable`) → recompute trigger fires **immediately** → `bills.paid_amount` updated → `bills.status` flips to `'paid'`
7. Function returns; transaction commits → **deferred** constraint trigger validates sum-per-payment + sum-per-bill + vendor-match + status invariants (P0014–P0017)

If any step fails, the entire function rolls back. Two JEs posted atomically or none. Audit log records three rows (`bill.created`, `bill.posted`, `bill_payment.created`) with a shared `request_id` metadata field for post-hoc grouping.

**Partial paid-on-entry not supported in v1** — only "paid in full" via this path. Partial payment of a freshly-created bill requires saving the bill first, then recording a partial payment via the separate Payments flow.

### 6.5 Closed-period interaction

The closed-period guard (P0002) from 12.1 fires on `journal_entries.posting_date`:

- Bill post blocked if `bills.bill_date` in closed period
- Payment recording blocked if `bill_payments.payment_date` in closed period
- Reversal of either always allowed: reversal JE's `posting_date = today`, which by definition is in the current open period

### 6.6 Idempotency

Existing `journal_entries_one_posted_per_source` partial unique index (predicate from 0037: `status='posted' AND source_kind != 'reversal' AND source_id IS NOT NULL AND source_kind != 'hmo_history_opening'`) covers both `bill_post` and `bill_payment` naturally — each source row has exactly one JE. No predicate extension needed.

### 6.7 WT specifics

**Rate range:** `0 ≤ wt_rate ≤ 1.0`. Realistic values: 0.0100 (1%, goods purchases WI010), 0.0200 (2%, professional services WI160), 0.0500 (5%, rentals WI100).

**Rounding:** Standard half-away-from-zero (PG default for `ROUND(numeric, 2)`). `ROUND(9999.99 × 0.02, 2) = 200.00`.

**Exempt vs zero-rate:** `wt_exempt = true` and `wt_rate = 0` produce identical JEs (no `2340` line). Both fields preserved on the bill row — FP2 (1601-EQ filing) distinguishes them for BMBE-registered vendors (`wt_exempt`) vs below-threshold transactions (`wt_rate = 0`).

**Defense-in-depth CHECK:** `wt_amount between 0 and gross_amount` prevents logic bugs from making `net_payable` negative.

## 7. RLS, audit, storage

### 7.1 RLS

All 9 new tables: `ENABLE ROW LEVEL SECURITY` + admin-only `FOR ALL` policy:

```sql
CREATE POLICY <table>_admin_all ON <table>
  FOR ALL USING (public.has_role(ARRAY['admin']))
       WITH CHECK (public.has_role(ARRAY['admin']));
```

**Counters (`bill_year_counters`, `bill_payment_year_counters`)**: RLS enabled with NO policy granting access. Only `SECURITY DEFINER` trigger functions (owned by `supabase_admin`, the migration runtime superuser) touch them. Admin never reads or writes directly.

**Read path** = Server Action `getBillAttachmentDownloadUrlAction(attachmentId)`:
1. `requireAdminStaff()` for auth
2. Load attachment + parent bill (RLS enforces admin-readable)
3. `supabase.storage.from('bill-attachments').createSignedUrl(path, 300)` — 5-minute URL
4. INSERT `audit_log` row: `bill_attachment.downloaded`
5. Return URL to client

**Delete semantics:**
- Hard-delete draft bill → cascade-delete `bill_attachments` rows + app-layer call to `supabase.storage.from('bill-attachments').remove(paths)`
- Void posted bill → `bill_attachments` rows preserved + files preserved (BIR 10-year retention)
- Individual attachment delete → ALLOWED only if `bill.status = 'draft'`; BLOCKED otherwise with user-facing error

### 7.3 Audit events

All written by Server Actions (or PG functions called from them, with `p_actor_id` passed through). JEs themselves are NOT audit-logged separately — they're observable in `journal_entries`. Recompute-driven status flip-backs are NOT separately audit-logged — observable from the bill row's current state.

| Event | Notes |
|---|---|
| `vendor.created` / `vendor.updated` / `vendor.deactivated` / `vendor.reactivated` | Distinct events for state transitions; updated captures field-level diff |
| `bill.created` | Bill row created (status='draft' at this point, even on the paid-on-entry path which then flips it via subsequent `bill.posted` audit row) |
| `bill.updated` | **Drafts only** (P0004 blocks posted edits) |
| `bill.posted` | Distinct even on paid-on-entry path |
| `bill.deleted` | **Drafts only**; voided bills keep their rows |
| `bill.voided` | Includes `void_reason` in metadata |
| `bill_payment.created` | Metadata includes allocations array |
| `bill_payment.voided` | Cascade-effect on bill statuses observable from bill rows; no separate audit row |
| `bill_payment.reallocated` | Metadata: `{before: [...], after: [...]}` |
| `bill_attachment.uploaded` / `bill_attachment.downloaded` / `bill_attachment.deleted` | Download captures signed-URL mint moment |
| `recurring_template.created` / `recurring_template.updated` / `recurring_template.deactivated` / `recurring_template.reactivated` | |
| `recurring_template.fired` | System actor (`actor_type='system'`, `actor_id=NULL`), metadata = `{bill_id, run_date}` |
| `recurring_template.fire_failed` | System actor, metadata = `{template_id, error}` |

**Paid-on-entry transaction:** three audit rows (`bill.created`, `bill.posted`, `bill_payment.created`) share a `request_id` UUID in metadata for post-hoc grouping.

### 8.6 Recurring cron — `/api/cron/recurring-bills`

Route handler in `src/app/api/cron/recurring-bills/route.ts`:

```
1. Validate request.headers.authorization === `Bearer ${CRON_SECRET}` → 401 otherwise
2. const today = todayManilaISODate()
3. Query: SELECT * FROM recurring_bill_templates WHERE is_active AND next_run_date <= today
4. For each template (per-template try/catch):
     a. Iterate up to MAX 12 times (loop safety against next_run_date stuck-advance bug):
        - call supabase.rpc('post_recurring_template', { p_template_id })
        - the function does (atomically): create draft bill + line + audit + advance next_run_date by 1 month
        - exit loop when next_run_date > today
     b. On per-template failure: reportError(err, {template_id}) + audit `recurring_template.fire_failed`; continue with next template
5. Return JSON { processed, drafts_created, failures }
```

**Schedule** in `vercel.json` (or `vercel.ts` — confirm at D6 which format the project uses): `"0 18 * * *"` UTC = 02:00 Manila next day (off-hours).

**Env var:** `CRON_SECRET` (32-byte hex); add to `.env.example`.

**Failure modes:**
- Vercel platform outage → next day's cron catches up (`next_run_date` still ≤ today)
- Per-template failure → Sentry + audit; others continue
- Drafts created but admin doesn't post → draft-aging widget surfaces them after 7 days
- Same-day re-invocation → idempotent (loop exits immediately)

**`payment_method_account_map` from 0030 is NOT reused for AP.** That table maps receive-side payment methods (patient GCash → 1030 etc.). AP is send-side; admin picks `cash_account_id` directly per `bill_payments` row. Documented to prevent future-implementer confusion.

### 9.4 Edge cases verified in smoke + browser

- Concurrent bill INSERTs preserve counter sequence (row lock)
- Concurrent void of same bill: second sees `voided`, no-ops (12.2 `.is('voided_at', null)` idempotency)
- Year boundary: bill created 2025-12-31 → BL-2025-XXXX; created 2026-01-02 → BL-2026-0001 (year from `created_at`)

- Storage: upload to draft OK; delete from draft OK; delete from posted/voided BLOCKED (BIR retention)
- Cron: no-due no-op; one-due creates draft + advances; vendor deactivated mid-loop → fail-and-continue; same-day re-run idempotent; backlog catch-up capped at 12 iterations
- Reallocation: same-vendor OK; cross-vendor → P0016 at commit, rolls back
- Inactive vendor: new draft blocked (Zod refinement); existing drafts can still be posted
- WT exempt vs rate=0: identical JE, distinct field values

## 10. Migration sequence — TWO files

12.4 schema is too large for a single migration to review reliably (~1100-1300 LoC total). Splitting into schema + behavior, mirroring 12.A's 0035+0036 pattern.

**`0048_ap_subledger_schema.sql` (~500 LoC):**

## 12. Risk register

| Risk | Mitigation |
|---|---|
| PG enum ALTER TYPE + same-migration function references | Section-0 ordering in 0048; functions in 0049 (separate migration) — eliminates the same-transaction concern entirely |
| Constraint trigger semantics (DEFERRABLE INITIALLY DEFERRED fires AT COMMIT) | Implementers must know recompute trigger is NOT deferred — it fires immediately; only allocation-sum invariants are deferred. Documented in trigger function header. |
| Bridge JE actor context | Triggers don't write audit_log; Server Actions / PG functions do (with p_actor_id) — already aligned |
| `vercel.json` vs `vercel.ts` for cron config | Confirm at D6 which the project uses; either works |
| Supabase MCP execute_sql ignoring BEGIN/ROLLBACK | Smoke runs on local Docker Supabase only; hard rule in smoke header |
| `0044_payroll 2.sql` filename with space | Investigate at D1 start; delete if leftover; document |
| Per-bill WT override mistakes | WT three-state UI (apply / exempt / 0%) reduces accidental misclassification; audit log captures field-level diff on draft edits |
| Reallocation intermediate state | `reallocate_bill_payment` PG function does DELETE then INSERT atomically; recompute fires twice but final state is correct; deferred trigger validates at commit |
| `pg_trgm` not enabled on staging | `CREATE EXTENSION IF NOT EXISTS pg_trgm` is idempotent; safe in 0048 |
| Concurrent allocation conflict (two simultaneous payments to the same bill exceeding net_payable under READ COMMITTED) | Accepted risk: vanishingly rare for a 1-admin clinic with manual data entry. Deferred constraint trigger validates per-transaction; cross-transaction collision would require serializable isolation, not pursued in v1. |
