Draft Design Specification Pre-beta · 2026-05-25

Legacy Customer Import & Attribution Capture

Wipe operational test data, import ~4,480 historical customers from the live Google Sheet, and extend the schema to stop discarding senior/PWD, referring-physician, referral-source, and result-release-preference data captured at intake. Scoped for staff-side beta on 2026-05-26.

00 · One-page summary

Why this exists, in one read

Beta begins tomorrow with reception, medtechs, pathologists, and admins walking through real workflows. The current patients table is full of dev noise; the legitimate customer history lives in a Google Sheet of ~4,480 rows that the team has maintained operationally. We need (a) the test data gone, (b) the sheet imported lossless, (c) the schema slightly extended so attribution data that reception has been collecting all along stops being discarded the moment we put it in a database.

Recommended path

One migration (0051) + two reversible scripts (wipe-operational, import-legacy-customers) + a thin UI surface (4 patient-form fields, 4 card badges, 1 receipt line, 1 visit-creation checkbox). Each script is dry-run by default; the importer produces a pre-flight report you review before committing.

01 · Scope

Goals & non-goals

Goals

  1. Wipe all operational test data — patients, visits, payments, GL bridge, audit_log — without touching reference/config tables.
  2. Migrate the live customer list from CUSTOMER LIST - CUSTOMER LIST2.csv (4,480 rows) into patients, lossless.
  3. Extend the schema so attribution and entitlement data (senior/PWD, referring physician, referral source, result-release preference) stops being thrown away.
  4. Ship enough in-app affordance that reception can use the imported records on day one of beta.
  5. Defer anything requiring external integrations (email/SMS delivery) or larger accounting work to follow-up — capture now, fulfil later.

Non-goals

  • Patient-portal login for legacy patients (beta is staff-side only). Legacy rows have no visits and no PINs.
  • The Google OAuth migration discussed earlier — separate project, separate session.
  • Backfilling stub visits to give legacy patients a portal login path.
  • Promoting referring-physician strings into the physicians table (follow-up).
  • BIR-compliant senior/PWD discount ledger view (follow-up; data captured now so the ledger is buildable later).

Observations that shape the importer

  • The dedicated Last Name / First Name / M.I. columns are empty for almost every row. The real name lives in Full Name as "Last, First" (sometimes "Last, First Middle").
  • The Age column is broken — nearly every row reads 126. Ignored entirely; not used for backfill.
  • Contact Number formats vary: 09xxxxxxxxx, 639xxxxxxxxx, with/without spaces, dashes, and parentheses.
  • Address is split across three columns (street, barangay, city); each often empty.
  • The Doctor column holds the referring physician as free text (e.g. DR. KATHERINE GAYO).
  • How did you know about DR Med? is inconsistent free text — needs mapping to a controlled vocabulary.
  • Trailing junk rows (, , , , ...) at the end of the sheet — filtered on import.

04 · Architecture

Four streams that compose into tomorrow's artefact

C1
Schema migration
Add 5 columns + nullable DOB + confirmed flag + discount + provenance table. Patch payment-status trigger.
C2
Operational wipe
Dry-run by default. Single transaction. Clears operational tables only — reference data preserved.
C3
Legacy importer
Dry-run produces pre-flight CSV. Commit phase inserts with provenance, enabling per-batch rollback.

05 · Component one

Schema migration 0051_legacy_intake_and_attribution.sql

C1.a · patients
Add attribution columns & loosen DOB
supabase/migrations/0051_legacy_intake_and_attribution.sql
-- Make birthdate optional; legacy rows often lack it.
alter table public.patients
  alter column birthdate drop not null;

alter table public.patients
  add column birthdate_confirmed boolean not null default false,

  -- Senior Citizens Act (RA 9994) + PWD Magna Carta (RA 10754):
  -- 20% mandatory discount + VAT exemption. Type + ID number must both
  -- appear on the receipt for the discount to be legally compliant.
  add column senior_pwd_type text
    check (senior_pwd_type in ('senior', 'pwd')),
  add column senior_pwd_id text,
  add constraint senior_pwd_id_required_with_type
    check ((senior_pwd_type is null) = (senior_pwd_id is null)),

  -- Free-text. Promote frequent strings to physicians table later.
  add column referring_physician text,

  -- Controlled vocabulary. Unmapped legacy values → 'other'.
  add column referral_source text
    check (referral_source in (
      'doctor_referral', 'walk_in', 'facebook', 'instagram',
      'tiktok', 'google_search', 'word_of_mouth',
      'returning_patient', 'gift_code', 'other'
    )),

  -- Captured now; portal/email/sms delivery hooks are follow-ups.
  add column result_release_pref text not null default 'portal_only'
    check (result_release_pref in (
      'portal_only', 'portal_and_email', 'print_pickup', 'sms_link'
    )),

  -- Lossless capture of original sheet row + parse warnings.
  add column legacy_intake jsonb,

  -- Provenance for rollback-by-batch.
  add column legacy_import_run_id uuid;
C1.b · visits
Discount column + payment-status trigger update
extends 0001_init.sql definitions
alter table public.visits
  add column discount_php   numeric(10,2) not null default 0
    check (discount_php >= 0),
  add column discount_reason text,
  add constraint visits_discount_le_total
    check (discount_php <= total_php);

total_php remains the gross total. discount_php is separate so the receipt can show both lines and accounting can later route the discount to a contra-revenue account. Net amount owed = total_php − discount_php.

Payment-status trigger

public.recalc_visit_payment() (defined in 0001_init.sql:348) compares paid_php against total_php. Migration 0051 replaces the function body to compare against total_php − discount_php, preserving the waived short-circuit and all other behaviour. Trigger binding unchanged. 

create or replace function public.recalc_visit_payment()
returns trigger language plpgsql as $$
declare
  v_net  numeric(10,2);
  v_paid numeric(10,2);
begin
  select total_php - discount_php into v_net
  from public.visits where id = new.visit_id for update;

  select coalesce(sum(amount_php), 0) into v_paid
  from public.payments where visit_id = new.visit_id;
  -- ...remainder identical, but v_net replaces v_total
end;
$$;
C1.c · provenance
legacy_import_runs + fuzzy name index
create table public.legacy_import_runs (
  id            uuid primary key default gen_random_uuid(),
  source        text not null,
  started_at    timestamptz not null default now(),
  ended_at      timestamptz,
  rows_in       int,
  rows_inserted int,
  rows_skipped  int,
  rows_flagged  int,
  dry_run       boolean not null,
  run_by        uuid references auth.users(id),
  notes         text

-- Rollback ergonomic:
--   DELETE FROM patients WHERE legacy_import_run_id = $1;
create index idx_patients_legacy_import_run
  on public.patients(legacy_import_run_id)
  where legacy_import_run_id is not null;

-- Reception name-search needs to be fuzzy: half the imported rows
-- have no DOB and no email, so name is the main lookup tool.
create extension if not exists pg_trgm;
create index idx_patients_name_trgm
  on public.patients using gin (
    (lower(coalesce(first_name,'') || ' '
       || coalesce(last_name,'') || ' '
       || coalesce(middle_name,''))) gin_trgm_ops
  );

Behaviour

  • Dry-run prints the row count it would delete from each table.
  • --commit requires the confirm token; aborts on bad input.
  • Uses the service-role client (src/lib/supabase/admin.ts).
  • Writes one final audit_log row after the wipe — action='ops.wipe' — so we have a trail of the wipe itself.
  • Exits non-zero on any failure inside the transaction; partial-wipe impossible.

07 · Component three

Legacy importer scripts/import-legacy-customers.ts

Pre-flight report (always produced)

Outputs tmp/legacy-import-preflight-<timestamp>.csv plus a human summary to stdout:

  • Total rows, parseable rows, junk rows skipped, with sample.
  • DOB present rate, phone format distribution, email present rate.
  • Top 30 referring physicians, frequency-sorted — partner reviews.
  • Within-sheet duplicates by (normalized_last, normalized_first, phone).
  • Potential matches against existing patients.
  • Per-row parse preview for the first 50 rows.

Row parser

  1. Skip rows where Full Name is empty/whitespace or matches /^,\s*$/.
  2. Name parse from Full Name: split on first , → [last, rest]; rest.trim().split(/\s+/) → first token = first_name, remainder joined = middle_name. Title-case each token.
  3. DOB parse: try M/D/YYYY, MM/DD/YYYY, YYYY-MM-DD. Reject Age column entirely.
  4. Phone normalize to E.164: strip non-digits → map 09xxxxxxxxx / 639xxxxxxxxx / 9xxxxxxxxx → +639xxxxxxxxx. Unparseable → NULL + raw kept in legacy_intake + warning.
  5. Email: lowercase + trim; empty → NULL.
  6. Address: join three columns with , , collapse whitespace, trim trailing commas.
  7. Sex: Female/F → female; Male/M → male; else NULL.
  8. Senior/PWD: both type and ID required (CHECK constraint); only-type → warning + NULL both.
  9. Referring physician: free-text copy of Doctor column, trim only.
  10. Referral source mapping — controlled vocabulary; unmapped → other + warning preserving the raw string.
  11. Result-release preference: similar mapping; default portal_only.

Dedup cascade

Within-sheet first (collapse exact dupes, append duplicate row indexes to legacy_intake.duplicate_of). Then against live patients:

OrderKeyUsed when
1(lower(last_name), lower(first_name), birthdate)DOB present
2(lower(last_name), lower(first_name), normalized_phone)Phone present
3(lower(last_name), lower(first_name), lower(email))Email present
4Insert; flag as name_only_match_possibleOtherwise

Never auto-merge on weak keys. Existing migration-0025 merge tool handles dupes manually if needed.

Commit phase (--commit)

  • Insert legacy_import_runs row with dry_run=false.
  • Bulk insert parsed patients in batches of 500 with legacy_import_run_id set.
  • One audit_log row per insert: action='patient.legacy_import'.
  • Print rollback command: DELETE FROM patients WHERE legacy_import_run_id = '<uuid>';

09 · Execution plan

Build sequence

  1. Migration 0051 + types regen
    Schema changes, payment-status trigger update, pg_trgm extension. Run npm run db:types.
  2. Operational wipe — dry-run, then commit
    Tag a pg_dump snapshot before commit: pre-legacy-import-2026-05-25.
  3. Pre-flight import dry-run
    Partner reviews the pre-flight report. Make any mapping corrections.
  4. Commit import
    --commit with a tagged legacy_import_runs row. Rollback is a single DELETE.
  5. Intake form updates
    Reception form first; public form gets one field.
  6. Patient-card badges
    Senior/PWD, referring physician, result pref, confirm-DOB.
  7. Visit creation discount checkbox + receipt line
    Wire the existing visit form to set discount_php and discount_reason.
  8. Smoke test
    Create a visit for a senior/PWD legacy patient, take payment, release a result. Verify the discount appears on the receipt and age-banded ranges fall back cleanly when DOB unconfirmed.
  9. Typecheck, lint, commit
    Conventional Commits. Tag a release when staff-side beta is live.

10 · Risk register

Risks & mitigations

SeverityRiskMitigation
High Wipe deletes something we should have kept Pre-wipe pg_dump tagged pre-legacy-import-2026-05-25. Dry-run shows row counts before commit.
High Import inserts wrong-parsed rows en masse Pre-flight report reviewed by partner before commit. --dry-run is default; --commit is opt-in with confirm token.
Med Rollback is hard legacy_import_run_id per-row + legacy_import_runs table → one-statement rollback.
Med Senior/PWD discount logic ships incomplete Scoped: capture data + show on receipt now; GL contra-revenue account + BIR ledger view in follow-up. Books are clean because nothing is posted yet.
Low Reception finds typos / duplicates in legacy data Existing migration-0025 merge tool handles dupes. Patient-edit page handles typos. Confirm-DOB badge surfaces missing data naturally.
Low Lab results pipeline breaks on null DOB Already handled — verified at src/lib/results/types.ts:190-243. Range-match falls back to unbanded ranges when DOB is null.
Low pg_trgm extension not available Available by default on Supabase Postgres. Migration uses create extension if not exists.

11 · Out of scope

Explicit deferrals — captured here so they aren't forgotten

Accounting follow-ups

Senior/PWD discount GL bridge — contra-revenue account 4900-Discounts-SeniorPWD. Books aren't being posted in beta; safe to follow up after.

Senior/PWD BIR audit ledger view — admin report listing every discounted transaction with ID number for monthly BIR submission.

Data-quality tooling

Promote referring-physician strings → physicians table — admin tool to convert frequent free-text values into proper FK relationships. Free-text is fine for first weeks.

Delivery integrations

Email & SMS result delivery — result_release_pref is captured now; delivery hooks fire later when SMTP / Semaphore (PH SMS) integrations ship.

Analytics

Top-referrers admin dashboard chart — partner-facing analytics. Data captured now (referring_physician, referral_source) makes this a small follow-up.

Patient portal

Patient-portal access for legacy patients — no visits and no PINs. Covered by the separate Google-OAuth project paused earlier in this session.
