# 12.6 — Payroll, Attendance, Payslips (Design)

**Author:** ianjamila + Claude (brainstorm 2026-05-18)
**Status:** Approved by user (full scope, no cuts)
**Predecessors:** 12.1 (GL primitives), 12.2 (GL bridge), 12.C (Staff advances subledger, cash drawer)
**Successor (dep):** 12.B (history import of payroll lines from 2023→2026 mastersheet)
**Target tag:** `v1.8.0`

---

## 1. Goal

Replace the manual-spreadsheet payroll workflow at DRMed with an in-app system that:

1. Ingests ZKTeco DTR exports (one CSV per cutoff).
2. Computes semi-monthly payroll for 5 staff employees per Philippine labor law: daily-rate basic pay, allowances, OT (slip-gated), night differential, holiday pay, tardiness deductions, perfect-attendance bonus, statutory contributions (SSS/PhilHealth/Pag-IBIG, employee + employer shares), withholding tax on compensation, 13th-month monthly accrual + December payout.
3. Manages employee leave balances (VL + SL) per the company contract — entitlement, monthly accrual, manual grants, period usage, year-end expiry, March 31 VL carry-over.
4. Settles outstanding `staff_advances` via payroll deduction (drawing down the subledger we shipped in 12.C).
5. Pays employees via cash (from `/cash-drawer`) or bank transfer per employee preference.
6. Posts the resulting JEs through the bridge-trigger pattern established in 12.2.
7. Emits a payslip PDF per employee per run; employees can view their own via `/staff/payslips`; admin can view all.
8. Lets admin maintain SSS/PhilHealth/Pag-IBIG/WT bracket tables in-app so the system stays current as government rates change.

**Out of scope (Future Phase 2):** BIR form generation (1601-C, 1604-CF, 2316), bank batch transfer file generation (admin records each bank transfer manually in v1), employee self-service OT slip submission, employee self-service leave request submission, doctor compensation (handled in 12.5).

**Critical-fix corrections from the post-design think-harder pass:**

| # | Fix | Notes |
|---|-----|-------|
| CF1 | `employees.monthly_salary_credit_php` is a **stored** field, not computed each period. Defaults to `daily_rate × 26` at create time; admin can override. All SSS/PhilHealth/Pag-IBIG lookups use this field, divided by 2 per semi-monthly cutoff. | Standard PH practice for daily-rate workers; prevents contribution oscillation with attendance variance. |
| CF2 | Cash-payout workflow has a `payout_status` ('pending'/'paid'/'voided') on `payroll_employee_runs`. Finalisation posts ONLY the gross-up JE. Each employee's payout JE fires when the "Mark paid" action is taken (on/after pay date), which is when an `eod_cash_adjustment(kind='salary_payout')` is created (cash) or a direct bank JE is posted. Drawer count drops at actual payout time, not finalisation time. | Avoids the "drawer lies all weekend" failure mode. |
| CF3 | December Run #1's gross-up JE has TWO posting groups: (a) standard accrual for Dec 1-15 work; (b) annual 13th-month payout (`DR 2350 / CR 2360`) for the YTD accumulated balance. Net pay on Dec 20 paychecks includes regular Dec 1-15 net + entire 13th-month. | Cleanly closes out the 13th-month payable for the year. |
| CF4 | Pipeline caps deductions so `net_pay ≥ 0`. Order of deduction priority when capping: 1. Statutory (SSS/PhilHealth/Pag-IBIG/WT) — always paid in full; 2. Tardiness — always full; 3. Staff_advance settlement — caps at remaining net; 4. Other deductions (loans, manual) — caps in entry order. Excess loan/advance flows to next period automatically. | Prevents negative paychecks. |
| CF5 | Leave management uses an event-sourced `employee_leave_records` table: `(employee_id, kind, record_kind, days_delta, effective_date, expiry_date, period_id, reason)`. Balance = `Σ days_delta` where rows have `effective_date ≤ now()` and `(expiry_date IS NULL OR expiry_date > now())`. SQL function `employee_leave_balance(employee_id, kind, as_of_date)`. | Captures the contract's accrual + grant + usage + expiry rules cleanly. |
| CF6 | Shifting-schedule absence detection: for fixed-schedule employees, "scheduled but absent" is auto-detected by intersecting `employees.rest_days` with `payroll_dtr_rows`. For `shifting_5of6_mon_sat`, the schedule is *defined by* the DTR — admin manually flags absences during run review if needed. | Removes ambiguity for shifting workers. |
| CF7 | Doctors are **NOT** in `employees`. They have their own table managed by 12.5 (compensation: PF / rent / none). Explicit callout. | Prevents scope creep. |

**Workflow-default confirmations:**

| # | Default | Notes |
|---|---------|-------|
| WD1 | **Mid-period rate change**: only takes effect at the NEXT period boundary. No effective-dating table; admin updates `employees.basic_daily_rate_php` directly, the change is captured in `audit_log`, and the new rate applies starting the next run. Document this rule in the UI. |
| WD2 | **Audit log volume**: log `payslip.generated` and `payslip.downloaded`. Do NOT log `payslip.viewed` (would pollute audit log with every page visit). |
| WD3 | **Earning/deduction line labels are employee-visible** on the payslip. UI helper text reminds admin to keep labels brief and professional. |
| WD4 | **OT slips use a dedicated table** with request + approval workflow (`payroll_ot_slips`). Admin enters slips in v1 (no employee self-service yet). Slip status: pending / approved / rejected. Only approved slips contribute to OT pay in run computation. |

## 3. Architecture overview

Same "operational bridge" pattern as 12.2 (payments → JE) and 12.C (cash adjustments → JE).

```
┌──────────────────────────────────────────────────────────────────────┐
│  Admin UI (Server Components + Server Actions)                       │
│  /staff/admin/payroll/{employees,periods,runs,dtr,rates,holidays,…}  │
└──────────┬────────────────────────────────────────────────┬──────────┘
           │                                                │
           ▼                                                ▼
┌──────────────────────────┐                  ┌─────────────────────────┐
│  Server Actions          │                  │  Computation engine     │
│  (Zod-validated)         │                  │  (pure TS functions)    │
└──────────┬───────────────┘                  └────────────┬────────────┘
           │                                               │
           ▼                                               ▼
┌────────────────────────────────────────────────────────────────────┐
│  Postgres (Supabase)                                               │
│  • employees, allowances, periods, runs, employee_runs             │
│  • earning_lines, deduction_lines, ot_slips, dtr_imports, dtr_rows │
│  • leave_records, holidays, contribution_brackets, wt_brackets     │
│  • RLS: admin-only on most; employee can read own employee_runs    │
│  • Bridge triggers on payroll_runs.status='finalised' and on       │
│    payroll_employee_runs.payout_status='paid' fire the JEs         │
└────────────┬───────────────────────────────────────────────────────┘
             │
             ▼
┌────────────────────────────────────────────────────────────────────┐
│  GL: journal_entries + journal_lines (from 12.1 / 12.2)            │
│  Subledger writes back to: staff_advances (12.C draw-down)         │
│  Drawer integration via: eod_cash_adjustments (12.C, salary_payout) │
└────────────────────────────────────────────────────────────────────┘
```

The compute engine is **pure TypeScript** (deterministic, testable in isolation): given a `payroll_run_id`, it reads inputs (DTR rows, allowances, OT slips, brackets, leave records, prior staff_advances balance) and writes results to `payroll_employee_runs` + `payroll_earning_lines` + `payroll_deduction_lines`. The DB layer is responsible for JE posting via bridge triggers — same pattern as `bridge_cash_adjustment_insert` from 12.C.

### 4.1 `employees`

```sql
create table public.employees (
  id                            uuid primary key default gen_random_uuid(),
  staff_profile_id              uuid unique not null references public.staff_profiles(id) on delete restrict,
```

**Loan workflow (Q21 (b))**: three-state — `requested` → `approved` → `active` → `paid_off` / `written_off` / `voided`.

- **requested**: admin creates the loan on behalf of the employee, status='requested'. No effect on payroll yet.
- **approved**: admin clicks "Approve". Sets `approved_at`, `approved_by`, `approval_notes`. Still no payroll deduction.
- **active**: admin marks "Disbursed" after handing over the cash (which is recorded separately via `/cash-drawer` as `kind='salary_advance'` for cash payouts, or as a direct bank JE for bank-paid disbursements). Sets `disbursed_at`, `disbursed_by`, `start_period_id` = next open period. The first amortization fires on the next run.
- **paid_off / written_off / voided**: terminal states. `paid_off` auto-set when outstanding_balance hits 0. `written_off` and `voided` are admin actions.

The pipeline auto-creates a `payroll_deduction_lines(kind='loan_amortization', loan_id=L.id, amount=L.amortization_per_period_php)` for each `active` loan during run compute, and decrements `outstanding_balance_php` on finalisation. When balance hits 0, status flips to `paid_off`.

**Entitlement seeding rules** (applied by an `apply_leave_entitlements(p_year int)` admin function, runnable via Server Action):

- **SL**: every regular employee gets a `record_kind='entitlement', days_delta=5, effective_date='YYYY-01-01' (or regularization_date if mid-year), expiry_date='YYYY-12-31'` row for `kind='SL'`.
- **VL**: every employee at 1-year+ tenure gets 12 monthly `record_kind='entitlement', days_delta=annual_entitlement/12` rows for `kind='VL'`, with `effective_date` on the 1st of each month and `expiry_date='(YYYY+1)-04-01'` (the Mar 31 carry-over deadline).

**Termination handling (Q22 (a) + override)**: when an admin sets `employees.termination_date`, the system auto-inserts a `record_kind='expiry', days_delta=-(current_VL_balance + current_SL_balance), effective_date=termination_date, reason='termination'` row per leave kind. This forfeits outstanding leave per contract section 6.2. Admin can override before the next run is finalised by adding manual `cash_conversion` records (with positive `days_delta` to undo the expiry + a corresponding `payroll_earning_lines(kind='one_time_bonus', label='Leave cash conversion on termination', amount_php=balance × daily_rate)` line on the employee's final run).

## 5. Run lifecycle (state machine)

```
                  ┌───────────────────┐
                  │  draft (initial)  │
                  └─────────┬─────────┘
       ┌───────────────────┴───────────────────┐
       │ admin: import DTR, edit OT slips,     │
       │ add ad-hoc earning/deduction lines    │
       │                                       │
       ▼                                       │
[compute_payroll_run_action]                   │
       │ writes results to payroll_employee_runs│
       ▼                                       │
                  ┌───────────────────┐        │
                  │     computed      │────────┤
                  └─────────┬─────────┘        │
                            │ admin reviews,   │
                            │ may add lines    │
                            │ + recompute      │
                            │                  │
                            ▼                  │
                  ┌───────────────────┐        │
                  │     finalised     │        │ (any of these
                  └─────────┬─────────┘        │  steps fail back
                            │                  │  to draft if no
                            │ admin: "Mark     │  payouts started)
                            │ paid" per emp    │
                            ▼                  │
                            (per-employee transitions)
                            payout_status: pending → paid
                            │
                            ▼ (if all employees paid)
                  ┌───────────────────┐
                  │  done (implicit)  │
                  └───────────────────┘

  Admin VOID action (only valid when payout_status='pending' for ALL employee_runs):
       finalised → voided (reverses gross-up JE)
       voided   → may re-create a new run for the same period (the voided run stays for audit)
```

**State transitions:**

- `create` (admin): inserts `payroll_runs(period_id, status='draft')` and one `payroll_employee_runs` row per active employee with all zeros.
- `import_dtr` (admin, allowed in `draft`/`computed`): inserts `payroll_dtr_imports` + `payroll_dtr_rows`; flips any prior import's rows to `superseded`. **If the run was `computed`, also flips it back to `draft`** (Q23 (a)) — prevents stale computed numbers from outdated DTR. Admin must explicitly re-click "Compute" to refresh.
- `compute` (admin, allowed in `draft`/`computed`): runs the pipeline (§6); updates `payroll_employee_runs` summary numbers; flips run status to `computed`.
- `add_line` (admin, allowed in `draft`/`computed`): inserts `payroll_earning_lines` or `payroll_deduction_lines`; admin re-runs compute after to refresh totals.
- `finalise` (admin, allowed only in `computed`): inserts the gross-up JE via the bridge trigger; flips run status to `finalised`. NO per-employee payouts yet.
- `mark_paid` (admin, per-employee, allowed only in `finalised`): updates `payroll_employee_runs.payout_status='paid'`, sets `paid_at, paid_by`. For cash-paid: inserts `eod_cash_adjustments(kind='salary_payout')` which bridges to a JE via 12.C trigger; sets `payout_cash_adjustment_id` + `payout_je_id`. For bank-paid: posts a direct JE: `DR 2360 Salaries Payable / CR 1020 Cash in Bank — BPI`; sets `payout_je_id`.
- `void` (admin, allowed only in `finalised` with ALL `payout_status='pending'`): reverses the gross-up JE; flips run status to `voided`.
- `reopen_voided` (admin, allowed only in `voided`): flips run status back to `draft`. The original gross-up JE + its reversal stay in the ledger for audit trail; recomputing + re-finalising posts a NEW gross-up JE. No JE side effect from this transition itself.
- `void_employee_payout` (admin, per-employee, allowed when employee_run.payout_status='paid'): flips `payout_status: paid → voided`. Reverses that employee's payout JE only (either by voiding the linked `eod_cash_adjustments` row — 12.C bridge handles the reversal — or by posting a reversal JE for bank payouts). Does NOT unlock the parent run (the per-employee void is a corrective action, not a re-opening of the run). After voiding, admin can either re-trigger `mark_paid` for that employee (e.g., reception found the missing cash) or leave them as `voided` and pay them via the next period as a `manual_adjustment` line.

**Locking rule:** Once any `payroll_employee_runs.payout_status='paid'`, the run-level state is locked. Admin cannot recompute the run, cannot add/remove lines, cannot run-level void. Per-employee `void_employee_payout` IS allowed (it's a corrective action for that specific paid employee). Substantive corrections affecting earnings/deductions flow to the next period as `manual_adjustment` lines.

## 6. Computation pipeline

The pipeline is a pure TypeScript function called from `computePayrollRunAction`. Reads from DB, computes, writes to DB in a single transaction.

For each `employee` in `payroll_employee_runs` for the run:

### 6.1 Pull attendance facts

- `dtr_rows` = `SELECT * FROM payroll_dtr_rows WHERE import_id = (latest import for period_id) AND employee_id = E AND status = 'parsed'`
- For each work_date in `[period.period_start, period.period_end]`:
  - If row exists in `dtr_rows`: marked `present`.
  - Else: check schedule.
    - For `fixed_5day_mon_fri`: scheduled if day-of-week ∈ Mon-Fri.
    - For `fixed_6day_mon_sat`: scheduled if day-of-week ∈ Mon-Sat.
    - For `shifting_5of6_mon_sat`: NOT auto-flagged; admin manually adjusts in run review.
  - Check `payroll_holidays` for the date; classify as regular/special.
  - Check `employee_leave_records` for an approved leave covering this date (kind='VL' or 'SL').

### 6.2 Categorise each day

For each calendar day in the period:

| Day state | Definition |
|---|---|
| `present_regular` | DTR row exists, day is NOT a holiday |
| `present_regular_holiday` | DTR row exists, day IS a regular holiday |
| `present_special_holiday` | DTR row exists, day IS a special non-working holiday |
| `unworked_regular_holiday` | NO DTR row, day IS a regular holiday, day was scheduled |
| `unworked_special_holiday` | NO DTR row, day IS a special holiday, day was scheduled (pays 0 per PH rule) |
| `vl_used` | NO DTR row, employee has approved VL covering this date |
| `sl_used` | NO DTR row, employee has approved SL covering this date |
| `unpaid_absent` | NO DTR row, scheduled day, no leave |
| `not_scheduled` | NO DTR row, not a scheduled work day (rest day or weekend) — no pay impact |

### 6.3 Compute earning components

**OT overage warning** (per Q28 — show warning when DTR overage exists without an approved slip): during compute, for each DTR row with `total_hours > standard_workday_hours`, if there's no `payroll_ot_slips(status='approved', employee_id=E, work_date=date)` row, the system records a diagnostic on the `payroll_employee_runs` row: `ot_overage_unpaid_minutes_total` (new column) = sum of unbilled OT minutes. The Run Review UI displays a yellow banner per employee: "X hours of overage detected without an OT slip — currently unpaid. Create an OT slip if appropriate." This is non-blocking; finalise is still allowed.

```
basic_pay = (count(present_regular) + count(present_regular_holiday) + count(present_special_holiday)
             + count(vl_used) + count(sl_used)) × basic_daily_rate

allowances_total = Σ (active_allowance.daily_amount × count(present_regular + present_*_holiday + vl_used + sl_used))
                   for each active allowance row in the period's date range

hourly_rate = basic_daily_rate / 8

ot_pay = Σ over approved OT slips with work_date in period:
           if work_date is on a rest day: hours × hourly_rate × ot_rate_rest_day × ot_rate_regular_day
           else:                          hours × hourly_rate × ot_rate_regular_day

night_diff_pay = Σ over DTR rows: nd_hours(time_in, time_out) × hourly_rate × night_diff_premium_rate
  where nd_hours = hours falling within [22:00, 06:00 next day] window

holiday_pay = Σ:
  count(present_regular_holiday)   × basic_daily_rate × (holiday_pay_regular_worked - 1)
  + count(unworked_regular_holiday) × basic_daily_rate × holiday_pay_regular_unworked
  + count(present_special_holiday)  × basic_daily_rate × (holiday_pay_special_worked - 1)
  + count(unworked_special_holiday) × basic_daily_rate × holiday_pay_special_unworked
  (the "-1" backs out the basic_pay already counted above for worked days)

incentives_total = Σ amount_php from payroll_earning_lines (any kind)

perfect_attendance_bonus = perfect_attendance_bonus_php (₱1000) if:
  tardiness_count == 0 AND days_unpaid_absent == 0 AND missing_punch_days == 0
  ELSE 0
```

### 6.4 Tardiness

For each `present_*` day:

```
scheduled_start = (scheduled_start_hour, scheduled_start_minute) in Asia/Manila for that date
minutes_late = max(0, time_in - scheduled_start)
if minutes_late > 0: tardiness_count += 1
total_minutes_late += minutes_late
```

```
tardiness_deduction = total_minutes_late × tardiness_per_minute_php
                    + (if tardiness_count ≥ tardiness_threshold_for_halfday_deduction: basic_daily_rate / 2 else: 0)
```

### 6.5 13th-month accrual

```
thirteenth_month_accrual = basic_pay × (1/12)
```

In December Run #1 (period_start = Dec 1), additionally compute payout:

```
thirteenth_month_payout = SUM(thirteenth_month_accrual) for THIS employee from all prior runs YTD + this run's accrual
                        = approximate annual basic_pay / 12

(Implementation: sum payroll_employee_runs.thirteenth_month_accrual_php for this employee where
 run's period_start in (Jan 1, Dec 15] of current year)
```

The payout zeroes out the `2350 13th-Month Payable` balance for this employee.

### 6.6 Gross pay

```
gross_pay = basic_pay + allowances_total + ot_pay + night_diff_pay + holiday_pay
          + incentives_total + perfect_attendance_bonus + thirteenth_month_payout
```

(Note: `thirteenth_month_accrual` is NOT included in `gross_pay` — it's a deferred liability not yet earned by the employee. Only `thirteenth_month_payout` enters gross.)

### 6.7 Statutory deductions

Look up brackets where `monthly_salary_credit_php` falls in `[min, max]` for each kind, using brackets active on `period.period_end`:

```
sss_ee_monthly      = bracket(kind='sss', msc=employee.msc).employee_share
sss_ee              = sss_ee_monthly / 2   -- semi-monthly half
sss_er              = bracket(kind='sss', ...).employer_share / 2

philhealth_ee       = bracket(kind='philhealth', ...).employee_share / 2
philhealth_er       = bracket(kind='philhealth', ...).employer_share / 2

pagibig_ee          = bracket(kind='pagibig', ...).employee_share / 2
pagibig_er          = bracket(kind='pagibig', ...).employer_share / 2
```

### 6.8 Withholding tax on compensation

```
taxable_compensation = gross_pay - (sss_ee + philhealth_ee + pagibig_ee) - (Σ allowances marked is_taxable=false × days_present)
wt_compensation      = wt_bracket_lookup(taxable_compensation).base_tax
                     + (taxable_compensation - wt_bracket_lookup(taxable_compensation).taxable_min)
                       × wt_bracket_lookup(taxable_compensation).marginal_rate
```

WT brackets are semi-monthly per BIR.

### 6.9 Staff advance settlement

```
outstanding_advance = SELECT SUM(outstanding_balance_php) FROM staff_advances WHERE staff_id = employee.staff_profile_id AND status = 'outstanding'
cap_pct = accounting_settings.value_php WHERE key = 'staff_advance_settlement_max_pct'   -- default 0.50, admin-editable
max_advance_deduction = (gross_pay - sss_ee - philhealth_ee - pagibig_ee - wt_compensation - tardiness_deduction) × cap_pct
staff_advance_settlement = min(outstanding_advance, max_advance_deduction)
```

The 50% cap is editable via `/staff/admin/payroll/settings` (per regulation/policy changes).

### 6.10 Other deductions

```
other_deductions_total = Σ amount_php from payroll_deduction_lines (kind='loan_amortization' auto-created from active loans, plus manual)
```

**Loan amortization auto-population — delete-and-recreate on every compute** (per Q29 (a)):

```
-- At the START of each compute pass:
DELETE FROM payroll_deduction_lines
  WHERE employee_run_id = E.id AND kind = 'loan_amortization';

-- Then re-insert from current active loans, ordered oldest first:
INSERT INTO payroll_deduction_lines (employee_run_id, kind, label, amount_php, loan_id)
SELECT E.id, 'loan_amortization',
       'Loan amortization — ' || coalesce(l.notes, 'loan #'||substr(l.id::text,1,8)),
       l.amortization_per_period_php,
       l.id
  FROM employee_loans l
  WHERE l.employee_id = E.employee_id
    AND l.status = 'active'
    AND l.start_period_id IS NOT NULL
    AND (SELECT period_start FROM payroll_periods p WHERE p.id = l.start_period_id) <= period.period_start
  ORDER BY l.created_at ASC;   -- oldest loan first → caps fall on newer loans
```

This is the source-of-truth approach: any admin override goes in as a SEPARATE `manual_adjustment` deduction line (with its own label like "Loan partial waiver" or "Adjustment to loan #X"). Never edit the auto-generated line directly — it'll be wiped on next recompute. Document this in the UI.

On finalise, the bridge updates `loan.outstanding_balance_php -= amount`; if balance hits 0, status → 'paid_off'.

**Mid-period loan disbursement (per Q31 (a))**: by default, when admin clicks "Mark disbursed" on a loan, `start_period_id` is auto-set to the CURRENT open period — so the loan deducts in the period during which it was disbursed. Admin can override to a future period via a "Start deducting from..." dropdown on the disbursement modal (e.g., "wait until next period to start"). UI nudge: small note saying "By default deductions start this period; change if the employee needs a one-cycle grace."

### 6.11 Net pay (with cap)

```
deductions_priority_order = [
  sss_ee + philhealth_ee + pagibig_ee,   -- always full
  wt_compensation,                        -- always full
  tardiness_deduction,                    -- always full
  staff_advance_settlement,               -- caps at remaining
  other_deductions_total                  -- caps at remaining
]

remaining = gross_pay
for d in deductions_priority_order:
  if d ≤ remaining: remaining -= d
  else: d = remaining; remaining = 0

net_pay = remaining
```

For "other deductions" (loans, manual), the cap is applied in **insertion order** (oldest auto-loan first, since they're inserted ordered by `created_at ASC` per §6.10; manual lines fall after the loan lines). Any deduction line that gets capped sees its `amount_php` written back as the actual capped amount. Excess loan principal carries forward implicitly (loan's outstanding_balance is decremented by only the capped amount on finalise).

**Cap ordering rationale (Q30)**: oldest loan first means newer loans absorb the cap. The oldest loan is being paid down on schedule; only newer ones get partial-deduction events. This matches PH common practice — first loan in, first loan out.

### 6.12 Write results

Update `payroll_employee_runs` row with all computed numbers. Flip run status to `computed`.

## 7. JE shapes

### 7.1 Gross-up JE (posted on finalise)

```
Posting date = period.period_end (typical: 15th or last day of month)
Description  = "Payroll run <period_start> to <period_end>"
Source kind  = 'payroll_run'  (new je_source_kind enum value)
Source id    = payroll_runs.id

DR  6100 Salaries & Wages                 Σ basic_pay + allowances + ot + nd + holiday + incentives + perfect_attendance_bonus
DR  6121 Employer SSS Contribution        Σ sss_er
DR  6122 Employer PhilHealth Contribution Σ philhealth_er
DR  6123 Employer Pag-IBIG Contribution   Σ pagibig_er
DR  6124 13th-Month Pay Expense           Σ thirteenth_month_accrual
  CR  2300 SSS Payable                    Σ sss_ee + sss_er
  CR  2310 PhilHealth Payable             Σ philhealth_ee + philhealth_er
  CR  2320 Pag-IBIG Payable               Σ pagibig_ee + pagibig_er
  CR  2330 WT Compensation Payable        Σ wt_compensation
  CR  2350 13th-Month Payable             Σ thirteenth_month_accrual
  CR  1130 Staff Advances                 Σ staff_advance_settlement
  CR  2360 Salaries Payable               Σ net_pay
```

The bridge function `bridge_payroll_run_finalise` creates this entry in one transaction. Per the established pattern (12.2, 12.C): draft → lines → flip to `posted`.

**December Run #1 is now TWO separate JEs (Q25 (b) — better audit trail):**

1. The regular gross-up JE above (source_kind='payroll_run').
2. A **separate** annual 13th-month payout JE (source_kind='payroll_13th_month_payout' — additional new `je_source_kind` value), source_id = run.id. Posted in the same transaction as (1) but distinct journal_entries row:

```
DR  2350 13th-Month Payable        Σ annual thirteenth_month_payout (zeros out the YTD payable)
  CR  2360 Salaries Payable        Σ annual thirteenth_month_payout (added to each employee's net pay)
```

This keeps the regular monthly P&L recognition (in #1) cleanly separate from the year-end balance flush (in #2), so the ledger reads obvious during year-end audit.

### 7.2 Per-employee payout JE (posted on "Mark paid")

**Cash-paid employee:**
- Inserts an `eod_cash_adjustments(kind='salary_payout', amount=net, recorded_by=admin, contra_account_id=2360)` row.
- 12.C's `bridge_cash_adjustment_insert` trigger fires and posts: `DR 2360 Salaries Payable / CR 1010 Cash on Hand`.
- `payroll_employee_runs.payout_cash_adjustment_id` is set to the adjustment row id; `payout_je_id` is set to the resulting JE id.

**Bank-paid employee:**
- A new bridge function `bridge_payroll_payout_bank` posts: `DR 2360 Salaries Payable / CR 1020 Cash in Bank — BPI`.
- `payroll_employee_runs.payout_je_id` is set.

### 7.3 Void / reversal JEs

**Run-level void (`voidRunAction`)** — when admin voids a finalised run (only allowed if all employees `payout_status='pending'`):
- The bridge function `bridge_payroll_run_void` posts a reversal JE for the gross-up JE (debits ↔ credits flipped). Source kind = 'reversal'.
- If the run was a December Run #1, ALSO post a reversal for the 13th-month payout JE.
- `payroll_runs.gross_up_je_id`'s status flips to 'reversed' with `reversed_by` set.
- **Staff_advance auto-restore (Q24 follow-up answer)**: the bridge ALSO restores `staff_advances.outstanding_balance_php` for each row that was settled by this run. For each employee_run with `staff_advance_settlement_php > 0`, the bridge finds the corresponding `staff_advances` row(s) drawn down during finalisation and increments `outstanding_balance_php` back up by the settled amount. Status flips back to 'outstanding' if it had been auto-closed. Audit-log row records the restore.

**Per-employee payout void (`voidEmployeePayoutAction`)** — when admin voids one employee's payout while the run is otherwise locked:
- Reverses ONLY that employee's payout JE (the per-employee DR 2360 / CR 1010 or 1020 entry).
- For cash payouts: voids the linked `eod_cash_adjustments` row (12.C's `bridge_cash_adjustment_void` handles the reversal JE).
- For bank payouts: posts a direct reversal JE.
- `payroll_employee_runs.payout_status` flips to `voided`. `payout_je_id` and `payout_cash_adjustment_id` remain (audit trail).
- **The gross-up JE is NOT touched** — the wage expense and the liability remain recorded. The employee is still legitimately owed their net pay (sitting in 2360 Salaries Payable). Admin must either:
  - (a) Re-trigger `markEmployeePaidAction` on the same employee_run later (reception found the cash, or bank transfer was re-attempted), OR
  - (b) Post a manual JE (outside 12.6's automation) to clear the 2360 balance for this employee — e.g., `DR 2360 / CR <other-account>` — and document the reason in audit notes. The system does NOT auto-deallocate the gross-up portion because that would silently destroy the recognized wage expense (which IS valid — the employee earned the wage; the only thing that didn't happen was the payment).

This matches accountant practice: voiding a payment doesn't void the underlying obligation.

## 8. Bridge functions + guard triggers

### 8.1 `bridge_payroll_run_finalise()`

Fires on `payroll_runs.UPDATE` when `OLD.status != 'finalised' AND NEW.status = 'finalised'`.
- Computes summary totals from `payroll_employee_runs` for the run.
- Creates the gross-up JE in `journal_entries` (status='draft' → lines inserted → flipped to 'posted').
- Sets `payroll_runs.gross_up_je_id`.

### 8.2 `bridge_payroll_payout_bank()`

Fires on `payroll_employee_runs.UPDATE` when:
- `OLD.payout_status='pending' AND NEW.payout_status='paid' AND NEW.payment_method_used='bank'`.

Posts the direct bank JE. Sets `payout_je_id`.

For `cash` payouts, no trigger needed — the Server Action inserts the `eod_cash_adjustments` row which fires the existing 12.C bridge.

### 8.3 `bridge_payroll_run_void()`

Fires on `payroll_runs.UPDATE` when `OLD.status='finalised' AND NEW.status='voided'`.
- Inserts a reversal JE for `OLD.gross_up_je_id`.
- If the run is a December Run #1, also inserts a reversal for the 13th-month payout JE.
- Marks both original JEs as `reversed`.
- **Restores `staff_advances`**: for each `payroll_employee_runs` row with `staff_advance_settlement_php > 0`, finds the linked `staff_advances` row(s) and increments `outstanding_balance_php` back by the settled amount. If the row had been auto-closed (status='settled'), flips it back to 'outstanding'.

### 8.4 `bridge_payroll_13th_month_payout()`

Fires when finalisation transition completes AND the run's period covers Dec 1–15. Inserts the year-end 13th-month payout JE as a SEPARATE journal_entries row (per Q25 (b)).

### 8.5 Guard triggers

- **P0020** `payroll_run_finalise_requires_dtr`: blocks `status='finalised'` if any `payroll_employee_runs` row in the run has `scheduled_days != days_present + days_vl_used + days_sl_used + days_unpaid_absent + days_regular_holiday_worked + days_regular_holiday_unworked + days_special_holiday_worked + days_special_holiday_unworked`. (Sanity check that all scheduled days are accounted for. Note: an employee on full-period leave with VL+SL records covering every scheduled day passes this check — they're not missing DTR, they're entirely on leave.)
- **P0021** `payroll_run_locked_after_payout`: blocks any UPDATE on `payroll_runs.status` or new INSERTs to earning/deduction lines if any employee_run in that run is `payout_status='paid'`.
- **P0022** `employees_require_daily_rate`: blocks `INSERT INTO payroll_employee_runs` if the employee has `basic_daily_rate_php IS NULL`. (Caught earlier in app code, but DB guard for safety.)
- **P0023** `ot_pay_requires_slip`: blocks `payroll_employee_runs.ot_pay_php > 0` if no `payroll_ot_slips(status='approved')` rows exist for this employee in the period.
- **P0024** `staff_advance_settlement_within_outstanding`: blocks `payroll_employee_runs.staff_advance_settlement_php > current outstanding balance`. (Caught earlier in app code.)
- **P0026** `payroll_employee_runs_no_insert_after_finalise`: blocks `INSERT INTO payroll_employee_runs` when the parent `payroll_runs.status IN ('finalised','voided')`. Prevents accidentally adding a missed employee to a closed run; admin must void + reopen + recompute instead.
- **P0027** `payroll_run_finalise_requires_nonzero_gross`: blocks `status='finalised'` if the SUM of `payroll_employee_runs.gross_pay_php` over the run = 0. Empty runs are an admin mistake (forgot to upload DTR? all employees voided?) and produce an empty JE — block to force admin to either populate or delete the run.

## 18. Out of scope (Future Phase / 12.6.x)

Features intentionally excluded from 12.6 v1:

- **Doctor compensation** (PF / rent / none) — handled in **12.5**. The `employees` table is staff-only.
- **BIR form generation** (1601-C monthly, 1604-CF annual, 2316 per-employee, alphalist) — **Future Phase 2**. The system captures all the data needed to file these; v1 just doesn't render the forms.
- **Bank batch transfer file** — admin records each bank transfer manually in v1 by clicking "Mark paid". Future: generate a BPI-format CSV admin uploads to bank.
- **Employee self-service OT slip submission** — admin enters all OT slips in v1. The `payroll_ot_slips` table supports a future employee-facing form.
- **Employee self-service leave request submission** — admin records leave usage in v1. Contract section 6.3 mandates email-based requests; that workflow stays out-of-app in v1.
- **Per-patient incentive auto-compute from `test_requests`** — admin enters incentives as free-form line items in v1 (per Q20). Future: rule-based auto-compute given `test_requests.medtech_id` denormalization.
- **Regularization workflow** — admin manually sets `employees.regularization_date` when probation ends. Future: calendar reminders, performance appraisal checklist.
- **Mid-year hire/termination pro-ration** — naturally handled by daily-rate compensation; no special case needed. Just confirms.
- **Cash conversion of expired VL** — contract section 6.2 allows it when employer-blocked. Admin records the conversion as a manual `payroll_earning_lines` (kind='one_time_bonus') + `employee_leave_records(record_kind='cash_conversion', days_delta=-N)` row.
- **Holiday work pre-approval workflow** — admin records who worked on holidays via DTR; no separate approval step in v1.
