Phase 12 · Sub-project 12.A

HMO History Import

Backfill three years of HMO claims captured manually in the DR MED MASTERSHEET workbook (Dec 2023 → 2025-12-31) into the 12.3 subledger so per-provider AR is correct in the GL, every claim is queryable in the aging views, and the “claims we forgot to bill” audit answer finally lands.

01 · Executive summary

Three concrete deliverables

12.3 shipped the subledger schema and the operational UI. 12.A loads the three-year backlog of claims into that schema, posts one opening-balance JE per provider at 2025-12-31, and surfaces the historical stuck-claim backlog through the existing 12.3 detection views.

In scope

Per-claim subledger backfill · synthetic op-rows (patients, visits, test_requests, payments) gated by is_historical flags · one balanced opening JE per provider at cutover · preview + commit UI at /staff/admin/accounting/hmo-history with header-asserting XLSX parser, alias-mapping flow, and HMO REFERENCE variance gate.

Out of scope · deferred

Historical P&L data & cash receipts (12.B) · VAT / withholding tax (Future Phase 1) · patient-bill / write-off resolution of historical stuck claims (admin uses existing 12.3 UI after import) · physician table linkage for historical consults (future phase).

LAB SERVICE column map

Decoded from sharedStrings.xml via XLSX archive inspection — header text drives the canonical field map at parse time, not column position.

-- 31 columns. 12.A reads:
A  DATE              → source_date (Excel serial)
D  PATIENT NAME      → split "Last, First" into last_name_raw / first_name_raw
E  HMO (YES/NO)      → filter (only YES imports)
F  PROVIDER          → resolve via hmo_provider_aliases / exact match
G  APPROVAL DATE     → hmo_approval_date (unbilled-but-approved signal)
H  SERVICE           → resolve via hmo_service_aliases / exact match
N  FINAL PRICE       → billed_amount (post-discount)
V  REFERENCE/TRACKING → reference_no (batches by this + W)
W  DATE SENT         → submission_date
Y  OR #              → or_number (groups rows into synthetic payments)
Z  DATE PAID         → payment_received_date

DOCTOR CONSULTATION is a 25-column subset; SERVICE contains the consulting doctor's last name. Header detection at row 1 drives the mapping for both tabs.

04 · Architecture

End-to-end flow

One admin page, four Server Actions, one SQL function that does the heavy lifting in a single transaction with an advisory lock and a session-scoped bridge bypass.

1 · Upload
Admin drops the workbook at /staff/admin/accounting/hmo-history. SHA-256 of bytes → file_hash on the run.
↓
2 · parseWorkbookAction
Read row 1; assert canonical headers per tab. Stream rows; filter E='YES' and DATE ≤ cutover. Insert into hmo_history_staging.
3 · validateRunAction
Normalize names / dates / amounts. Resolve provider + service via aliases. Compute visit_group_key + content_hash. Cross-row scans. Parse HMO REFERENCE aging.
↓
4 · Preview & gates
7 sections: counts · unmapped providers · unmapped services · reconciliation · errors · warnings · commit footer. Commit disabled while any blocker exists.
↓
5 · commit_hmo_history_run
Single SQL function · single transaction · pg_advisory_xact_lock · SET LOCAL app.skip_bridge_historical = 'true' · upsert patients/visits/test_requests/batches/items/payments/allocations · unset bypass · post per-provider opening JEs · stamp summary + audit.
↓
6 · 12.3 surfaces light up
/staff/admin/accounting/hmo-claims, v_hmo_ar_aging, v_hmo_stuck now answer the full historical question. Stuck claims surface as synthetic draft batches.
05 · Schema

Migration 0035 deltas

Three op-tables get an is_historical flag with partial indexes for fast operational queries. Four new tables hold the staging and alias state. Two bridge functions get a skip-flag check. No new payment method, no new services.kind, no new CoA codes.

Op-table flags & CHECK relaxation

ALTER TABLE public.patients
  ADD COLUMN is_historical boolean NOT NULL DEFAULT false;
ALTER TABLE public.visits         ADD COLUMN is_historical boolean NOT NULL DEFAULT false;
ALTER TABLE public.test_requests  ADD COLUMN is_historical boolean NOT NULL DEFAULT false;

-- Birthdate is required for operational patients; allowed NULL for historical
-- (the workbook has no DOB). Existing rows are unaffected.
ALTER TABLE public.patients ALTER COLUMN birthdate DROP NOT NULL;
ALTER TABLE public.patients
  ADD CONSTRAINT patients_birthdate_required_when_not_historical
  CHECK (is_historical = true OR birthdate IS NOT NULL);

CREATE UNIQUE INDEX idx_patients_historical_one_per_name
  ON public.patients (last_name, first_name) WHERE is_historical = true;

New tables

TablePurposeLifetime
hmo_import_runsOne row per dry-run or commit. Carries file_hash, summary jsonb, committed_at, override reason.Append-only · audit-grade
hmo_history_stagingOne row per parsed workbook row. Holds raw + normalized + resolved + validation_errors jsonb. Status flips parsed → validated → committed.Permanent (no auto-cleanup)
hmo_provider_aliasesPersistent string → provider_id mapping. Survives across runs so re-imports skip the alias dance.Permanent
hmo_service_aliasesPersistent string → service_id mapping. Covers both lab service strings and doctor last names.Permanent

Subledger extension

ALTER TABLE public.hmo_claim_items
  ADD COLUMN hmo_approval_date date;  -- workbook col G; "approved but unbilled" signal

ALTER TABLE public.hmo_claim_batches
  ADD COLUMN import_run_id     uuid REFERENCES public.hmo_import_runs(id),
  ADD COLUMN historical_source text;  -- 'mastersheet:LAB SERVICE' etc.

ALTER TYPE public.je_source_kind ADD VALUE IF NOT EXISTS 'hmo_history_opening';

Bridge bypass

Both INSERT-fired JE bridge functions get a session-flag guard at the top. The UPDATE-fired test-request release bridge doesn't need a bypass — historical test_requests are INSERTed directly with status='released', so the UPDATE path never fires.

Bypass targets

bridge_payment_insert() — AFTER INSERT on payments
bridge_hmo_claim_resolution_insert() — AFTER INSERT on hmo_claim_resolutions (defensive; no resolutions inserted by 12.A)

No bypass needed

trg_bridge_test_request_released — AFTER UPDATE only
trg_bridge_test_request_cancelled — AFTER UPDATE only
trg_test_requests_payment_gate — BEFORE UPDATE only

08 · Reconciliation

The 5% variance gate

The single biggest quality control. Before any GL change, the importer reads the HMO REFERENCE tab's per-month aging block, computes ending balance per provider at cutover, and compares it to the per-row staging total. A divergent provider blocks commit. 

-- per provider
wb_ending     = HMO REFERENCE 'ENDING BALANCE' @ cutover month
staged_ar     = SUM(billed - paid - patient_billed - written_off)
                  WHERE provider_id_resolved = p
variance_pct  = (staged_ar - wb_ending) / NULLIF(wb_ending, 0)
Green band
≤ 1%variance
No action needed
Yellow band
1–5%variance
Investigate but commit allowed
Red band
> 5%variance
Blocks commit · override reason required

10 · Risks & dependencies

What could go wrong

Each risk is mitigated by a specific design choice in the spec. Severity reflects "what happens if this fails" — critical means the GL or reception experience is harmed.

Reconciliation variance > 5% with no obvious causecritical
Mitigation: Block commit; require override reason stored on the run for audit. The single hardest-to-detect failure mode if absent.
Bridge bypass leaks past the commit transactioncritical
Mitigation: SET LOCAL is transaction-scoped; the commit function explicitly unsets the flag before posting opening JEs as belt-and-braces; operational requests in other connections never see the flag.
Service-role client used from a place that ships to the browsercritical
Mitigation: All six Server Actions are 'use server'; the admin client is imported only from server modules. CLAUDE.md hard rule enforced.
Workbook header drift breaks parsing silentlywarn
Mitigation: Parser asserts the canonical header set at row 1; fails fast with a precise message ("Expected 'PATIENT NAME' at column D, found 'PATIENT NAMEE'").
Patient false-merge (two real people sharing a name)warn
Mitigation: Affects per-patient drill-down only; AR balance is unaffected. Documented in the preview's PII acknowledgment.
Concurrent imports collidewarn
Mitigation: pg_advisory_xact_lock(hashtext('hmo_history_import')) at the start of commit_hmo_history_run. Second concurrent commit waits or aborts cleanly.
Reception sees thousands of historical patientsmanageable
Mitigation: is_historical=false filter on the four list pages enumerated in §5A; partial indexes back the filter. Admin pages intentionally show both.
Workbook keeps changing under usmanageable
Mitigation: Admin uploads a frozen snapshot; file_hash records which version was imported. Content-hash idempotency makes re-uploads safe.
Re-running the importer changes existing patientsmanageable
Mitigation: Patient upsert uses ON CONFLICT (last_name, first_name) WHERE is_historical = true DO NOTHING — never touches operational patients.

11 · Future

Deferred decisions

Things consciously left out of 12.A. Each has a "revisit when…" trigger so they don't get lost.

services.is_historical

Not added in 12.A. Revisit if reception accidentally picks "Doctor Consultation — Dr. Gayo" as a service for a fresh visit. Adding the column then is a small migration plus a service-picker filter.

Voiding an entire import

Out of scope for the UI. Manual procedure documented at the post-commit page: (a) void each opening JE via the 12.1 JE UI, (b) set committed_at = NULL + status='discarded', (c) DELETE synthetic op-rows by is_historical = true scoped to the run's content_hashes, (d) audit-log the rollback.

HMO REFERENCE aging tab parsing

Exact row/column layout of the per-month aging block is finalized during D2 against a fresh read of the workbook. If layout drifts (header text changed), parser fails fast like the transaction tabs.

Doctor consultation → physicians linkage

For 12.A doctors stay as service rows with kind='doctor_consultation'. A later phase can backfill test_requests.physician_id once a physicians table has a clean mapping.
