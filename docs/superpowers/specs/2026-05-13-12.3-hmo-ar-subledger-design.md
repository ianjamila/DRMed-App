# 12.3 — HMO AR Subledger + Unbilled / Stuck-Claim Detection — Design

**Status:** Draft, pending user review.
**Project:** Accounting + Phase 12 (`docs/superpowers/specs/2026-05-11-accounting-project-overview.md`)
**Prerequisites:** 12.1 (GL foundation, v1.2.0), 12.2 (Op→GL bridge, v1.3.0)
**Author:** ianjamila (admin / CoA owner)
**Date:** 2026-05-13

---

## 1. Goal

Make `1110 AR HMO` (currently a single account credited by the 12.2 bridge) into a real, per-provider subledger with claim-level lifecycle tracking. Three concrete deliverables:

1. **Per-provider HMO AR balance** answerable on demand (e.g., "Maxicare owes us ₱X").
2. **Unbilled-claim detection** — released-to-HMO `test_requests` that haven't made it into a claim batch yet.
3. **Stuck-claim detection** — claims submitted to the HMO that are past the contractual `due_days_for_invoice` without payment.

12.3 ships the schema, the bridge integration, and the admin UI. Historical backfill (2023→present) lands in 12.A using 12.3's schema as the target.

## 2. Decisions locked in (from brainstorm 2026-05-13)

| Decision | Choice | Why |
|---|---|---|
| CoA shape | **Single `1110 AR HMO` + subledger** (no per-provider sub-accounts) | Bridge already credits 1110; per-provider sub-accounts would force a 12.2 rewrite and bloat CoA by ~11 codes for no analytical gain. |
| Claim grain | **Per-provider-per-submission batch** | Matches real PH practice (clinic mails one SOA covering many patients per provider per cadence). |
| Items per batch | **Lab + doctor share batches** | All HMO-billable rows live in `test_requests` (services have `kind` for lab/consult/procedure); no polymorphism needed. |
| Lifecycle | **Both batch-level and per-item** resolution | HMO can pay/reject whole batches OR mixed per-item. |
| Allocation | **Hybrid, default = per-item full** | Admin picks batch, sees per-item rows defaulted to billed amount, edits down for partials/rejections. No pro-rata fabrication of partial states. |
| Resolution destination | **Admin-picked per rejection: patient_bill or write_off** | Real-world fault attribution varies; admin has the context. |
| JE posting on resolution | **Auto-post both branches** | "One operational event → one JE" invariant. Audit trail lives on the resolution row, not the posting step. |
| Unbilled threshold | **14 days default, configurable per provider** | One submission cycle's grace; configurable since some providers run weekly cadence. |
| Stuck threshold | **`provider.due_days_for_invoice`** (no second knob) | Single config, escalation handled by aging buckets. |
| Aging buckets | **5-tier calendar from `released_at`: 0-30 / 31-60 / 61-90 / 91-180 / 180+** | Cross-provider comparison stays simple; splitting 91-180 vs 180+ matters once 3 years of history is loaded. |
| Bulk HMO response | **In scope** for 12.3 (operational, no JE impact) | Batches of 50 items shouldn't require 50 clicks to mark HMO's response. |

### 4D. `hmo_claim_batches`

```sql
CREATE TABLE hmo_claim_batches (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_id uuid NOT NULL REFERENCES hmo_providers(id),
  status text NOT NULL DEFAULT 'draft'
    CHECK (status IN ('draft', 'submitted', 'acknowledged', 'partial_paid', 'paid', 'rejected', 'voided')),
  reference_no text,  -- clinic-issued batch ref (e.g., "MAXI-2026-05-A")
  submitted_at date,
  submitted_by uuid REFERENCES staff_profiles(id),
  medium text CHECK (medium IS NULL OR medium IN ('mail', 'courier', 'email', 'portal', 'fax', 'in_person')),
  hmo_ack_ref text,
  notes text,
  voided_at timestamptz,
  voided_by uuid REFERENCES staff_profiles(id),
  void_reason text,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_hmo_claim_batches_provider ON hmo_claim_batches (provider_id);
CREATE INDEX idx_hmo_claim_batches_status ON hmo_claim_batches (status)
  WHERE voided_at IS NULL;
```

`status` is updated by trigger as items resolve (see §6).

### 4E. `hmo_claim_items`

Amounts model, not status enum. Resolution amounts are denormalized; source of truth is `hmo_claim_resolutions` for patient_billed/written_off and `hmo_payment_allocations` for paid.

```sql
CREATE TABLE hmo_claim_items (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  batch_id uuid NOT NULL REFERENCES hmo_claim_batches(id),
  test_request_id uuid NOT NULL REFERENCES test_requests(id),
  billed_amount_php numeric(12,2) NOT NULL CHECK (billed_amount_php > 0),
  paid_amount_php numeric(12,2) NOT NULL DEFAULT 0 CHECK (paid_amount_php >= 0),
  patient_billed_amount_php numeric(12,2) NOT NULL DEFAULT 0 CHECK (patient_billed_amount_php >= 0),
  written_off_amount_php numeric(12,2) NOT NULL DEFAULT 0 CHECK (written_off_amount_php >= 0),
  hmo_response text NOT NULL DEFAULT 'pending'
    CHECK (hmo_response IN ('pending', 'paid', 'rejected', 'no_response')),
  hmo_response_date date,
  hmo_response_notes text,
  batch_voided boolean NOT NULL DEFAULT false,  -- denormalized for partial unique index
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT amounts_within_billed
    CHECK (paid_amount_php + patient_billed_amount_php + written_off_amount_php <= billed_amount_php)
);

CREATE INDEX idx_hmo_claim_items_batch ON hmo_claim_items (batch_id);
CREATE INDEX idx_hmo_claim_items_test_request ON hmo_claim_items (test_request_id);
CREATE UNIQUE INDEX idx_hmo_claim_items_one_active_per_tr
  ON hmo_claim_items (test_request_id) WHERE batch_voided = false;
```

`batch_voided` mirrors `hmo_claim_batches.voided_at IS NOT NULL`, maintained by trigger on batch UPDATE.

### 4F. `hmo_payment_allocations`

```sql
CREATE TABLE hmo_payment_allocations (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  payment_id uuid NOT NULL REFERENCES payments(id),
  item_id uuid NOT NULL REFERENCES hmo_claim_items(id),
  amount_php numeric(12,2) NOT NULL CHECK (amount_php > 0),
  voided_at timestamptz,
  voided_by uuid REFERENCES staff_profiles(id),
  void_reason text,
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_hmo_payment_allocations_item ON hmo_payment_allocations (item_id) WHERE voided_at IS NULL;
CREATE INDEX idx_hmo_payment_allocations_payment ON hmo_payment_allocations (payment_id);
```

Allocations are soft-voided (not hard-deleted) to preserve audit history. When 12.2's `voidPaymentAction` soft-voids a payment, a new trigger `tg_payment_void_cascade_allocations` (AFTER UPDATE on `payments` when `voided_at` transitions NULL → NOT NULL) cascade-sets `voided_at` on all the payment's allocations, copying `voided_by` and a synthesized `void_reason` ("cascade from payment void: <payment void_reason>").

No JE posts on allocation insert — bridge's payment-side JE (`DR 1090 / CR 1110`) already moved the books at the gross payment level. Allocations are operational metadata that distribute that credit across the subledger.

Allocations update `hmo_claim_items.paid_amount_php` via AFTER INSERT/UPDATE trigger (sum of non-voided allocations per item).

### 4G. `hmo_claim_resolutions`

One row per resolution event. Drives the JE posting (each resolution row = one JE source). Lets a single item resolve via multiple events (e.g., paid 80% then 20% later split into patient_bill + write_off) without collision with 12.2's `journal_entries_one_posted_per_source` partial unique index.

```sql
CREATE TABLE hmo_claim_resolutions (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  item_id uuid NOT NULL REFERENCES hmo_claim_items(id),
  destination text NOT NULL CHECK (destination IN ('patient_bill', 'write_off')),
  amount_php numeric(12,2) NOT NULL CHECK (amount_php > 0),
  resolved_at timestamptz NOT NULL DEFAULT now(),
  resolved_by uuid REFERENCES staff_profiles(id),
  notes text,
  voided_at timestamptz,
  voided_by uuid REFERENCES staff_profiles(id),
  void_reason text
);

CREATE INDEX idx_hmo_claim_resolutions_item ON hmo_claim_resolutions (item_id)
  WHERE voided_at IS NULL;
```

Resolutions update `hmo_claim_items.patient_billed_amount_php` / `written_off_amount_php` via AFTER INSERT/UPDATE trigger.

## 5. State machines

### Batch status

```
draft ──submit──→ submitted ──HMO ack──→ acknowledged
  │                  │                       │
  │                  └───────────┬───────────┘
  │                              ↓
  │                       (per-item resolution)
  │                              ↓
  │                  partial_paid / paid / rejected
  ↓
voided   (only if no allocations and no resolutions on any item)
```

Status transitions:

- `draft → submitted` — admin action, requires `medium` and sets `submitted_at`, `submitted_by`
- `submitted → acknowledged` — optional, admin action, sets `hmo_ack_ref`
- `(submitted|acknowledged) → paid` — auto via trigger when all items resolved with paid_amount = billed
- `(submitted|acknowledged) → partial_paid` — auto via trigger when ≥1 item resolved, mix of paid/patient_billed/written_off, sum_paid > 0
- `(submitted|acknowledged) → rejected` — auto via trigger when all items resolved with paid_amount = 0
- `* → voided` — admin action, guarded by P0010

### Item amount machine

Items have no status enum; state is derived from amounts:

- `open` ⟺ `paid + patient_billed + written_off < billed`
- `resolved` ⟺ `paid + patient_billed + written_off = billed`
- `voided` (if batch voided)

`hmo_response` is independent operational metadata (`pending` / `paid` / `rejected` / `no_response`), not part of accounting state.

## 6. Bridge integration

### 6A. New JE source kind

```
source_kind = 'hmo_claim_resolution'
source_id   = hmo_claim_resolutions.id
```

### 6B. New triggers

**`hmo_claim_resolution_to_je`** — AFTER INSERT on `hmo_claim_resolutions`:

```
IF NEW.destination = 'patient_bill' THEN
  Post JE:
    DR 1100 AR Patients      NEW.amount_php
    CR 1110 AR HMO           NEW.amount_php
    description = 'HMO claim resolved → patient bill — batch <ref> item <id>'
    source_kind = 'hmo_claim_resolution'
    source_id = NEW.id
ELSIF NEW.destination = 'write_off' THEN
  Post JE:
    DR 6920 Bad Debt         NEW.amount_php
    CR 1110 AR HMO           NEW.amount_php
    description = 'HMO claim resolved → write-off — batch <ref> item <id>'
    source_kind = 'hmo_claim_resolution'
    source_id = NEW.id
END IF
```

**`hmo_claim_resolution_reversal`** — AFTER UPDATE on `hmo_claim_resolutions` when `voided_at` transitions from NULL → NOT NULL:

```
Post reversal JE via existing 12.2 reversal mechanism:
  reverses the JE for source_kind='hmo_claim_resolution', source_id=NEW.id
  reversal posting_date = today (subject to 12.1's closed-period rules)
```

### 6C. Denormalization triggers

**`tg_hmo_item_paid_amount_recompute`** — AFTER INSERT on `hmo_payment_allocations` OR AFTER UPDATE on `hmo_payment_allocations` when `voided_at` changes:

```
UPDATE hmo_claim_items
   SET paid_amount_php = (
     SELECT COALESCE(SUM(amount_php), 0)
     FROM hmo_payment_allocations
     WHERE item_id = affected_item_id
       AND voided_at IS NULL
   ),
   updated_at = now()
 WHERE id = affected_item_id;
```

**`tg_payment_void_cascade_allocations`** — AFTER UPDATE on `payments` when `voided_at` transitions NULL → NOT NULL:

```
UPDATE hmo_payment_allocations
   SET voided_at = NEW.voided_at,
       voided_by = NEW.voided_by,
       void_reason = 'cascade from payment void: ' || COALESCE(NEW.void_reason, '(no reason)')
 WHERE payment_id = NEW.id
   AND voided_at IS NULL;
-- tg_hmo_item_paid_amount_recompute fires on each cascade-voided row to refresh item amounts.
```

**`tg_hmo_item_resolution_amounts_recompute`** — AFTER INSERT or UPDATE on `hmo_claim_resolutions`:

```
UPDATE hmo_claim_items
   SET patient_billed_amount_php = (sum of non-voided patient_bill resolutions),
       written_off_amount_php = (sum of non-voided write_off resolutions),
       updated_at = now()
 WHERE id = NEW.item_id;
```

**`tg_hmo_batch_voided_propagate`** — AFTER UPDATE on `hmo_claim_batches` when `voided_at` changes:

```
UPDATE hmo_claim_items
   SET batch_voided = (NEW.voided_at IS NOT NULL),
       updated_at = now()
 WHERE batch_id = NEW.id;
```

**`tg_hmo_batch_status_rollup`** — AFTER UPDATE on `hmo_claim_items` (any amount column) or AFTER UPDATE on `hmo_claim_resolutions`:

```
Recompute parent batch.status from items:
  all items resolved AND all paid_amount = billed_amount       → 'paid'
  all items resolved AND all paid_amount = 0                   → 'rejected'
  any item resolved with paid > 0 OR mixed dispositions        → 'partial_paid'
  else (some items still open) keep current status (submitted/acknowledged)
```

### 6D. Guard triggers

| Code | Trigger | Rule |
|---|---|---|
| **P0008** | BEFORE UPDATE on `hmo_claim_items` | Reject change to `billed_amount_php` if any non-voided allocation or non-voided resolution exists for the item. |
| **P0009** | BEFORE DELETE on `hmo_claim_resolutions` | Always reject. Resolutions must be soft-voided (sets `voided_at`) so the reversal JE can be posted. |
| **P0010** | BEFORE UPDATE on `hmo_claim_batches` when `voided_at` IS being SET | Reject if any item in the batch has a non-voided allocation or non-voided resolution. |
| **P0011** | BEFORE INSERT on `hmo_claim_resolutions` | Reject if `NEW.amount_php > (item.billed - item.paid - item.patient_billed - item.written_off)`. (Overshoot beyond unresolved balance.) |
| **P0012** | BEFORE INSERT on `hmo_payment_allocations` | Reject if `(item.paid_amount_php + NEW.amount_php) > item.billed_amount_php`. (Allocation overshoots billed.) |

### 6E. Replay harness

12.2 added `bridge_replay_summary(start, end) returns jsonb`. Extend the helper's source-kind list to include `'hmo_claim_resolution'` so a single call returns counts/sums across all bridge sources.

## 7. Detection views

All computed at query time; thresholds read fresh on each query. Views are RLS-bypassing (admin-only context).

### 7A. `v_hmo_unbilled`

```sql
CREATE OR REPLACE VIEW v_hmo_unbilled AS
SELECT
  tr.id                            AS test_request_id,
  tr.visit_id,
  v.hmo_provider_id                AS provider_id,
  hp.name                          AS provider_name,
  tr.released_at,
  tr.hmo_approved_amount_php       AS billed_amount_php,
  (CURRENT_DATE - tr.released_at::date) AS days_since_release,
  (CURRENT_DATE - tr.released_at::date) > hp.unbilled_threshold_days
                                   AS past_threshold
FROM test_requests tr
JOIN visits v ON v.id = tr.visit_id
JOIN hmo_providers hp ON hp.id = v.hmo_provider_id
WHERE tr.status = 'released'
  AND v.hmo_provider_id IS NOT NULL
  AND COALESCE(tr.hmo_approved_amount_php, 0) > 0
  AND NOT EXISTS (
    SELECT 1 FROM hmo_claim_items i
    WHERE i.test_request_id = tr.id
      AND i.batch_voided = false
  );
```

### 7B. `v_hmo_stuck`

```sql
CREATE OR REPLACE VIEW v_hmo_stuck AS
SELECT
  i.id                             AS item_id,
  i.batch_id,
  b.provider_id,
  hp.name                          AS provider_name,
  b.submitted_at,
  (CURRENT_DATE - b.submitted_at)  AS days_since_submission,
  (i.billed_amount_php - i.paid_amount_php
     - i.patient_billed_amount_php - i.written_off_amount_php)
                                   AS unresolved_balance_php
FROM hmo_claim_items i
JOIN hmo_claim_batches b ON b.id = i.batch_id
JOIN hmo_providers hp ON hp.id = b.provider_id
WHERE b.status IN ('submitted', 'acknowledged', 'partial_paid')
  AND b.voided_at IS NULL
  AND (i.billed_amount_php - i.paid_amount_php
       - i.patient_billed_amount_php - i.written_off_amount_php) > 0
  AND b.submitted_at IS NOT NULL
  AND (CURRENT_DATE - b.submitted_at) > hp.due_days_for_invoice;
```

### 7C. `v_hmo_ar_aging`

```sql
CREATE OR REPLACE VIEW v_hmo_ar_aging AS
WITH items AS (
  SELECT
    i.id, b.provider_id, hp.name AS provider_name,
    tr.released_at,
    (i.billed_amount_php - i.paid_amount_php
       - i.patient_billed_amount_php - i.written_off_amount_php)
                                 AS unresolved_balance_php,
    (CURRENT_DATE - tr.released_at::date) AS age_days
  FROM hmo_claim_items i
  JOIN hmo_claim_batches b ON b.id = i.batch_id
  JOIN test_requests tr   ON tr.id = i.test_request_id
  JOIN hmo_providers hp   ON hp.id = b.provider_id
  WHERE b.voided_at IS NULL
    AND (i.billed_amount_php - i.paid_amount_php
         - i.patient_billed_amount_php - i.written_off_amount_php) > 0
  UNION ALL
  SELECT
    tr.id, v.hmo_provider_id, hp.name,
    tr.released_at,
    tr.hmo_approved_amount_php,
    (CURRENT_DATE - tr.released_at::date)
  FROM test_requests tr
  JOIN visits v ON v.id = tr.visit_id
  JOIN hmo_providers hp ON hp.id = v.hmo_provider_id
  WHERE tr.status = 'released'
    AND v.hmo_provider_id IS NOT NULL
    AND COALESCE(tr.hmo_approved_amount_php, 0) > 0
    AND NOT EXISTS (
      SELECT 1 FROM hmo_claim_items i2
      WHERE i2.test_request_id = tr.id AND i2.batch_voided = false
    )
)
SELECT
  provider_id, provider_name,
  CASE
    WHEN age_days <= 30 THEN '0-30'
    WHEN age_days <= 60 THEN '31-60'
    WHEN age_days <= 90 THEN '61-90'
    WHEN age_days <= 180 THEN '91-180'
    ELSE '180+'
  END                              AS bucket,
  SUM(unresolved_balance_php)      AS total_php,
  COUNT(*)                         AS item_count
FROM items
GROUP BY provider_id, provider_name, bucket;
```

### 7D. `v_hmo_provider_summary`

```sql
CREATE OR REPLACE VIEW v_hmo_provider_summary AS
SELECT
  hp.id                            AS provider_id,
  hp.name                          AS provider_name,
  hp.due_days_for_invoice,
  hp.unbilled_threshold_days,
  COALESCE(unresolved.total, 0)    AS total_unresolved_ar_php,
  COALESCE(unbilled.total, 0)      AS total_unbilled_php,
  COALESCE(stuck.total, 0)         AS total_stuck_php,
  COALESCE(unresolved.oldest, NULL) AS oldest_open_released_at,
  COALESCE(paid_ytd.total, 0)      AS paid_ytd_php,
  COALESCE(reso_ytd.patient_billed, 0) AS patient_billed_ytd_php,
  COALESCE(reso_ytd.written_off, 0)    AS written_off_ytd_php
FROM hmo_providers hp
LEFT JOIN ( /* per-provider sums from items joined to batches */ ) unresolved ...
LEFT JOIN ( /* unbilled totals from v_hmo_unbilled */ )           unbilled ...
LEFT JOIN ( /* stuck totals from v_hmo_stuck */ )                 stuck ...
LEFT JOIN ( /* YTD paid from allocations */ )                     paid_ytd ...
LEFT JOIN ( /* YTD resolution sums */ )                           reso_ytd ...
WHERE hp.is_active = true;
```

## 8. Server Actions

All under `src/app/(staff)/staff/(dashboard)/admin/accounting/hmo-claims/actions.ts`. Every action: `requireAdminStaff()` → Zod validation → `createAdminClient()` → DB ops → audit → revalidate → typed return.

### Action surface

| Action | Input shape | Effect |
|---|---|---|
| `createClaimBatchAction` | `{ provider_id, test_request_ids[] }` | Inserts batch (draft) + items. Audit. |
| `addItemsToBatchAction` | `{ batch_id, test_request_ids[] }` | Adds items to draft batch. Audit per item. |
| `removeItemFromBatchAction` | `{ item_id }` | Removes item from draft batch (P0008 guard if any allocation/resolution). Audit. |
| `submitBatchAction` | `{ batch_id, submitted_at, medium, reference_no? }` | Flips draft → submitted. Audit. |
| `acknowledgeBatchAction` | `{ batch_id, hmo_ack_ref? }` | Flips submitted → acknowledged. Audit. |
| `voidBatchAction` | `{ batch_id, void_reason }` | Sets `voided_at`; trigger propagates `batch_voided` to items. P0010 guard. Audit. |
| `updateItemHmoResponseAction` | `{ item_id, hmo_response, hmo_response_date, hmo_response_notes? }` | Updates response fields. Audit. |
| `bulkSetHmoResponseAction` | `{ batch_id, response, response_date, scope: 'pending_only'\|'all', notes? }` | UPDATEs items in one statement. `scope='pending_only'` filters `WHERE hmo_response = 'pending'`. `scope='all'` updates all items in the batch. Audit with `items_updated` / `items_skipped` counts. |
| `createResolutionAction` | `{ item_id, destination, amount_php, notes? }` | Inserts resolution row → trigger posts JE. P0011 guard. Audit. |
| `voidResolutionAction` | `{ resolution_id, void_reason }` | Sets `voided_at` → trigger posts reversal JE. Audit. |
| `recordHmoSettlementAction` | `{ batch_id, total_amount_php, payment_date, bank_reference?, items: [{item_id, amount_php}] }` | Atomic. Algorithm: (1) look up each `item_id` → `test_request_id` → `visit_id`. (2) group items by `visit_id`; for each group, sum `amount_php` → that's the per-visit payment amount. (3) INSERT one `payments` row per `visit_id` with `method='hmo'`, `amount_php = group_sum`, `reference = bank_reference`, `received_at = payment_date`. (4) INSERT one `hmo_payment_allocations` row per item with the row's specified amount and the `payment_id` from its visit's payment. Bridge auto-posts 1090/1110 per payment. Audit. One settlement = one batch; if a single wire covers multiple batches, admin records N settlements (each tagged with the same `bank_reference`). |
| `allocateExistingPaymentAction` | `{ payment_id, allocations: [{item_id, amount_php}] }` | For payments already created (e.g., reception logged it on a visit). Sum must equal `payment.amount_php`. P0012 guard. Audit. |

Return shape: `{ ok: true, data } | { ok: false, error }` (project convention).

## 15. Out of scope (deferred, captured here so they don't sneak in)

- **HMO claim retry / versioning workflow.** If a batch is rejected and admin resubmits, today's path is: void → new batch with same items. No explicit "v2" linkage; the new batch is just a new batch.
- **HMO provider portal integration.** All submissions remain manual (mail/courier/email/in-person).
- **Bulk HMO history import.** That's 12.A — ingests into 12.3's schema.
- **Per-item rejection-code taxonomy.** Free-text `notes` on resolution rows for now; could evolve into a `rejection_reason_code` enum if rejection-analytics needs emerge.
- **SOA / billing document generation** (the PDF or email mailed to the HMO). Admin produces this in their existing workflow.
- **Patient-side notification on rejection-billed items.** JE moves AR to 1100; reception sees patient owes on next visit. No automated SMS/email.
- **Clawback workflow.** Admin can void allocations (schema supports it; UI doesn't surface a dedicated clawback flow). Rare; manual JE in 12.7 if it happens.
- **HMO contract enforcement** (e.g., reject claim submissions to providers past `contract_end_date`). Out of scope for 12.3.
- **HMO advance / deposit handling.** Some HMOs send advances before billing. Out of scope (would require a new CoA account and workflow).
- **Multi-currency** (foreign HMO claims). Future Phase 6.
- **Real-time updates** on subledger tables. Admin-only surfaces, low concurrency, not worth wiring.
