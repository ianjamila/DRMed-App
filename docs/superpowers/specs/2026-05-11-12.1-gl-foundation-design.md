# 12.1 — GL Foundation: Chart of Accounts, Periods, Journal Entries

**Status:** Approved 2026-05-11. Implementation plan pending.
**Parent project:** [`2026-05-11-accounting-project-overview.md`](./2026-05-11-accounting-project-overview.md)
**Migration ID:** `0028_gl_foundation.sql`

## Goal

Ship the database spine of the GL — four tables, two enforcement triggers, helper functions, RLS policies, seed, and the admin-only management UI for CoA and Periods. **Nothing posts to the GL yet** — 12.2 wires operational events to JEs. This sub-project alone is a non-functional scaffold; its value is unblocking everything downstream.

## Scope

In:
- Schema: `chart_of_accounts`, `accounting_periods`, `journal_entries`, `journal_lines`, `je_year_counters`
- Enforcement triggers: JE balance check, period lock check, auto entry-number
- DB helper functions: `je_next_number()`, `period_status_for()`
- Admin UI: CoA CRUD pages + Periods grid with quarterly close/reopen
- RLS policies (admin-only) + audit logging
- Seed: 108 monthly periods (Jan 2020 → Dec 2028), year counters, `9999 - Suspense` system account

Out (handled by other sub-projects):
- JE entry UI (12.6 — manual JEs land with period-close workflow)
- Automatic JE posting from operational events (12.2)
- Subledger schemas — HMO AR (12.3), AP (12.4), COGS + Doctor PF (12.5), Payroll (12.6)
- Financial statement rendering (12.8)
- Real CoA seed beyond the safety-net Suspense account — user populates via the admin UI

## Data model

### `chart_of_accounts`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | `gen_random_uuid()` default |
| `code` | text unique not null | Stable human-readable identifier (e.g. `4100`). Renames change `name`, never `code`. Immutable after creation. |
| `name` | text not null | e.g. `Lab Tests Sales Revenue` |
| `type` | enum `account_type` not null | `asset` / `liability` / `equity` / `revenue` / `expense` / `contra_revenue` / `contra_expense` / `memo` |

| `normal_balance` | enum `account_normal_balance` not null | `debit` / `credit`. Derived-but-stored. CHECK enforces consistency with `type` (assets/expenses/contra-revenue = debit; the rest = credit; memo accounts may be either). |
| `is_active` | boolean not null default true | Soft-disable. |
| `description` | text | |
| `created_at`, `updated_at` | timestamptz default now() | + `touch_updated_at` trigger |

Proposed code-range convention (user-adoptable, not enforced):
- `1xxx` Assets · `2xxx` Liabilities · `3xxx` Equity · `4xxx` Revenue · `5xxx` COGS · `6xxx` Operating Expenses · `7xxx` Other Expenses (interest, tax penalties) · `8xxx` Other Income · `9xxx` Memo / Suspense

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `entry_id` | uuid FK `journal_entries(id)` not null, ON DELETE RESTRICT | Lines die with parent — but we never delete parents (we reverse them). |
| `account_id` | uuid FK `chart_of_accounts(id)` not null | |

## Triggers and helper functions

### `je_next_number(p_fiscal_year int) returns text`
Upserts `je_year_counters` for the year, increments `next_n`, returns `'JE-' || p_fiscal_year || '-' || lpad(n::text, 4, '0')`. Atomic via row lock.

### `period_status_for(p_date date) returns text`
Returns `'open'` / `'closed'` / `'unknown'`. Used by the period-lock check and by application code that wants to short-circuit before attempting an insert.

### `auto_assign_entry_number()` trigger
BEFORE INSERT on `journal_entries`. If `entry_number IS NULL`, sets it to `je_next_number(extract(year from posting_date))`.

### `je_lines_balance_check()` trigger
BEFORE INSERT/UPDATE/DELETE on `journal_lines`. Defers validation to STATEMENT-level so all lines for a single transaction are visible. On commit-of-statement, recomputes the parent JE's `sum(debit_php)` vs `sum(credit_php)`. If JE is `posted` and the sums differ (by any amount, even ₱0.01), raises:

```
RAISE EXCEPTION USING
  ERRCODE = 'P0001',
  MESSAGE = 'Journal entry % is unbalanced: debits ₱% vs credits ₱% (off by ₱%)',
              entry_number, total_debits, total_credits, diff;
```

Draft entries are exempt — you can save in-progress drafts that don't balance.

### `je_period_lock_check()` trigger
BEFORE INSERT/UPDATE on `journal_entries` when `status='posted'`. Calls `period_status_for(posting_date)`; if `closed`, raises:

```
RAISE EXCEPTION USING
  ERRCODE = 'P0002',
  MESSAGE = 'Cannot post journal entry dated % — that period is closed.', posting_date;
```

Reversals (`source_kind = 'reversal'`) are exempt **only if** their own `posting_date` is in an open period. The reversed JE itself can be in a closed period — that's the whole point of a reversal.

## Admin UI

Pages live under `src/app/(staff)/staff/(dashboard)/admin/accounting/`:

```
/staff/admin/accounting                            (existing — Sheets sync — untouched)
/staff/admin/accounting/chart-of-accounts          (new)
/staff/admin/accounting/chart-of-accounts/new      (new)
/staff/admin/accounting/chart-of-accounts/[id]/edit (new)
/staff/admin/accounting/periods                    (new)
```

### `chart-of-accounts` list

- Grouped by `type` in order: Assets · Liabilities · Equity · Revenue · COGS · Operating Expenses · Other Expenses · Other Income · Memo.
- Within each group: flat code-sorted table; children indented under parent.
- Columns: `code` · `name` · `normal_balance` pill · `is_active` · `posted JE count` (future link) · row-action menu.
- Filter chips: All / Active / Inactive.
- Search by code or name (client-side filter).
- `+ New account` button → `/new` page.

### `chart-of-accounts/new` form

- `code` (required, unique, immutable after submit)
- `name` (required)
- `type` (required select)
- `parent_id` (optional select, filtered to same-type accounts)
- `description` (optional textarea)
- `normal_balance` display-only, derived from `type`

### `chart-of-accounts/[id]/edit`

Same form. `code` read-only with tooltip. Adds `is_active` toggle. Deactivation blocked when there are posted JE lines in open periods referencing the account — error message names the count.

### `periods` grid

- Year picker at top (defaults to current FY).
- 4 quarter cards × 3 months each.
- Each card header: `Q1 2026 · Jan–Mar` · status badge · action button.
- Each month inside: month name + posted JE count + status badge.
- Quarter actions:
  - **Close quarter** — only shown when `today >= period_end of last month in quarter` and all three months are `open`. Confirm dialog warns about future-posting rejection.
  - **Reopen quarter** — only shown on closed quarters. Confirm dialog requires a reason note (free text, audit-logged).

Both pages are mobile-responsive (390×844 minimum), follow project conventions (44×44 tap targets, `overflow-x-auto` on tables, `MobileDrawer` for nav).

## Server Actions

All return `{ok: true, data} | {ok: false, error}`.

In `src/app/(staff)/staff/(dashboard)/admin/accounting/coa/actions.ts`:

```ts
createAccount(formData: FormData)
updateAccount(id: string, formData: FormData)
toggleAccountActive(id: string)
```

In `src/app/(staff)/staff/(dashboard)/admin/accounting/periods/actions.ts`:

```ts
closeQuarter(fiscalYear: number, fiscalQuarter: 1|2|3|4, notes?: string)
reopenQuarter(fiscalYear: number, fiscalQuarter: 1|2|3|4, reason: string)
```

Period actions are transactional — closing Q1 atomically locks all three monthly rows or none. Audit log fires inside the same transaction.

Each action:
1. Calls `requireAdminStaff()` first — throws 403 if not admin.
2. Zod-validates input.
3. Uses `createAdminClient()` for the DB write.
4. Translates Postgres errors via `src/lib/accounting/pg-errors.ts` helper (unique-violation 23505, check-violation 23514, RAISE EXCEPTION P0001/P0002).
5. Writes `audit_log` row in the same transaction.

## User-facing error messages

| Trigger | Message |
|---|---|
| CoA create — duplicate code | `Account code '{code}' already exists. Codes must be unique.` |
| CoA create — parent type mismatch | `Parent account must be the same type (asset / liability / etc.).` |
| CoA toggle — has open-period postings | `Cannot deactivate: {n} posted journal lines reference this account in open periods. Move those to another account first, or close the period.` |
| Period close — already closed | `Q{q} {year} is already closed.` |
| Period close — future quarter | `Cannot close Q{q} {year} — the quarter hasn't ended yet.` |
| Period reopen — quarter is open | `Q{q} {year} is already open. No action needed.` |
| Period reopen — missing reason | `A reason note is required to reopen a closed quarter.` |
| JE trigger P0001 (balance) | `Journal entry {entry_number} is unbalanced: debits ₱{d} vs credits ₱{c} (off by ₱{diff}).` |
| JE trigger P0002 (period lock) | `Cannot post journal entry dated {date} — Q{q} {year} is closed. Reopen the quarter or post to a later open period.` |

Sentry receives the full PG context on each error path.

## Seed (inside `0028_gl_foundation.sql`)

| Seed | Detail |
|---|---|
| 108 monthly periods | Jan 2020 → Dec 2028, status `open`. Generated with a `generate_series` loop. Wider-than-needed range gives margin for opening-balance JEs dated pre-2023 (the clinic existed before the Sheet started) and avoids "unknown period" gaps for any plausible posting date. |
| Year counters | Rows for 2020–2028, `next_n=1`. |
| CoA `9999 - Suspense` | `type='memo'`, `is_active=true`. Safety-net account for 12.2 when an operational event has no obvious CoA mapping. |

No other CoA rows are seeded — user populates the real CoA through the admin UI.

## Testing

The project has no test runner. Verification is three-prong manual:

### 1. SQL integrity test — `supabase/tests/0028_gl_foundation_smoke.sql`

Inside `BEGIN; ... ROLLBACK;` envelope. Asserts:

- Inserting a balanced JE (DR 1000, CR 1000) → succeeds.
- Inserting an unbalanced JE (DR 1000, CR 999) → raises P0001.
- Closing Q1 2026 → all three months flip atomically to `closed`.
- Inserting a JE dated `2026-02-15` after Q1 close → raises P0002.
- Inserting a reversal JE dated `2026-04-05` that references the Feb-5 entry → succeeds.
- Reopening Q1 2026 → flips back atomically.
- Trying to deactivate a CoA account with open-period postings → fails (Server-Action-level check, also tested as a query).
- Trying to deactivate a CoA account whose only postings are in closed periods → succeeds.

Run via `supabase db execute --file supabase/tests/0028_gl_foundation_smoke.sql` (or `psql -f`) against local Supabase.
