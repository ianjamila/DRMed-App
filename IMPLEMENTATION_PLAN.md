# drmed.ph ŌĆö Implementation Plan

A unified Next.js + Supabase application for a Philippine medical laboratory: marketing site, patient portal, and staff portal. This plan is structured as sequential phases. Complete each phase fully before moving to the next. After every phase, run the verification checklist at the bottom of that phase before proceeding.

---

## Project Context

**Domain:** drmed.ph
**Compliance:** Philippine Data Privacy Act (RA 10173)
**Locale:** Philippines (en-PH, Asia/Manila timezone, PHP currency)
**Deployment:** One Vercel project, one Supabase project

### Three surfaces, one app

1. **Marketing site** (`/`) ŌĆö public, SEO-optimized, services and schedule info
2. **Patient portal** (`/portal/*`) ŌĆö patients view and download released results
3. **Staff portal** (`/staff/*`) ŌĆö reception, medtechs, and admins manage operations

### Operational flow (the actual lab workflow)

```
Patient walks in
  ŌåÆ Reception creates patient + visit (issues DRM-ID + receipt PIN)
  ŌåÆ Reception requests test(s) [test_request.status = requested]
  ŌåÆ Medtech sees queue, claims test [in_progress]
  ŌåÆ Medtech uploads PDF result [result_uploaded]
  ŌåÆ (Optional) Pathologist signs off [ready_for_release]
  ŌåÆ Patient pays at reception [visit.payment_status = paid]
  ŌåÆ Reception releases result [released] ŌĆö gated by payment
  ŌåÆ Patient receives SMS/email notification
  ŌåÆ Patient logs into portal with DRM-ID + PIN, downloads PDF
```

## Phase 1 ŌĆö Database Schema & RLS

**Goal:** Complete database schema with row-level security policies. No app code yet.

Create as a Supabase migration: `supabase/migrations/0001_init.sql`.

### Tables

**`patients`**
- `id` uuid PK default `gen_random_uuid()`
- `drm_id` text unique not null ŌĆö public ID, e.g. `DRM-0001` (auto-generated by sequence)
- `first_name`, `last_name`, `middle_name` text
- `birthdate` date not null
- `sex` text check in (`male`, `female`)
- `phone` text ŌĆö PH format, validated app-side (`+639XXXXXXXXX`)
- `email` text
- `address` text
- `created_at`, `updated_at` timestamptz default `now()`
- `created_by` uuid references `auth.users(id)`

**`staff_profiles`**
- `id` uuid PK references `auth.users(id)` on delete cascade
- `full_name` text not null
- `role` text not null check in (`reception`, `medtech`, `pathologist`, `admin`)
- `is_active` boolean default true
- `created_at`, `updated_at` timestamptz

**`services`** ŌĆö catalog of tests/services offered
- `id` uuid PK
- `code` text unique ŌĆö e.g. `CHEM`, `CBC`, `URINALYSIS`
- `name` text ŌĆö display name
- `description` text
- `price_php` numeric(10,2) not null
- `is_active` boolean default true
- `turnaround_hours` int ŌĆö typical hours to result

**`visits`**
- `id` uuid PK
- `patient_id` uuid references `patients(id)` not null
- `visit_number` text unique not null ŌĆö auto-generated, e.g. `1234`
- `visit_date` date default `current_date`
- `payment_status` text default `unpaid` check in (`unpaid`, `partial`, `paid`, `waived`)
- `total_php` numeric(10,2) default 0
- `paid_php` numeric(10,2) default 0
- `notes` text
- `created_at`, `updated_at` timestamptz
- `created_by` uuid references `auth.users(id)`

**`visit_pins`** ŌĆö receipt PIN for patient portal access
- `id` uuid PK
- `visit_id` uuid references `visits(id)` unique not null
- `pin_hash` text not null ŌĆö bcrypt hash of the PIN
- `expires_at` timestamptz not null ŌĆö default `now() + interval '60 days'`
- `last_used_at` timestamptz
- `failed_attempts` int default 0
- `locked_until` timestamptz ŌĆö populated on too many failed attempts

**`test_requests`**
- `id` uuid PK
- `visit_id` uuid references `visits(id)` not null
- `service_id` uuid references `services(id)` not null
- `status` text default `requested` check in (`requested`, `in_progress`, `result_uploaded`, `ready_for_release`, `released`, `cancelled`)
- `requested_by` uuid references `auth.users(id)` not null
- `requested_at` timestamptz default `now()`
- `assigned_to` uuid references `auth.users(id)`
- `started_at`, `completed_at`, `signed_off_at`, `released_at` timestamptz
- `signed_off_by`, `released_by` uuid references `auth.users(id)`
- `cancelled_reason` text

**`results`**
- `id` uuid PK
- `test_request_id` uuid references `test_requests(id)` unique not null
- `storage_path` text not null ŌĆö path in `results` storage bucket
- `file_size_bytes` int
- `uploaded_by` uuid references `auth.users(id)` not null
- `uploaded_at` timestamptz default `now()`
- `notes` text ŌĆö internal medtech notes, NOT visible to patient

**`payments`**
- `id` uuid PK
- `visit_id` uuid references `visits(id)` not null
- `amount_php` numeric(10,2) not null
- `method` text check in (`cash`, `gcash`, `maya`, `card`, `bank_transfer`)
- `reference_number` text ŌĆö for non-cash methods
- `received_by` uuid references `auth.users(id)` not null
- `received_at` timestamptz default `now()`
- `notes` text

**`appointments`** ŌĆö for scheduling
- `id` uuid PK
- `patient_id` uuid references `patients(id)` ŌĆö nullable for new patients
- `walk_in_name` text ŌĆö for non-registered bookings
- `walk_in_phone` text
- `service_id` uuid references `services(id)`
- `scheduled_at` timestamptz not null
- `status` text default `confirmed` check in (`confirmed`, `arrived`, `cancelled`, `no_show`, `completed`)
- `notes` text
- `created_at` timestamptz, `created_by` uuid references `auth.users(id)`

**`audit_log`** ŌĆö RA 10173 access log
- `id` bigserial PK
- `actor_id` uuid ŌĆö staff user id, or NULL if patient/anonymous
- `actor_type` text check in (`staff`, `patient`, `system`, `anonymous`)
- `patient_id` uuid ŌĆö subject of the access, when applicable
- `action` text not null ŌĆö e.g. `result.viewed`, `result.downloaded`, `pin.failed`, `payment.recorded`
- `resource_type` text, `resource_id` uuid
- `metadata` jsonb
- `ip_address` inet
- `user_agent` text
- `created_at` timestamptz default `now()`

### Sequences and helpers

- Sequence `drm_id_seq` and a function `generate_drm_id()` returning `'DRM-' || lpad(nextval('drm_id_seq')::text, 4, '0')`.
- Sequence `visit_number_seq` similarly for visit numbers.
- Function `is_staff()` returning bool based on `staff_profiles` row + `is_active`.
- Function `staff_role()` returning the current user's role.
- Function `has_role(text[])` returning bool if current user's role is in list.

### Triggers

- `updated_at` auto-update trigger on every table that has it.
- **Payment-gating trigger** on `test_requests`: when transitioning `status` to `released`, check the parent visit's `payment_status = 'paid'`. If not, raise an exception.
- Trigger on `payments` insert: recalculate `visits.paid_php` and `visits.payment_status`.
- Trigger on `results` insert: auto-flip parent `test_requests.status` from `in_progress` to `result_uploaded` (or to `ready_for_release` if no pathologist sign-off is configured for this service ŌĆö default to `ready_for_release` initially).

### RLS policies

Enable RLS on every table. Then add:

**`patients`**
- Staff with role in (`reception`, `medtech`, `pathologist`, `admin`) ŌåÆ `SELECT`, `INSERT`, `UPDATE`
- Patients (via the patient-PIN session ŌĆö see Phase 2) can `SELECT` only their own row

**`visits`** ŌĆö same staff rules; patients see only their own visits

**`test_requests`**
- Reception ŌåÆ `INSERT`, and `UPDATE` only the `released_*` and status to `released` (with the payment-gating trigger enforcing the rule)
- Medtech ŌåÆ `SELECT` all, `UPDATE` to set `assigned_to`, `started_at`, transition `requested` ŌåÆ `in_progress`
- Pathologist ŌåÆ `UPDATE` to sign off
- Admin ŌåÆ all
- Patient ŌåÆ `SELECT` only their visit's test requests, only when `status = 'released'`

**`results`**
- Medtech, Pathologist, Admin ŌåÆ `INSERT`, `SELECT`
- Reception ŌåÆ `SELECT` (to confirm a result exists; cannot read storage without signed URL)
- Patient ŌåÆ `SELECT` only when their parent `test_request.status = 'released'`. Patients NEVER access storage directly ŌĆö they get signed URLs from a Server Action that audits the access.

**`payments`**
- Reception, Admin ŌåÆ `INSERT`, `SELECT`
- Medtech, Pathologist ŌåÆ no access
- Patient ŌåÆ `SELECT` only their own visit's payments (so they can see what they paid)

**`audit_log`**
- Admin ŌåÆ `SELECT`
- Everyone else ŌåÆ no `SELECT`. Inserts happen via service-role from server actions.

**`services`** ŌĆö public read; admin write

**`appointments`** ŌĆö staff read/write; public can `INSERT` for booking (rate-limited via Edge Function in Phase 6)

### Storage

Create a private bucket `results`. Paths follow `patient_id/visit_id/test_request_id.pdf`. No public access. All downloads go through signed URLs created by a Server Action that:
1. Verifies the requester is the patient who owns the visit OR a permitted staff member.
2. Inserts an `audit_log` row.
3. Returns a signed URL valid for 5 minutes.

### Verification

- [ ] Migration runs cleanly on a fresh Supabase project
- [ ] Inserting a `payment` updates the `visits.paid_php` correctly
- [ ] Attempting to set a `test_request.status` to `released` while `visit.payment_status = 'unpaid'` raises an error
- [ ] A logged-in staff user with role `medtech` cannot `SELECT` from `payments`
- [ ] Generated TypeScript types exist at `src/types/database.ts` (`supabase gen types typescript --linked > src/types/database.ts`)

## Phase 2 ŌĆö Authentication

**Goal:** Two distinct auth flows working: staff via Supabase Auth (email + password + optional MFA), patients via DRM-ID + receipt PIN.

### Staff auth

Standard Supabase Auth email/password. After a successful sign-in, middleware checks `staff_profiles.is_active = true`; if not, sign out immediately. MFA via Supabase's TOTP enrollment (deferred to Phase 5 ŌĆö for now, password-only is acceptable but mark it as a TODO with a clear note).

Pages:
- `/staff/login` ŌĆö email + password (mirrors screenshot 3 layout ŌĆö STAFF GATE LOGIN). Drop the "Backend API Base" field ŌĆö that was a placeholder for the multi-backend idea you scrapped. Keep MFA code field as optional.
- `/staff/logout` ŌĆö handled by Server Action.

### Patient auth (custom ŌĆö not Supabase Auth)

Patients do NOT have Supabase Auth accounts. Their credentials are `drm_id + pin` against the `visit_pins` table.

Implementation:

1. **Server Action `signInPatient(drm_id, pin)`**:
   - Look up the patient by `drm_id`.
   - Find the most recent `visit_pins` row for any of the patient's visits where `expires_at > now()` and `locked_until` is null or in the past.
   - `bcrypt.compare(pin, pin_hash)`. If fails, increment `failed_attempts`; if >= 5, set `locked_until = now() + interval '15 minutes'`. Audit-log the failure.
   - On success, mint a signed JWT (HS256, server secret) with payload `{ patient_id, drm_id, visit_id, exp: now + 30min }`, and set it as an HttpOnly, Secure, SameSite=Strict cookie named `drmed_patient_session`.
   - Audit-log the success.

2. **Patient session middleware** at `/portal/*`:
   - Reads the cookie, verifies signature and expiry.
   - On failure, redirects to `/portal/login`.
   - On success, makes `patient_id` available via a Server Component helper `getPatientSession()`.

3. **RLS bridge:** since patients don't authenticate to Postgres, the patient portal queries must run through Server Actions or Route Handlers using the service-role client BUT scoped manually by the `patient_id` from the verified session. Alternatively (cleaner): define a Postgres function `set_patient_context(patient_id uuid)` that the server calls before queries, and write RLS policies against `current_setting('app.current_patient_id')`. Use the function approach ŌĆö it keeps RLS as the single source of truth.

4. **PIN generation** ŌĆö server-side, when reception creates a visit:
   - 8 characters from a reduced alphabet (no `0`, `O`, `1`, `I`, `l`) for legibility on receipts.
   - Hash with bcrypt (cost 12). Store hash, never the plain PIN.
   - Return the plain PIN ONCE in the API response so reception can print the receipt.

### Pages

- `/portal/login` (mirrors screenshot 4 ŌĆö "Your Results, Securely Accessible") ŌĆö DRM-ID and PIN inputs.
- `/portal/logout` ŌĆö clears cookie.
- `/staff/login` (mirrors screenshot 3 layout, simplified).

### Verification

- [ ] Staff can log in, hit `/staff`, get redirected to `/staff/login` after logout
- [ ] A staff user with `is_active = false` is rejected
- [ ] A patient with valid DRM-ID + PIN can reach `/portal/results`
- [ ] After 5 wrong PINs, the account is locked for 15 minutes (verify in DB)
- [ ] Patient cookie cannot be reused after expiry (manually expire the JWT)
- [ ] Patient session does NOT grant access to other patients' rows (manually try accessing another `patient_id` via the Server Action ŌĆö should fail at RLS)

## Phase 5 ŌĆö Patient Portal

**Goal:** Patients sign in and see/download their released results, matching screenshot 1.

### Pages

- `/portal/login` ŌĆö DRM-ID + PIN form (already built in Phase 2).
- `/portal` (after login) ŌĆö same UI as screenshot 1: patient name and DRM-ID header, table of visits with columns Visit #, Test, Test Date, Released, Status, Action (Download).

- `/portal/help` ŌĆö FAQ and contact info.
- `/portal/logout` ŌĆö clears the patient session cookie.

### Download flow

1. User clicks Download on a released test.
2. Hits a Server Action `getResultDownloadUrl(test_request_id)`.
3. Server verifies: patient session is valid AND owns the visit AND test status is `released`.
4. Inserts `audit_log` with `action = 'result.downloaded'`.
5. Returns a 5-minute signed URL.
6. Browser navigates to the signed URL (browser handles the actual download).

### Notifications (basic)

When `test_requests.status` flips to `released`, fire a Supabase Edge Function `notify_patient`:
1. Look up patient phone and email.
2. Send SMS via Semaphore: "Hi {first_name}, your lab results from drmed.ph are ready. Sign in at https://drmed.ph/portal with your DRM-ID and Secure PIN from your receipt. ŌĆö drmed.ph"
3. Send email via Resend with a similar message.
4. Failed sends should be logged but should NOT block the release.

Add env vars: `RESEND_API_KEY`, `SEMAPHORE_API_KEY`, `SEMAPHORE_SENDER_NAME`.

### Verification

- [ ] Patient sees only their own visits and only released tests
- [ ] Download produces a working PDF and an audit log entry
- [ ] Releasing a result triggers SMS + email (test with real numbers)
- [ ] Cookie expires after 30 minutes of inactivity (or extend on activity ŌĆö pick one and document)

### Cron job

```json
{ "crons": [{ "path": "/api/cron/sync-accounting", "schedule": "0 17 * * *" }] }
```

The route handler:
1. Verifies the Vercel cron secret header.
2. Pulls today's visits, payments, and released results.
3. Appends rows to the Sheet using the Google Sheets API.
4. Records last-synced timestamp in a `sync_state` table to make it resumable.

### Verification

## Phase 8 ŌĆö Hardening & Compliance

**Goal:** Production-ready security, RA 10173 compliance, and operational readiness.

### Security

- [ ] Enforce HTTPS everywhere (Vercel default).
- [ ] Set strict Content Security Policy headers in `next.config.ts`.
- [ ] HSTS, X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy strict-origin-when-cross-origin.
- [ ] Rate limit login attempts (staff and patient) by IP and by identifier.

## Cross-Cutting Conventions

### Code style
- TypeScript strict mode on
- No `any` without a comment justifying it
- Server Components by default; mark client components with `'use client'` only when needed
- All Supabase calls typed with the generated `Database` type

### Error handling
- Never expose stack traces to users
- All Server Actions return `{ ok: true, data } | { ok: false, error }`
- User-facing errors are short and actionable; technical details go to Sentry

### Naming
- Routes: kebab-case
- DB columns: snake_case
- TypeScript: camelCase for variables, PascalCase for types/components
- Files: kebab-case for routes, PascalCase for components

### Commits
- Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`)
- Each phase ends with a tagged release (`v0.1.0` after Phase 1, etc.)

### What NOT to do
- Do NOT use Supabase Auth for patients ŌĆö they auth via DRM-ID + PIN
- Do NOT expose the service-role key client-side
- Do NOT skip RLS ŌĆö it is the single source of truth for access
- Do NOT log plain PINs anywhere, ever
- Do NOT hardcode service prices in the frontend ŌĆö always read from `services` table
- Do NOT add a "Backend API Base" field to the staff login (legacy from a scrapped multi-backend design)
