## The one decision that shaped this design

The current system **books card receipts straight to Cash on Hand (1010)** â€”
`payment_method_account_map` says verbatim *"Card receipts treated as cash on hand;
refine when Veritas settlement is modelled."* There is **no** Veritas Pay
clearing/receivable account, **no** settlement event, **no** `card_settlements`
table. So the sheet's CREDIT CARD **OUT (settled)** and **RECEIVABLE balance**
(rows 90â€“91) and the **"Received Veritas Pay"** cash line (row 67) have **no data
source**.

**Decisions (user-approved 2026-06-07):**

1. **Credit card = read-layer only.** Show card collected (**IN**) as same-day cash
   (matching the GL), with an honest note that settlement/receivable is **not
   tracked yet**. Modelling Veritas settlement is a *future* sub-project, not B1.2.
2. **HMO-received = a single aggregate line** in the cash-collected block (the
   per-provider IN/OUT/ending subledger stays **B1.4**).
3. **Cash collected = `ÎŁ payments.amount_php`** by `received_at` Ă— method, split
   Lab/Consult by the visit's single section. Ties to `payments` (and, going
   forward, the EOD cash drawer).
4. **Reconciliation panel = forward-looking.** Build it reading `eod_close_records`;
   it shows a clean empty state until reception starts closing shifts, then
   auto-activates. (Approved over dropping it.)

## Architecture

Same posture as B1.1: **read-only, admin-only, no schema change to write paths.**
- **Data layer:** two new `security_invoker = on` views; **no** grant to
  `anon`/`authenticated` (only the service-role admin client reads them â€” these

  **no `server-only` import**, reusing B1.1's `src/lib/operations/daily-report.ts`
  helpers.
- **UI:** a new Server-Component route `admin/operations/cash/page.tsx`, added as a
  third `SectionTabs` tab. Gated by `requireAdminStaff`.

## Views â€” migration `0095_ops_cash_views.sql`

### `v_ops_daily_collections` â€” grain `(business_date, section, method)`

Cash-basis receipts. **Section is classified per-visit via `EXISTS` (one row per
visit) â€” never join `payments â†’ test_requests` directly, or each payment fans out
by test count.**

```sql
create or replace view public.v_ops_daily_collections
with (security_invoker = on) as
select
  (p.received_at at time zone 'Asia/Manila')::date as business_date,
  case
    when exists (
      select 1 from public.test_requests tr
      join public.services s on s.id = tr.service_id
      where tr.visit_id = p.visit_id and s.kind = 'doctor_consultation'
    ) then 'consult'
    when exists (select 1 from public.test_requests tr where tr.visit_id = p.visit_id)
      then 'lab'
    else 'unknown'           -- defensive: 0 such rows in prod today
  end as section,
  p.method,
  count(*)                                       as line_count,
  coalesce(sum(p.amount_php), 0)::numeric(14,2)  as amount
from public.payments p
where p.voided_at is null
  and p.method <> 'hmo'      -- 'hmo'-method (7 rows / â‚±5,947) is not a cash receipt;
                             -- HMO is surfaced via v_ops_daily_hmo_received instead.
group by 1, 2, p.method;

alter view public.v_ops_daily_collections owner to postgres;
```

> **Section rule:** "consult-wins" â€” a visit with any `doctor_consultation` test is
> `consult`, else `lab` (the 1 historical mixed visit â†’ consult; deterministic).
> A visit with no tests â†’ `unknown`, surfaced in the UI only if non-zero.
>
> **No separate totals view.** Unlike B1.1 (which needed one for non-additive
> distinct-customers/PF), collection **amounts are additive**, so per-method and
> grand totals roll up in JS from this single view.

### `v_ops_daily_hmo_received` â€” grain `(received_date, source)`

The "Received HMO Receivable" line â€” HMO settlements actually collected, by the
date paid. Union of the live subledger and the historic audit ledger.

```sql
create or replace view public.v_ops_daily_hmo_received
with (security_invoker = on) as
-- Live 12.3 subledger: paid claim items, excluding voided batches.
select
  i.hmo_response_date                              as received_date,
  'live'::text                                     as source,
  count(*)                                         as claim_count,
  coalesce(sum(i.paid_amount_php), 0)::numeric(14,2) as amount
from public.hmo_claim_items i
join public.hmo_claim_batches b on b.id = i.batch_id
where i.hmo_response = 'paid'
  and i.hmo_response_date is not null
  and b.voided_at is null
group by i.hmo_response_date
union all
-- 12.B historic audit ledger: paid claims with a settlement date.
select
  h.date_paid                                      as received_date,
  'historic'::text                                 as source,
  count(*)                                         as claim_count,
  coalesce(sum(h.final_amount_php), 0)::numeric(14,2) as amount
from public.historic_hmo_claims h
where h.status = 'paid'
  and h.date_paid is not null
group by h.date_paid;

alter view public.v_ops_daily_hmo_received owner to postgres;
```

> Paid-but-no-date rows (44 historic) are excluded â€” they can't be placed on a day;
> surfaced as a footnote count in validation, not in the matrix.

## Pure core â€” `src/lib/operations/cash-report.ts`

Imports `channelLabel`, `num`, `enumerateDays`, `groupDaysByMonth`, and the
`Section` type from `./daily-report.ts` (no duplication). Method display order:
`cash Â· gcash Â· bpi Â· bank_transfer Â· card` (â†’ Cash/GCash/BPI/BDO/Card pay via
`channelLabel`); any other method (e.g. `maya`, `gift_code`) appends as an extra

- `buildCollectionsMatrix(rows, days)` â†’ the day-matrix: section blocks
  **Lab** and **Consult**, each with one row per method + a section TOTAL; then a
  cross-section **HMO received** row (from the hmo view); then a **TOTAL CASH
  COLLECTED** grand-total row = `ÎŁ collections (lab + consult, all methods) + HMO
  received` (excludes rent/APE â€” out of scope, so it reads under the sheet's r69 on
  those days, mirroring B1.1's total caveat). Reuses `groupDaysByMonth` for
  collapsible months.
- `buildCreditCardPanel(rows, days)` â†’ card-method (IN) by day + total; carries a
  `settlementTracked: false` flag for the UI note.
- `buildCashReconRows(eodRows, days)` â†’ per-day expected/counted/variance; flags
  days with no close as `reconciled: false`; whole-range "no closes" handled by the
  page's empty state.

All `numeric` values arrive as strings from Supabase â†’ coerce with `num`.
