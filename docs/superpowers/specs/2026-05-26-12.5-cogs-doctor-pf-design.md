# 12.5 — COGS + Doctor PF Subledger — Design Spec

**Status:** Design locked 2026-05-26. Pending plan + implementation.
**Predecessors:** 12.1 (GL foundation, v1.2.0), 12.2 (op→GL bridge, v1.3.0), 12.3 (HMO AR subledger, v1.4.0), 12.4 (AP subledger, v1.9.0).
**Companion sub-project:** 12.5b — Doctor Rent AR subledger. Split out of original 12.5 scope; specced separately after 12.5 ships.
**Tag target:** v1.12.0

---

## 1. Scope

12.5 closes two open subledger gaps before the 12.B operational history import:

1. **Doctor PF subledger.** Per-line `doctor_pf_php` (already captured on `test_requests` since migration 0011) is split out of the existing `4200 Doctor Consultation Sales Revenue` aggregate into a dedicated `2110 AP — Doctors` liability, attributed to a specific physician, disbursed via end-of-day batch payouts. Recognition is hybrid: cash patients accrue at release; HMO patients defer to settlement via a holding account.

2. **Send-out COGS subledger.** Per-test cost accrual for `services.is_send_out=true` test_requests, offset to a new `2150 Accrued Send-Out` liability, trued up against Hi Precision vendor bills via the 12.4 AP system.

In-house lab supplies COGS remains periodic (month-end aggregate JE driven by 12.4 vendor bills); no subledger needed.

### Explicitly out of scope (deferred)

| Item | Disposition |
|---|---|
| Doctor rent AR (`1120` / `4300`) — rent-paying physicians' monthly invoices, aging, payments | **12.5b** — follows immediately after 12.5 |
| `patient_bill` HMO resolution PF accrual (when HMO denies and patient is re-billed; PF stays in `2160` holding until cash payment lands) | **12.5b followup** — rare path; documented as limitation |
| EWT withholding on doctor PFs (10% Expanded WT per BIR 1601-EQ) | **Future Phase 1/2** — clinic currently pays gross; doctors do their own books separately |
| Doctor-facing PF statement PDF export | Future quality-of-life |
| Visit-level physician change after release via UI (only via SQL today) | Future quality-of-life |
| Bulk send-out unit_cost editor | Future quality-of-life |
| Per-doctor PF subledger drill-in page (`/staff/admin/accounting/pf-doctors/[id]`) | Future quality-of-life |
| BIR filing surfaces (alphalist, 1601-EQ) | Future Phase 2 |

## 2. Decisions locked (12 product/policy)

| # | Decision | Rationale |
|---|---|---|
| D1 | Physician attribution at **line level** (`test_requests.attending_physician_id`) + **visit level** (`visits.attending_physician_id`); trigger reads `COALESCE(line, visit)`. Reception sets visit-level once, per-line override exists for rare multi-doctor visits. | Most visits have one ordering doctor; visit-level avoids 5 clicks per visit. Per-line override preserves flexibility. |
| D2 | Hybrid PF recognition: cash visits accrue PF **at test_request release**; HMO visits accrue PF **at `hmo_payment_allocations` insert** (proportional to settlement amount). | Matches user's actual contract — "doctor earns PF when clinic gets paid"; HMO denial risk shared with doctor. |
| D3 | COGS hybrid: in-house lab supplies = periodic month-end (no subledger, driven by 12.4 vendor bills). Send-outs = per-test accrual via this subledger. | In-house unit-cost tracking is out of scope (no perpetual inventory); send-out unit cost is knowable per service. |
| D4 | Send-out cost source: `services.send_out_unit_cost_php` (static, snapshot at release) + offset to new `2150 Accrued Send-Out` + monthly true-up against Hi Precision bills via 12.4 AP. | Real-time per-test margin + full reconciliation to actual vendor invoice within 30 days. |
| D5 | No EWT withholding. PF disbursed gross. | Doctors maintain their own books separately; clinic doesn't act as withholding agent today. Flagged as compliance gap. |
| D6 | EOD batch payout page (`/staff/admin/accounting/pf-payouts`), one disbursement per doctor per batch (1:N), method = cash / gcash / bank_transfer. | Matches user's described workflow (doctors paid same-day or next-day via cash/gcash) and 12.C cash recon UX cadence. |
| D7 | PF accrual at HMO settlement is **proportional** to `allocation.amount_php / item.billed_amount_php`. In practice always 1.0 or 0.0 per user. | Gracefully handles partials should they ever occur. |
| D8 | `physicians.compensation_arrangement` enum: `'pf_split' \| 'rent_paying' \| 'shareholder'`. Drives visit form's `clinic_fee_php` default per doctor (was always 100). | Resolves the 4 shareholder doctors + N rent-paying doctors' systematic misclassification under today's default. |
| D9 | Shareholder doctor PF tracks in `2110` like any other doctor — same EOD batch flow, no separate equity-side account in 12.5. | Operationally simplest; "shareholder vs non-shareholder" report derives from `compensation_arrangement` at query time. |
| D10 | Send-out without `unit_cost_php`: release proceeds with zero accrual + `cogs_send_out_entries` row at `unit_cost_php=0` + audit_log + admin banner. NOT a hard block. | Reception cannot fix admin-only fields; blocking strands patients at counter. True-up flow picks up the missing accrual as variance at bill time. |
| D11 | `2160 Doctor PF Pending HMO Settlement` holding account. Release for HMO consults credits `4200` for `clinic_fee_php` and `2160` for `doctor_pf_php`. Settlement moves `2160 → 2110`; writeoff moves `2160 → 6920`. | Avoids revenue overstatement between release and settlement. Auditor-defensible. |
| D12 | EOD payouts list groups by **PF-cleared date** (when accrual fires), not visit date. HMO-settled PFs appear on settlement day, not visit day. | Doctors see one consolidated payable balance regardless of cash vs HMO origin. |

### 3.5 Data scrub — admin-triggered, NOT in migration

The scrub cannot run inside `0064` because (a) `physicians.compensation_arrangement` is created with default `'pf_split'` in the same migration — no rent_paying / shareholder rows exist yet, so the scrub would be a no-op on first run; and (b) historical released tests already have posted JEs that the scrub can't update without creating data/JE divergence.

Instead, after migration:

1. Admin classifies rent_paying / shareholder physicians via `/staff/admin/physicians/[id]/edit` (column now exists with the default).
2. Admin clicks "Recompute clinic-fee on unreleased tests" button on `/staff/admin/physicians` index page. The button calls a new Server Action `recomputeClinicFeeForUnreleased`:

```sql
-- Server Action body (typed in TS, executes via supabase.rpc or direct query):
update public.test_requests tr
  set clinic_fee_php = 0,
      doctor_pf_php = tr.final_price_php
  where coalesce(tr.attending_physician_id,
                 (select v.attending_physician_id from public.visits v where v.id = tr.visit_id))
        in (select id from public.physicians where compensation_arrangement in ('rent_paying', 'shareholder'))
    and tr.clinic_fee_php > 0
    and not exists (
      select 1 from public.journal_entries je
      where je.source_kind = 'test_request'
        and je.source_id = tr.id
        and je.status = 'posted'
    );
```

The `not exists` predicate ensures we never touch a test_request whose JE has already posted — preserving GL integrity. Already-released-under-old-bridge tests stay as-is in their wrong-but-historical state; 12.B history reconciliation handles them separately.

Audit log: `pf.clinic_fee_scrub_run` with metadata `{rows_affected, physicians_classified}`.

## 4. JE bridge / trigger logic

### 4.1 Modified: `bridge_test_request_released`

The existing function (in `0030`) collapses the full consult/procedure amount into `4200` (or `4500` for procedures). 12.5 rewrites it. The new logic per kind:

**For `doctor_consultation` and `doctor_procedure`:**

```
v_physician_id := coalesce(tr.attending_physician_id, v.attending_physician_id);

if v_physician_id is null then
  raise exception 'P0034 attending_physician_id required for consult/procedure release on test_request %', tr.id
    using errcode = 'P0034';
end if;

if v.hmo_provider_id is null then
  -- Cash path
  draft_je(source_kind='test_request', source_id=tr.id, posting_date=tr.released_at::date);
    line: DR <cash_account>     tr.final_price_php
    line: CR 4200 or 4500       if tr.clinic_fee_php > 0
    line: CR 2110               if tr.doctor_pf_php > 0
    line: DR 4910 or 4920       if discount > 0  (contra-revenue, unchanged)
  insert into doctor_pf_entries (
    test_request_id, physician_id, pf_php, recognition_basis='cash_at_release',
    recognized_at=now(), journal_entry_id=je.id
  ) if tr.doctor_pf_php > 0;
else
  -- HMO path
  draft_je(source_kind='test_request', source_id=tr.id, posting_date=tr.released_at::date);
    line: DR 1110 AR HMO        tr.final_price_php
    line: CR 4200 or 4500       if tr.clinic_fee_php > 0
    line: CR 2160               if tr.doctor_pf_php > 0   -- holding account
    line: DR 4910 or 4920       if discount > 0
  insert into doctor_pf_entries (
    test_request_id, physician_id, pf_php, recognition_basis='hmo_at_settlement',
    recognized_at=null, journal_entry_id=null  -- fires later at settlement
  ) if tr.doctor_pf_php > 0;
end if;
```

**For all kinds where `s.is_send_out = true`:** ADDITIONALLY emit on the same JE (or skip if unit_cost is 0):

```
if s.send_out_unit_cost_php > 0 then
  add lines to the JE drafted above:
    line: DR 6420              s.send_out_unit_cost_php
    line: CR 2150              s.send_out_unit_cost_php
  insert into cogs_send_out_entries (
    test_request_id, service_id, vendor_id, unit_cost_php=s.send_out_unit_cost_php,
    journal_entry_id=je.id, accrued_at=now()
  );
else
  -- D10: do not block, do not emit COGS lines, still record entry for visibility
  insert into cogs_send_out_entries (
    test_request_id, service_id, vendor_id=null, unit_cost_php=0,
    journal_entry_id=null, accrued_at=now()
  );
  insert into audit_log (
    actor_id=null, actor_type='system', action='send_out.unit_cost_missing',
    resource_type='test_request', resource_id=tr.id,
    metadata=jsonb_build_object('service_id', s.id, 'service_code', s.code)
  );
end if;
```

**For other kinds** (lab_test, lab_package, vaccine, home_service, etc.): unchanged from current 0030 behavior.

**Idempotency:** the partial unique index `journal_entries_one_posted_per_source` (from 0030 + 0037) prevents double-emit. No additional logic needed.

**Line-skipping rule:** every line emit is guarded by `if v_amount > 0 then INSERT`. This satisfies the `journal_lines` CHECK `debit_php > 0 or credit_php > 0` for shareholder doctors (clinic_fee=0).

### 4.2 Modified: `bridge_test_request_cancelled`

Existing logic: mirror the original JE with debits/credits swapped, `source_kind='reversal'`. 12.5 additions:

- Same mirroring works for the new split lines (no change to the reversal-JE shape; lines reverse 1:1).
- After the reversal JE posts: `UPDATE doctor_pf_entries SET voided_at=now() WHERE test_request_id = tr.id AND voided_at IS NULL;` (cancellation also voids any pending PF entry).
- `UPDATE cogs_send_out_entries SET voided_at=now() WHERE test_request_id = tr.id AND voided_at IS NULL;` (same for COGS).

### 4.3 NEW: `trg_bridge_pf_at_hmo_allocation`

Fires `AFTER INSERT ON hmo_payment_allocations`. For each new allocation row:

```
v_item := select * from hmo_claim_items where id = NEW.item_id;
v_tr   := select * from test_requests where id = v_item.test_request_id;
v_pfe  := select * from doctor_pf_entries
          where test_request_id = v_tr.id
            and recognition_basis = 'hmo_at_settlement'
            and recognized_at is null
            and voided_at is null;

if v_pfe.id is null then return new; end if;  -- no pending HMO PF entry; nothing to do

settlement_ratio := NEW.amount_php / v_item.billed_amount_php;
pf_to_accrue := round(v_tr.doctor_pf_php * settlement_ratio, 2);

if pf_to_accrue <= 0 then return new; end if;

-- Posting date derives from the parent payment row (allocations table has no
-- date column of its own; use the payment's posting_date for GAAP-correct timing).
v_posting_date := (select posting_date from payments where id = NEW.payment_id);

draft_je(source_kind='doctor_pf_accrual', source_id=v_pfe.id,
         posting_date=v_posting_date);
  line: DR 2160               pf_to_accrue
  line: CR 2110               pf_to_accrue

update doctor_pf_entries set
  recognized_at = now(),
  journal_entry_id = je.id,
  hmo_allocation_id = NEW.id,
  pf_php = pf_to_accrue   -- snapshot the actual accrued amount (may be < tr.doctor_pf_php for partials)
where id = v_pfe.id;
```

**Trigger name resolution:** existing 0034 trigger on related ops is `tg_bridge_hmo_claim_resolution_insert` (different table — `hmo_claim_resolutions`). On `hmo_payment_allocations`, the only existing AFTER INSERT trigger is `tg_hmo_alloc_paid_recompute` (denorm). The new trigger fires after that (alphabetical: `tg_hmo_alloc_paid_recompute` → `trg_bridge_pf_at_hmo_allocation`). Denorm completes first; no shared row UPDATE conflict.

### 4.4 NEW: `trg_bridge_pf_at_hmo_writeoff`

Fires `AFTER INSERT ON hmo_claim_resolutions WHEN destination='write_off'`. For each:

```
v_item := select * from hmo_claim_items where id = NEW.item_id;
v_tr   := select * from test_requests where id = v_item.test_request_id;
v_pfe  := select * from doctor_pf_entries
          where test_request_id = v_tr.id
            and recognition_basis = 'hmo_at_settlement'
            and recognized_at is null
            and voided_at is null;

if v_pfe.id is null then return new; end if;

writeoff_ratio := NEW.amount_php / v_item.billed_amount_php;
pf_to_clear := round(v_tr.doctor_pf_php * writeoff_ratio, 2);

if pf_to_clear <= 0 then return new; end if;

draft_je(source_kind='doctor_pf_accrual', source_id=v_pfe.id,
         posting_date=NEW.resolved_at::date);
  line: DR 2160               pf_to_clear
  line: CR 6920               pf_to_clear

update doctor_pf_entries set
  voided_at = now(),
  void_reason = 'hmo_writeoff',
  pf_php = pf_to_clear,
  journal_entry_id = je.id
where id = v_pfe.id;
```

Net economic effect for the denied portion: revenue earned = clinic_fee (4200), bad debt expensed = clinic_fee from existing 12.3 writeoff (offsetting net = 0 — correct, since neither party got paid). Doctor PF holding cleared without distorting revenue.

### 4.5 NEW: `trg_bridge_payment_void_pf_cascade`

Fires `AFTER UPDATE ON payments WHEN OLD.voided_at IS NULL AND NEW.voided_at IS NOT NULL`. For each test_request in the voided payment's visit:

```
v_visit_paid := select payment_status from visits where id = NEW.visit_id;

if v_visit_paid = 'paid' then return new; end if;   -- visit still paid via other payments; do nothing

for tr in (select * from test_requests where visit_id = NEW.visit_id) loop
  for pfe in (select * from doctor_pf_entries
              where test_request_id = tr.id
                and recognition_basis = 'cash_at_release'
                and voided_at is null) loop

    if pfe.disbursement_id is not null then
      -- D-Risk-2 (b): PF was already disbursed; surface alert, don't auto-reverse
      insert into audit_log (action='pf_clawback.alert',
        resource_type='doctor_pf_entries', resource_id=pfe.id,
        metadata=jsonb_build_object('visit_id', NEW.visit_id,
                                    'physician_id', pfe.physician_id,
                                    'pf_php', pfe.pf_php));
      -- Insert a clawback row that will surface as negative-balance UI on next EOD page
      insert into doctor_pf_entries (
        test_request_id, physician_id, pf_php=(-pfe.pf_php),
        recognition_basis='clawback', recognized_at=now(), journal_entry_id=null
      );
      continue;
    end if;

    -- PF not yet disbursed: reverse the original accrual JE + soft-void the entry
    reverse_je_for_source(source_kind='test_request', source_id=tr.id, mode='partial_pf_only');
    -- Reversal limited to the 2110 credit + cash debit attributable to PF (see helper below)
    update doctor_pf_entries set voided_at = now(), void_reason='payment_voided' where id = pfe.id;
  end loop;
end loop;
```

Helper `reverse_je_for_source(... mode='partial_pf_only')` is a SECURITY DEFINER function that drafts a reversal JE for only the cash + 2110 lines (preserving 4200 since the clinic_fee revenue stays recognized — patient was at the counter; that's not in dispute).

### 4.6 NEW: `trg_bridge_pf_disbursement_post`

Fires `AFTER INSERT ON doctor_pf_disbursements`. Standard JE pattern:

```
draft_je(source_kind='doctor_pf_disbursement', source_id=NEW.id, posting_date=NEW.posted_date);
case NEW.method
  when 'cash'           then line: CR 1010 NEW.total_php
  when 'gcash'          then line: CR 1030 NEW.total_php
  when 'bank_transfer'  then line: CR 1020 NEW.total_php  -- BPI by default; admin can override via cash_routing if 1021 needed
end case;
  line: DR 2110           NEW.total_php
post_je();

update doctor_pf_disbursements set journal_entry_id = je.id where id = NEW.id;
```

The disbursement Server Action sets `disbursement_id` on each cleared `doctor_pf_entries` row before inserting the disbursement header, so the FK is consistent at JE-emission time.

### 4.7 NEW: `trg_bridge_cogs_send_out_trueup`

Fires `AFTER INSERT ON cogs_send_out_trueups`. Variance JE logic (4 cases):

```
v_variance := NEW.billed_total_php - NEW.accrued_total_php;

if v_variance = 0 then
  -- No variance; mark entries as trued-up without a JE
  update cogs_send_out_entries set trueup_id = NEW.id, trued_up_at = now()
  where vendor_id = NEW.vendor_id
    and accrued_at::date between NEW.period_start_date and NEW.period_end_date
    and trueup_id is null and voided_at is null;
  return new;
end if;

draft_je(source_kind='cogs_send_out_trueup', source_id=NEW.id, posting_date=NEW.matched_at::date);
if v_variance > 0 then
  -- under-accrued: book more COGS, credit liability
  line: DR 6420  v_variance
  line: CR 2150  v_variance
else
  -- over-accrued: reverse some COGS
  line: DR 2150  abs(v_variance)
  line: CR 6420  abs(v_variance)
end if;
post_je();

update cogs_send_out_entries set trueup_id = NEW.id, trued_up_at = now()
where vendor_id = NEW.vendor_id
  and accrued_at::date between NEW.period_start_date and NEW.period_end_date
  and trueup_id is null and voided_at is null;
update cogs_send_out_trueups set journal_entry_id = je.id where id = NEW.id;
```

Edge case `accrued=0, billed=X` is handled by the `v_variance > 0` branch (variance = full billed amount), emitting a full DR 6420 / CR 2150. The corresponding 12.4 AP bill posting (`DR 6420 or 2150 / CR 2100 AP Trade`) separately closes the AP loop.

## 5. Server Actions

All actions live under `src/lib/actions/accounting/`. Standard return shape `{ ok: true, data } | { ok: false, error }`. Every write writes `audit_log` with descriptive action name + metadata.

| File | Actions | Notes |
|---|---|---|
| `pf-disbursements.ts` | `createPfDisbursement`, `voidPfDisbursement` | Per-doctor; uses `pf_disbursement_year_counters` for batch_number. Void reverses JE via draft-flip pattern. |
| `pf-bulk-payout.ts` | `createBulkPfPayoutCash` | "Pay all (cash)" bulk action. Atomic: all-or-nothing. Iterates physicians; each gets its own disbursement row. |
| `cogs-send-out-trueups.ts` | `createSendOutTrueup`, `voidSendOutTrueup` | Computes accrued total + variance server-side. Validates bill_id belongs to vendor. |
| `physicians-compensation.ts` | `updateCompensationArrangement`, `recomputeClinicFeeForUnreleased` | Compensation arrangement field admin edit; post-classification scrub action (see §3.5). |
| `services-send-out.ts` | `updateSendOutConfig` | Sets unit_cost + vendor; refuses save if is_send_out=true and either is null (UI-level validation backed by DB-level check). |
| `visits-attending.ts` | `setVisitAttendingPhysician`, `setLineAttendingPhysician` | Updates visit.attending_physician_id or test_requests.attending_physician_id. Used by /staff/visits/new and edit paths. |

### Server Action ↔ trigger boundary

Actions never emit JEs directly. Actions INSERT into subledger tables; triggers emit JEs. This matches 12.3/12.4 pattern.

Exception: void actions (`voidPfDisbursement`, `voidSendOutTrueup`) MUST use the draft-flip pattern (`feedback_je_cleanup_pattern.md`):
1. UPDATE posted JE to status='draft'
2. INSERT reversal JE (via `reverse_je_for_source` helper or inline)
3. UPDATE original JE back to status='posted' (or to 'reversed' per existing 12.4 pattern)
4. Soft-void the subledger row

## 7. RLS + audit

### 7.1 RLS policies

All four new subledger tables (`doctor_pf_entries`, `doctor_pf_disbursements`, `cogs_send_out_entries`, `cogs_send_out_trueups`) + `pf_disbursement_year_counters`:

- **Enable RLS.**
- **Admin SELECT policy** for all four entry/disbursement/trueup tables. Reception does not read these tables directly (visit detail badge data is computed server-side via the page's data fetch under admin role check; reception sees rendered output, not raw rows).
- **Counter table has NO SELECT or write policies** — only SECURITY DEFINER trigger functions touch it (mirroring `bill_payment_year_counters`).

```sql
alter table public.doctor_pf_entries enable row level security;
create policy "doctor_pf_entries: admin read"
  on public.doctor_pf_entries for select to authenticated
  using (public.has_role(array['admin']));

-- analogous for the other three tables
```
