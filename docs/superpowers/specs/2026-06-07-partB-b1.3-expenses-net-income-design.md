## Goal

Reproduce the lower half of the clinic's manual **`DAILY MONITORING`** sheet:
the **17-line EXPENSES block (rows 70–86)** → **TOTAL EXPENSES (87)** →
**NET INCOME (88)**, and the **CASH FLOW roll (rows 125–128)**
(Starting + Collected − Expenses = Ending). Rendered as a 4th **"Expenses & P&L"**
tab under `Admin → Operations`, in the same day-matrix idiom as B1.1/B1.2.

This makes B1.1's placeholder "rough Net" real: instead of `revenue − Σ(all
expense accounts)`, the report now shows the full per-line P&L grouped exactly
like the sheet, and ties to the GL Income Statement.

## Architecture

### Data layer — two new views (migration 0096)

**(1) `v_ops_daily_expense_accounts`** — grain `(business_date, code, name)`:

```sql
create or replace view public.v_ops_daily_expense_accounts
with (security_invoker = on) as
select
  je.posting_date                                       as business_date,
  coa.code,
  coa.name,
  coalesce(sum(jl.debit_php - jl.credit_php), 0)::numeric(14,2) as expense_php
from public.journal_lines jl
join public.journal_entries je  on je.id = jl.entry_id
join public.chart_of_accounts coa on coa.id = jl.account_id
where coa.type = 'expense'
  and je.status = 'posted'
group by je.posting_date, coa.code, coa.name;

alter view public.v_ops_daily_expense_accounts owner to postgres;
```

- Same logic as `0094 v_ops_daily_expenses`, but **per-account** (keeps `code` +
  `name`) instead of a single total. `Σ expense_php` over a day equals the 0094
  total by construction → the two views stay consistent.
- `posting_date` is a plain DATE (no Manila cast) — matches the Income Statement
  and 0094. (Cash collected, from B1.2, is cash-basis on `received_at`; the
  cash-flow note states this accrual-vs-cash mix.)
- `security_invoker = on`, **no** grant to `anon`/`authenticated` — admin
  service-role client only (clinic-wide financials past patient RLS). Same posture
  as 0093/0094/0095.
- `0094 v_ops_daily_expenses` **stays** (B1.1's rough-net card still uses it).
- Applied to prod via the Supabase **MCP** (IPv6 limitation,
  `feedback_remote_db_ops_ipv6`); both views' `Row` types **surgically inserted**
  into `src/types/database.ts` (no wholesale overwrite — known schema drift).

The second view (`v_ops_daily_pnl`) is defined in the next subsection.

### Reconciliation-to-books net income — second small view `v_ops_daily_pnl`

The "reconciliation to books" figure (the formal GL Income-Statement net income
for the range) needs server-side aggregation of revenue / contra_revenue / expense
balances. The Supabase JS client **cannot** do a `GROUP BY` aggregate inline
(only `select`/filter against a view or RPC), so this is a **second small view**
— grain `(business_date)`, three pre-signed balance columns. (Views are the
established pattern here — 0093/0094/0095 — and this one is reusable for the B2
Trends pack.)

```sql
create or replace view public.v_ops_daily_pnl
with (security_invoker = on) as
select
  je.posting_date as business_date,
  -- per-account normal_balance sign convention (mirrors financial-statements/page.tsx):
  --   credit-normal → credit - debit ; debit-normal → debit - credit
  coalesce(sum(case when coa.type='revenue'
    then (jl.credit_php - jl.debit_php) else 0 end),0)::numeric(14,2) as revenue_php,
  coalesce(sum(case when coa.type='contra_revenue'
    then (jl.debit_php - jl.credit_php) else 0 end),0)::numeric(14,2) as contra_revenue_php,
  coalesce(sum(case when coa.type='expense'
    then (jl.debit_php - jl.credit_php) else 0 end),0)::numeric(14,2) as expense_php
from public.journal_lines jl
join public.journal_entries je  on je.id = jl.entry_id
join public.chart_of_accounts coa on coa.id = jl.account_id
where je.status='posted'
  and coa.type in ('revenue','contra_revenue','expense')
group by je.posting_date;

alter view public.v_ops_daily_pnl owner to postgres;
```

`balance` columns are each positive (revenue credit-normal, contra_revenue +
expense debit-normal). Reconciliation:
- `books_net_income = Σ(revenue_php − contra_revenue_php − expense_php)` over the range.
- Cross-check: `Σ v_ops_daily_pnl.expense_php = Σ v_ops_daily_expense_accounts.expense_php
  = 0094 v_ops_daily_expenses` (the books-tie invariant, asserted in `validate.sql`).

A pure helper `booksNetIncome(rows)` does the subtraction (vitest-tested);
`validate.sql` asserts the same on prod.

Migration **0096** therefore ships **two** views: `v_ops_daily_expense_accounts`
(per-account, the 17 lines) and `v_ops_daily_pnl` (per-day, the recon). Both
`security_invoker = on`, no grant; both `Row` types surgically inserted into
`database.ts`.

### Pure core — `src/lib/operations/expense-report.ts` (vitest, no `server-only`)

Reuses `num`, `enumerateDays`, `groupDaysByMonth` from `daily-report.ts`.

- **`EXPENSE_LINES`** — static config: the 17 lines in order, each with `{ category,
  label, codes: string[] }`; Benefits has `codes:['6120','6121','6122','6123','6124']`,
  all others a single code. A `CATEGORY_ORDER` lists the 4 categories.
- **`buildExpenseMatrix(rows, days)`** → `{ days, categories: [{ name, lines:[{label, byDay,
  total}], subtotal }], other: MatrixRow|null, total: MatrixRow }`. Maps each view
  row's `code` to its line (unknown codes → the "Other expenses" bucket, emitted
  only when non-zero). The TOTAL row sums **all** lines + other (the
  ties-to-books invariant; asserted in tests).
- **`buildNetIncome(grossProfitByDay, totalExpensesByDay, days)`** → operational
  net income per day. `grossProfitByDay` comes from B1.1's `v_ops_daily_totals`
  (`net`, lab+consult, reusing `buildDailyMatrix(...).totals.net`).
- **`buildCashFlow(collectedByDay, expensesByDay, days)`** → rows Starting / (+)
  Collected / (−) Expenses / (=) Ending, **running balance from ₱0** at the start
  of the selected range. `collectedByDay` reuses B1.2's
  `buildCollectionsMatrix(...).total` (TOTAL CASH COLLECTED) row.
- Small helpers: `largestCategory(matrix)` (for a summary card), `cashFlowEnding`
  (Σcollected − Σexpenses over range), and **`booksNetIncome(rows)`** —
  `Σ(revenue_php − contra_revenue_php − expense_php)` over `v_ops_daily_pnl` rows
  (the reconciliation figure).

### UI — 4th "Expenses & P&L" tab

Route `src/app/(staff)/staff/(dashboard)/admin/operations/expenses/page.tsx`
(Server Component, `requireAdminStaff`). Added to `_components/operations-tabs.tsx`
**before** the Trends stub: *Daily report · Cash & cards · Expenses & P&L · Trends*.

Mirrors B1.1/B1.2 exactly: shared `date-controls.tsx` (This month / This year /
Last year + jump-to-year Select + custom from/to; default current year; pass
`basePath` for this route), navy brand, shadcn `ui/*` primitives, collapsible-month
matrix (`groupDaysByMonth`), sticky label column + horizontal scroll on mobile
(390×844), en-PH PHP inline formatter, **no recharts**.

The page fetches in parallel (admin client):
1. `v_ops_daily_expense_accounts` (range) → `buildExpenseMatrix`
2. `v_ops_daily_totals` (range) → gross profit → `buildNetIncome`
3. `v_ops_daily_collections` + `v_ops_daily_hmo_received` (range) →
   `buildCollectionsMatrix(...).total` → `buildCashFlow`
4. `v_ops_daily_pnl` (range) → `booksNetIncome` → reconciliation figure

**Layout (top → bottom):**
- **Summary cards:** Total expenses · Net income (operational) · Largest expense
  category · Cash-flow ending (net operational flow) — for the range.
- **Expenses matrix** (`_components/expense-matrix.tsx`): 4 category groups, each
  with its line items + a **category subtotal**; then **Other expenses** (only if
  non-zero); then **TOTAL EXPENSES**. Columns = days, collapsible by month.
- **P&L block** (`_components/pnl-summary.tsx`): TOTAL GROSS PROFIT (lab+consult,
  from B1.1) → − TOTAL EXPENSES → **= NET INCOME (operational)**, plus a muted
  **"Reconciliation to books"** line = the formal GL Income-Statement net income
  for the range, with a note that the difference is the **excluded rent/APE/
  procedures revenue** (B1.1 caveat) **plus** accrual `posting_date` vs `released_at`
  month-boundary timing.
- **Cash-flow block** (`_components/cash-flow-panel.tsx`): Starting / (+) Cash
  collected / (−) Expenses / (=) Ending, running from ₱0, with the note:
  *"operational roll from ₱0; not a bank-reconciled balance. Collections are
  cash-basis, expenses accrual; the ₱100/day shareholder rent is not a payment row
  and is excluded from collected."*
- **CSV export** at `src/app/api/admin/operations/expenses.csv/route.ts` (mirrors
  `cash.csv`/`daily.csv`): the full expense matrix + P&L + cash-flow rows.

### Data flow

Server Component → `createAdminClient()` (read-only) → 4 parallel reads filtered by
`business_date` / `posting_date` range → pivot in JS via the pure core → render.
No Server Actions (read-only screen). Standard error/empty handling: friendly "no
activity" state; the short server-error pattern (no stack traces).

   Income-Statement total expenses** to the peso.
2. **Net income**: operational (B1.1 gross profit − total expenses) computes; the
   books-recon aggregate computes; the diff is explained (excluded revenue +
   timing).
3. **Sheet spot-check Dec 2 2023**: r70 Salaries 700 / r71 Doctors Payroll 80 /
   r87 TOTAL 780 / r88 NET 6,630 — validate the per-line mapping against a real
   sheet day. (Expect minor divergence only from the catch-all / unmapped, which
   is ₱0.)
