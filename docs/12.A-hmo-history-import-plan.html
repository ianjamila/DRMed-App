05 · Risks & dependencies

What could derail the plan

Each risk is paired with the mitigation already baked into the plan (task ID where it's addressed).

Bridge bypass leaks past the commit transactioncritical
Mitigation: SET LOCAL in T28 limits to the transaction; T28 also explicitly unsets the flag before posting opening JEs (belt-and-braces). D1 smoke (T8/T9) confirms the mechanism in isolation before any UI work touches it.
Service-role client used from client codecritical
Mitigation: Every Server Action in T18–T20 + T29 starts with "use server". The admin client is imported only from actions.ts; client components call through Server Actions.
HMO REFERENCE aging-block layout drift from spec §3warn
Mitigation: T15 ships a "scan for max numeric value" fallback so reconciliation degrades gracefully if the column layout shifts; the implementer refines against the live workbook during D2. Spec §17 documents this as an accepted deferred refinement.
Workbook header drift breaks parsing silentlywarn
Mitigation: T14 asserts canonical headers at row 1 of each tab; mismatch returns a precise error message naming the missing columns.
Concurrent imports collidewarn
Mitigation: T28's commit function takes pg_advisory_xact_lock(hashtext('hmo_history_import')). Second concurrent commit waits or aborts cleanly.
Reception sees thousands of historical patientsmanageable
Mitigation: T1 adds partial indexes where is_historical = false to back the reception filters. Operational pages must add WHERE is_historical = false in queries (spec §5A enumerates the affected routes).
Re-import accidentally changes existing patientsmanageable
Mitigation: T28 step 1 uses ON CONFLICT (last_name, first_name) WHERE is_historical = true DO NOTHING — never touches operational patients.
Cutover lands in a closed periodmanageable
Mitigation: DB period-guard P0001 trigger refuses the opening JE; T29 surfaces the error through translatePgError, commit rolls back. Admin reopens via /staff/admin/accounting/periods and re-commits.

06 · Handoff

Execution options

The plan document is the source of truth for execution. Two execution modes are available; the recommended one for this size of change is subagent-driven.

Recommended · Subagent-driven

One fresh subagent per task, reviewed between tasks. Fast iteration, narrow context window per subagent, clear blast radius. Requires the orchestrator to apply the superpowers:subagent-driven-development sub-skill.

Alternative · Inline execution

Execute tasks in the current session with batch checkpoints for review. Slower context-wise, but simpler tooling. Requires superpowers:executing-plans.
