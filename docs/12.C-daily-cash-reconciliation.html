Phase 12 · Sub-project 12.C

Daily Cash Reconciliation

Reception's end-of-day close moves from the DAILY MONITORING tab in DR MED MASTERSHEET into the system. Every cash payout, every float adjustment, and every till count fires a balanced JE through the 12.2 bridge — the GL stays in lockstep with operational reality, and stale cash gets surfaced the moment the variance doesn’t zero out.

01 · Executive summary

Three sub-deliverables, one daily flow

12.C is the reception-facing end of Phase 12. It takes the work the bookkeeper currently does in the DAILY MONITORING tab and re-homes it in the app, while extending the GL bridge so the books reconcile automatically instead of after the fact.

In scope

Reception: /staff/payments/cash-drawer (running daily flow + payouts) & /staff/payments/eod (close). Admin: /cash-routing (kind → contra map), /reports/daily-revenue (service × day pivot), /reports/staff-advances (outstanding balances). Bridge: triggers on eod_cash_adjustments + eod_close_records emit balanced JEs. Receivable: salary_advance payouts create staff_advances rows that 12.6 will draw down via payslip deductions.

Out of scope · deferred

Bank / wallet reconciliation for GCash·BPI·Maybank·card (future 12.D) · petty-cash voucher print · cash-advance payslip deduction (owned by 12.6 when payroll lands) · multi-shift handover-count UI (schema supports it; defer the UI) · bulk-edit / bulk-void of cash adjustments.

02 · Decisions

Locked in from the brainstorm

Five clarifying questions (Q1–Q5), three standard defaults (A/B/C), and one user-added field requirement. Each one drove a concrete schema or UX implication.

QDecisionImplication
Q1 Flex schema. v1 ships one default shift; multi-shift unlocks later without migration. cash_shifts lookup. eod_close_records keyed on (business_date, shift_id). UI hides the shift picker while only one is active.
Q2 Adjustable float. Baseline setting + per-date float_topup / float_pullout adjustments. New accounting_settings KV table (key default_change_fund_php, default ₱2,000). Effective opening = baseline + Σ float adjustments.
Q3 Dedicated /staff/payments/cash-drawer. Running daily flow lives here; EOD page consumes the totals. Two reception routes. Both reach for cash_drawer_state(date, shift_id) SECURITY DEFINER function as the single source of truth for math.
Q4 Admin-only daily-revenue pivot. Service × day. NOT on reception's EOD page. Read-only view v_daily_revenue_by_service at /staff/admin/reports/daily-revenue. Sidecar deliverable.
Q5 Full staff_advances integration now. 12.6 (payroll) draws down later. New staff_advances receivable table maintained by trigger. View v_staff_advances_outstanding drives admin report.
+ closed_by on the close record. Two receptionists may be on duty simultaneously. FK column on eod_close_records. Per-payment received_by already captures the other on-duty actor.
A Variance threshold. Any non-zero variance requires a reason note. No tolerance. CHECK (variance_php = 0 OR variance_reason IS NOT NULL) + Zod refinement client-side.
B Method coverage. v1 counts physical cash only. GCash / BPI / Maybank / card balances display on the drawer page for context but are not reconciled. Future 12.D owns bank reconciliation.
C Reopen-after-close. Closed records are DB-locked; admin reopens with audit. P0015 trigger blocks writes to payments + eod_cash_adjustments targeting closed (date, shift). reopenEodCloseAction flips status, audits reason.
03 · Architecture

Where the bytes flow

Same shape as 12.2: operational writes flow into trigger-driven JEs in the same transaction. Reception's existing payment surface is untouched; 12.C only adds tables, routes, and triggers.

Daytime activity
Reception logs payouts & float adjustments on /cash-drawer. Payments continue to land via the visit page (untouched).
↓
eod_cash_adjustments (INSERT / UPDATE-void)
Triggers bridge_cash_adjustment_insert, bridge_cash_adjustment_void, and staff_advance_sync fire in the same transaction.
Guard triggers
P0015 (eod_lock) blocks writes targeting a closed (date, shift). P0017 freezes JE-bearing rows.
↓
End of day
Reception goes to /eod, counts physical cash, enters reason if variance ≠ 0, clicks Close day.
↓
eod_close_records (INSERT)
bridge_eod_close posts a "Cash Short / Over" JE when variance ≠ 0. P0015 now blocks further cash-side writes for that (date, shift).
↓
journal_entries · journal_lines (12.1 GL spine)
New je_source_kind values: cash_adjustment · eod_close. Existing partial unique index from 0030 covers idempotency unchanged.
Architectural invariants

(1) Debits = credits on every posted JE. (2) No JE posts to a closed accounting period (P0002 surfaces). (3) One operational event → one posted JE (partial unique index, unchanged). (4) CoA is append-only. (5) Reception's existing payment UI is untouched. (6) RLS is the source of truth for access. (7) Every write is audit-logged.

Five tables, three CoA codes, two enum values

All changes ride in migration 0043_eod_cash_reconciliation.sql. The CoA seed is idempotent (ON CONFLICT (code) DO NOTHING) so fresh-clone supabase db reset keeps working.

07 · Errors

New pg-error codes

Claimed range P0015–P0019. P0016 is reserved (the variance_reason requirement is enforced client-side via Zod refinement; CHECK constraint doubles as safety net but should never surface).

CodeNameUser-facing message
P0015eod_lockEnd of day is already closed for that date. Ask an admin to reopen first.
P0017cash_adjustment_immutableCannot edit this cash adjustment after its journal entry has posted. Void and re-create instead.
P0018staff_advance_overdrawStaff advance cannot go below zero.
P0019cash_account_map_inactiveThat account is inactive. Pick a different one.
Translation layer

src/lib/accounting/pg-errors.ts gets four new case arms. Server Actions continue to return { ok: true, data } | { ok: false, error } with the translated message. Sentry receives the raw PgError via reportError as usual.

09 · Risks & dependencies

Things that could bite

Each risk is rated by likelihood × blast radius. Mitigations are concrete and either ship in the migration (DB-enforced) or are owned by a specific dispatch.

P0015 fires on legitimate void of pre-close payments High impact · medium likelihood
Scenario: A payment recorded at 14:00 turns out to be a duplicate; reception tries to void at 19:00 after the day is closed. Current spec blocks. Mitigation: Documented behavior — voids after close require admin reopen because they retroactively change cash balance. UI surfaces a "Day closed · admin must reopen" message with the close metadata. Alternative (re-snapshot variance) was considered and rejected as too magical for v1.
P0015 needs single-active-shift assumption on payments Tech-debt risk · low likelihood
Scenario: v1 infers shift via "the one active row in cash_shifts" because payments doesn't carry a shift_id. When multi-shift UI ships, every payment will need to claim a shift. Mitigation: Comment in trigger explicitly references this; deferred follow-up adds payments.shift_id with a backfill. Schema is forward-compatible.
Closed accounting period blocks an EOD close Medium impact · low likelihood
Scenario: Admin closes Q1 2026 (per 12.1), then reception tries to close a back-dated day in Q1. bridge_eod_close JE trips P0002. Mitigation: The Server Action surfaces the canonical "That accounting period is closed." message. Admin must reopen the period in /staff/admin/accounting/periods before the close can land. Documented in spec §6.6.
Race condition: two receptionists both click "Close day" Low impact · medium likelihood
Scenario: Both receptionists submit close form within seconds. Mitigation: UNIQUE (business_date, shift_id) WHERE status='closed' partial index. Second submission trips 23505 → translated to "Day already closed at HH:MM by X" in the UI.
Suspense postings under-surfaced Operational hygiene
Scenario: Reception ignores the contra-account picker; bridge falls back to 9999 with a coa.suspense_post audit row. Visible only in raw audit_log today. Mitigation: Deferred follow-up — add a "Suspense activity" sidecar on the cash-routing page (and on payment-routing while we're there). Same audit query, just surfaced.
Daily-revenue pivot performance on year-range queries Performance · low likelihood
Scenario: Admin opens daily-revenue at year scale (~50 services × 365 days = 18K cells). Mitigation: Default range is current month. View is a simple GROUP BY on test_requests + services; existing indexes handle it. Add range cap (max 12 months) in the page client if it becomes a concern.
Bridge idempotency already proven (12.2 / 12.3 / 12.A) No action needed
Partial unique index on journal_entries (source_kind, source_id) WHERE status='posted' AND source_kind NOT IN ('reversal', 'hmo_history_opening') already permits the two new source kinds. Tested under load across three sub-projects.
