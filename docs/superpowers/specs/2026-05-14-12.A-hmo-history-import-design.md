## 1. Goal

Load all HMO transactions captured manually in the DR MED MASTERSHEET workbook (Dec 2023 → 2025-12-31) into the 12.3 subledger so that:

1. Per-provider opening AR balance at 2025-12-31 is correct in the GL (one balanced JE per provider).
2. Every historical HMO claim is queryable in `v_hmo_ar_aging` / `v_hmo_stuck` / `v_hmo_provider_summary` with the same drill-down UX as operational claims.
3. The "claims we forgot to bill" audit answer lands (rows where `E=YES` in the workbook but `V` REFERENCE/TRACKING is blank → synthetic `draft` batches surface in `v_hmo_stuck`).

Out of scope for 12.A:
- Historical P&L data (revenue, COGS) — deferred to 12.B.
- Historical cash receipts (the JE side of the OR# settlements) — deferred to 12.B.
- VAT / withholding tax on historical claims — deferred to Future Phase 1.
- Patient-bill / write-off resolution of historical stuck claims — admin uses the existing 12.3 UI post-import.

## 2. Decisions locked in (from brainstorm 2026-05-14)

| Decision | Choice | Why |
|---|---|---|
| Detail level | **Per-claim detail** (every HMO row in LAB SERVICE + DOCTOR CONSULTATION becomes one `hmo_claim_item`) | Aggregate-only loses the audit answer for stuck and unbilled claims. |
| FK strategy | **Full synthetic op-rows** with `is_historical` flags | Subledger consistency; per-patient drill-down works; reception UI filters synthetic rows. |
| JE strategy | **Bypass bridge** during import; one **opening-balance JE per provider** posted at the cutover date | Avoids reopening 12 closed quarters; simpler audit trail; P&L 2023–2025 stays as the user's Income Statement tabs already reflect. |
| Cutover date | **2025-12-31** | Clean year boundary; aligns with the Income Statement Monthly tabs already used to populate the CoA. |
| Cutover semantics | **Net unpaid AR only** in the opening JE (sum of `billed - paid - patient_billed - written_off` for items not fully resolved) | The Balance Sheet at cutover is correct; cash receipts history stays out of scope for 12.A. |
| Opening JE granularity | **One JE per provider** with non-zero opening AR | Easy to void/correct surgically. Up to 10 JEs total. |
| Import UX | **Upload XLSX or CSV → preview → commit** at `/staff/admin/accounting/hmo-history` | Single admin page, dry-run friendly, supports re-runs. |
| Unknown providers | **Block commit** until admin maps each via `hmo_provider_aliases` | Prevents accidental duplicate provider rows. |
| Unknown services | **Block commit** until admin maps each via `hmo_service_aliases` | Same reasoning; needed because LAB SERVICE `SERVICE` is free text and DOCTOR CONSULTATION `SERVICE` contains doctor last names. |
| Patient dedup | **Normalized name only** (case-insensitive, trimmed, comma-spacing collapsed) | False-merges affect drill-down only, not AR balance; pragmatic given workbook has no DOB/PHIC IDs. |
| Unbilled rows | **Synthetic `draft` batch per `(provider_id, year-month)`** for rows with no REFERENCE/TRACKING | Surfaces them in `v_hmo_stuck` immediately so admin can resolve via the existing 12.3 UI. |
| Payment rollup | **One synthetic `payments` row per unique `(provider_id, or_number)`** | Mirrors reality: one HMO check covers multiple claims. |
| Visit grouping | **Group rows by `(source_date, normalized_patient_name, provider_id)`** into one synthetic visit with N test_requests | Realistic drill-down ("Alava had three tests on one visit"); smaller op-row count. |
| Doctor consultation services | **Treat each doctor name as a separate `services` row** of `kind='doctor_consult'`, mapped via `hmo_service_aliases` | Per-doctor reporting works out of the box. |
| Reconciliation gate | **Compare computed opening AR vs HMO REFERENCE ending balance per provider; block commit on variance > 5%** | Single biggest quality control; catches data-entry gaps before they hit the GL. |
| Hidden tabs | **Skip `HMO AUDIT`, `LAB HMO (DEC 1-19)`, `CONSULT HMO (DEC 1-19)`** | Stale working artifacts; admin doesn't recognize HMO AUDIT and its formulas are broken (#REF!). Documented in audit trail. |
| Idempotency | **Content-hash natural key** `(source_tab, normalized_patient_name, source_date, provider_id_resolved, service_id_resolved, billed_amount, reference_no)` | Re-uploading the workbook is a no-op; small workbook edits get picked up; row reordering survives. |

## 5. Schema deltas (migration `0035_hmo_history_import.sql`)

### 5A. Op-table `is_historical` flags

```sql
ALTER TABLE public.patients
  ADD COLUMN is_historical boolean NOT NULL DEFAULT false;
ALTER TABLE public.visits
  ADD COLUMN is_historical boolean NOT NULL DEFAULT false;
ALTER TABLE public.test_requests
  ADD COLUMN is_historical boolean NOT NULL DEFAULT false;

-- Partial indexes keep the operational paths fast.
CREATE INDEX idx_patients_active        ON public.patients      (full_name) WHERE is_historical = false;
CREATE INDEX idx_visits_active          ON public.visits        (created_at DESC) WHERE is_historical = false;
CREATE INDEX idx_test_requests_active   ON public.test_requests (created_at DESC) WHERE is_historical = false;

-- Partial unique index supporting the patient upsert in §6 step 1.
-- Operational patients are keyed by drm_id; historical patients are deduped by full_name.
CREATE UNIQUE INDEX idx_patients_historical_one_per_name
  ON public.patients (full_name) WHERE is_historical = true;
```

Reception's existing list queries must add `WHERE is_historical = false`. Routes affected (read-only audit at plan time):
- `/staff/patients` list page
- `/staff/patients/[id]` detail (still readable individually for now; only the list filters)
- `/staff/visits` list page
- `/staff/dashboard` patient/visit counters
- `/staff/queue` (released test_requests visible to staff)

The `is_historical` filter is additive — admin pages (e.g., `/staff/admin/accounting/hmo-claims`) intentionally show both so historical claims appear in aging.

### 5B. `payments.notes` historical-tag convention

To avoid altering the `payments` table schema, synthetic historical payments get `notes = '[historical-import:' || run_id || ']' || coalesce(other_notes, '')`. The 12.3 bridge layer already ignores `payments.notes` content; only the `hmo_payment_allocations` rows matter for the subledger.

A view `v_historical_payments` is provided for admin reporting:
```sql
CREATE VIEW v_historical_payments AS
SELECT * FROM public.payments
 WHERE notes LIKE '[historical-import:%]%';
```

### 5D. Content hash for idempotency

```sql
-- content_hash is set during validateRunAction:
--   sha256(source_tab || '|' || normalized_patient_name || '|' || source_date::text
--          || '|' || provider_id_resolved::text || '|' || service_id_resolved::text
--          || '|' || billed_amount::text || '|' || coalesce(reference_no, ''))

-- Partial unique index — only enforced for committed rows, so dry-runs can re-stage freely.
CREATE UNIQUE INDEX idx_hmo_history_staging_one_commit_per_content
  ON public.hmo_history_staging (content_hash)
  WHERE status = 'committed';
```

A second commit run skips rows whose `content_hash` already exists in a committed row.

`content_hash` is computed during `validateRunAction` (after provider and service resolution), not at parse time, because the hash depends on `provider_id_resolved` and `service_id_resolved`. Staging rows with unresolved aliases have `content_hash = NULL` and cannot reach `status='validated'`.

## 6. The commit function

```sql
CREATE OR REPLACE FUNCTION public.commit_hmo_history_run(p_run_id uuid)
RETURNS jsonb
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
AS $$
DECLARE
  v_cutover date;
  v_summary jsonb := '{}'::jsonb;
  v_advisory_key bigint := hashtext('hmo_history_import');
  v_n_patients int := 0; v_n_visits int := 0; v_n_test_requests int := 0;
  v_n_batches int := 0;  v_n_items int := 0;  v_n_payments int := 0;
  v_n_allocations int := 0; v_n_jes int := 0;
BEGIN
  -- One concurrent import at a time.
  PERFORM pg_advisory_xact_lock(v_advisory_key);

  SELECT cutover_date INTO v_cutover FROM public.hmo_import_runs WHERE id = p_run_id FOR UPDATE;
  IF v_cutover IS NULL THEN
    RAISE EXCEPTION 'no such import run %', p_run_id USING ERRCODE = 'P0013';
  END IF;

  -- Refuse to commit if any error-severity validations remain.
  IF EXISTS (
    SELECT 1 FROM public.hmo_history_staging
     WHERE run_id = p_run_id
       AND status = 'validated'
       AND jsonb_path_exists(validation_errors, '$[*] ? (@.severity == "error")')
  ) THEN
    RAISE EXCEPTION 'commit blocked: error-severity rows remain' USING ERRCODE = 'P0014';
  END IF;

  -- Bypass the two JE-posting triggers for this transaction.
  SET LOCAL app.skip_bridge_historical = 'true';

  -- 1. Upsert patients. ON CONFLICT targets the partial unique index from §5A.
  WITH new_patients AS (
    INSERT INTO public.patients (full_name, drm_id, is_historical, created_at)
    SELECT DISTINCT
      normalized_patient_name,
      'HIST-' || substring(md5(normalized_patient_name) FROM 1 FOR 12),
      true,
      now()
    FROM public.hmo_history_staging
     WHERE run_id = p_run_id AND status = 'validated'
    ON CONFLICT (full_name) WHERE is_historical = true DO NOTHING
    RETURNING 1
  ) SELECT count(*) INTO v_n_patients FROM new_patients;

  -- 2-7: visits, test_requests, batches, items, payments, allocations (mechanical inserts)
  --      detailed in plan-level pseudocode; each step adds to v_summary counters.

  -- 8. Unset bypass before posting opening JEs.
  SET LOCAL app.skip_bridge_historical = '';

  -- 9. Compute per-provider net opening AR and post one JE per non-zero provider.
  INSERT INTO public.journal_entries (posting_date, source_kind, source_id, status, created_by, narrative)
  SELECT
    v_cutover,
    'hmo_history_opening',
    p_run_id,
    'posted',
    (SELECT uploaded_by FROM public.hmo_import_runs WHERE id = p_run_id),
    'Opening HMO AR for ' || hp.name || ' (12.A historical import)'
  FROM public.hmo_providers hp
  WHERE EXISTS (
    SELECT 1 FROM public.hmo_claim_items hci
      JOIN public.hmo_claim_batches hcb ON hcb.id = hci.batch_id
     WHERE hcb.provider_id = hp.id
       AND hcb.import_run_id = p_run_id
       AND hci.billed_amount_php - hci.paid_amount_php
         - hci.patient_billed_amount_php - hci.written_off_amount_php > 0
  );
  -- Inserting paired DR/CR lines via a second statement (see implementation plan for the exact join).
  GET DIAGNOSTICS v_n_jes = ROW_COUNT;

  -- 10. Stamp the run.
  UPDATE public.hmo_import_runs
     SET committed_at = now(),
         finished_at  = now(),
         summary      = jsonb_build_object(
                          'patients', v_n_patients,
                          'visits', v_n_visits,
                          'test_requests', v_n_test_requests,
                          'batches', v_n_batches,
                          'items', v_n_items,
                          'payments', v_n_payments,
                          'allocations', v_n_allocations,
                          'opening_jes', v_n_jes
                        )
   WHERE id = p_run_id;

  UPDATE public.hmo_history_staging SET status = 'committed' WHERE run_id = p_run_id AND status = 'validated';

  -- 11. Audit.
  INSERT INTO public.audit_log (action, actor_id, entity, entity_id, metadata)
  VALUES ('hmo_history_import.committed',
          (SELECT uploaded_by FROM public.hmo_import_runs WHERE id = p_run_id),
          'hmo_import_runs', p_run_id,
          v_summary);

  RETURN v_summary;
END;
$$;
```

## 7. Server Actions

All four return `{ ok: true, data } | { ok: false, error }` per CLAUDE.md convention.

### 7A. `parseWorkbookAction(formData: FormData)`

- Body: `file: File` (XLSX or CSV-per-tab as multipart, max 25 MB).
- Steps:
  1. Compute SHA-256 of the bytes.
  2. Insert `hmo_import_runs(run_kind='dry_run', file_hash, file_name, uploaded_by)`.
  3. Stream-parse with `exceljs` (XLSX) or `csv-parse` (CSV).
  4. For each target tab, read row 1; assert canonical headers (per §3). On mismatch, throw `header_mismatch`.
  5. For each subsequent row: filter (`E='YES'` and `DATE ≤ cutover`); insert into `hmo_history_staging` with `status='parsed'` (or `'skipped_post_cutover'`).
  6. Update `hmo_import_runs.staging_count`.
- Returns: `{ run_id, parsed_count, skipped_post_cutover_count }`.
- Audit: `hmo_history_import.uploaded` with file_hash + counts.

### 7B. `validateRunAction(run_id: uuid)`

- Steps:
  1. Load staging rows where `status='parsed'`.
  2. Normalize names, dates, amounts; resolve provider/service via aliases + exact match; compute `visit_group_key` and `content_hash`.
  3. Run per-row severity rules (§9.1).
  4. Run cross-row scans: cross-provider duplicate OR# (warn), workbook-wide patient-name → multiple DRM-IDs sanity.
  5. Parse HMO REFERENCE aging columns, compute per-provider ending balance at cutover, write to `hmo_import_runs.summary.reconciliation`.
  6. Flip rows to `status='validated'`; update `hmo_import_runs.error_count` and `warning_count`.
- Returns: `{ error_count, warning_count, unmapped_providers: [...], unmapped_services: [...], reconciliation: [...] }`.
- Audit: `hmo_history_import.validated` with counts.

### 7C. `mapProviderAliasAction(run_id, alias, provider_id | 'create')`

- Inserts into `hmo_provider_aliases`. If `provider_id='create'`, first inserts a new `hmo_providers` row with the alias as the name.
- Re-runs the provider-resolution step for that alias across the staging table.
- Returns `{ remaining_unmapped: int }`.
- Audit: `hmo_provider_alias.mapped` with before/after.

### 7D. `mapServiceAliasAction(run_id, alias, service_id | 'create')`

- Same shape; for `create`, a new `services` row is inserted with `kind='lab_service'` (LAB SERVICE tab) or `kind='doctor_consult'` (DOCTOR CONSULTATION tab) — the tab is captured at parse time.
- Audit: `hmo_service_alias.mapped`.

### 7E. `commitRunAction(run_id, variance_override_reason?)`

- If any provider has reconciliation variance > 5%, `variance_override_reason` is required and stored on the run.
- Calls `commit_hmo_history_run(p_run_id)` via the service-role client.
- Returns the summary JSON.
- Audit: handled inside the SQL function.

### 7F. `discardRunAction(run_id)`

- Soft-deletes (status='discarded') for non-committed runs; refuses for committed.

## 9. Validation rules and severities

### 9.1 Per-row rules (run in `validateRunAction`)

| # | Rule | Severity on fail | Notes |
|---|---|---|---|
| 1 | `source_date` parses to a valid date | error | Excel serial out of 1900–2100 range is invalid |
| 2 | `source_date > '2025-12-31'` | (special) | Mark row `status='skipped_post_cutover'`; no further checks |
| 3 | `provider_id_resolved IS NOT NULL` | error | Maps via `hmo_provider_aliases` or exact match |
| 4 | `service_id_resolved IS NOT NULL` | error | Same |
| 5 | `parseDecimal(billed_amount) > 0` | error | Covers blank, zero, negative |
| 6 | If `payment_received_date IS NOT NULL`: `parseDecimal(paid_amount) ≤ parseDecimal(billed_amount)` | error | `amounts_within_billed` CHECK would fail otherwise |
| 7 | If `source_tab='LAB SERVICE'` and `E='YES'` and `O='CASH'` is non-blank | warning | Cash payment recorded on an HMO row — possibly miscategorized |
| 8 | `reference_no` IS NULL AND `submission_date` IS NULL | (info) | Becomes an "unbilled" item; no error |

### 9.2 Cross-row rules

| # | Rule | Severity |
|---|---|---|
| A | Same `or_number` under two distinct `provider_id_resolved` | warning |
| B | Same `normalized_patient_name` → multiple distinct DOB hints in surrounding columns | (none — workbook has no DOB) |
| C | Two staging rows with identical `content_hash` within the same run | error |

### 9.3 Reconciliation rule

Per provider, compute:
- `wb_ending = HMO REFERENCE 'ENDING BALANCE' column at cutover month`
- `staged_ar = SUM(billed - paid - patient_billed - written_off)` across rows whose `provider_id_resolved = this provider`
- `variance_pct = (staged_ar - wb_ending) / NULLIF(wb_ending, 0)`

Severity color: green ≤ 1%, yellow 1–5%, red > 5%. Red blocks commit unless `variance_override_reason` is supplied.

If HMO REFERENCE doesn't contain a clear ending-balance row for a provider, that provider's row reads "no reference data — skipping reconciliation gate" (info-level only).

The HMO REFERENCE tab's exact row/column layout for the per-month aging block is finalized during D2 implementation against a fresh read of the workbook. The parser handles the case where the tab structure has shifted (header text drift) the same way as the transaction tabs: assert the canonical header set at row 1, fail with a clear message if the workbook layout has changed.

### 9.4 Commit-time invariants (DB-enforced)

These are guarantees, not validations — the schema CHECK constraints and triggers will reject violators:
- `hmo_claim_items.billed_amount_php > 0`
- `paid_amount_php + patient_billed_amount_php + written_off_amount_php ≤ billed_amount_php`
- Posted JEs in 12.A must balance (P0001).
- Posting date 2025-12-31 must be in an open period at commit time (admin keeps 2025 open through commit; closes via existing 12.1 UI afterwards).

## 10. Operational notes

- **Run during off-hours.** The commit takes an `pg_advisory_xact_lock`; concurrent users hitting the same op tables (patients / visits / test_requests) will block until commit completes. For ~27K rows we estimate < 60s on the linked Supabase project. Admin sees a "running…" spinner and a "Cancel" link that aborts the transaction.
- **PII acknowledgment.** A checkbox in the commit footer reads: "I confirm we hold these patient names as part of our legitimate business records under RA 10173 and have a legal basis to retain them after import." Required to enable Commit.
- **Memory + work area.** `hmo_history_staging` holds at most ~30K rows per active run. Discarded runs keep their rows for audit purposes; manual cleanup if it ever grows.
- **JE counter for 2025.** `je_year_counters.next_n` for 2025 is currently 1. The opening JEs will be `JE-2025-0001` through `JE-2025-{N≤10}`. 12.B's later historical imports will continue from N+1.

## 15. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Patient false-merge (two real people same name) | Affects drill-down only; AR balance unaffected. Documented in preview. |
| Workbook header drift (someone renames a column) | Parser asserts headers at row 1; fails fast with clear message. |
| Workbook has Excel-only formulas with #REF! | Parser reads values, not formulas; #REF! values become NULL and trip row-1 / row-2 validations. |
| Reconciliation variance > 5% with no obvious cause | Block commit; require override reason captured on the run for audit. |
| Bridge bypass leaks (skip flag survives commit) | `SET LOCAL` is transaction-scoped; unset explicitly before posting opening JEs as belt-and-braces. |
| Concurrent imports | `pg_advisory_xact_lock` at the start of `commit_hmo_history_run`. |
| Service-role client used for bypass-flag-setting from a place that ships to the browser | All four Server Actions are `'use server'`; the admin client never enters a client component. |
| Reception sees thousands of historical patients | `is_historical=false` filter on the four list pages; partial indexes back the filter. |
| Re-importing changes existing patients | Patient upsert uses `ON CONFLICT (full_name) WHERE is_historical = true DO NOTHING` — never updates existing operational patients. |
| HMO REFERENCE tab missing or malformed | Reconciliation panel reports "no reference data"; admin commits with informational note. |
| The Drive workbook keeps changing under us | Admin uploads a frozen snapshot; `file_hash` records which version was imported. |
