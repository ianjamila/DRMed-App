# 12.C — Daily Cash Reconciliation (EOD close)

**Status:** Draft 2026-05-18. Awaiting user review.
**Parent project:** [`2026-05-11-accounting-project-overview.md`](./2026-05-11-accounting-project-overview.md)
**Depends on:** 12.1 (GL foundation, `v1.2.0`), 12.2 (op→GL bridge, `v1.3.0`), 12.A (HMO history, `v1.5.0`). No dependency on 12.3 / 12.4 / 12.5.
**Migration ID(s):** `0043_eod_cash_reconciliation.sql` (single migration; may add `0044_*` for post-D1 fixes per the 12.A/12.3 pattern).
**Effort estimate:** ~5 working days · 1 week calendar.

## 1. Goal

Give reception a daily place to (a) record cash leaving the drawer during the day, (b) count the physical till at end of day, (c) lock the day's cash story. Wire every cash movement to the GL so the books reflect operational reality without a separate journal-entry step. Give admin two read-only sidecars: a service × day revenue pivot and an outstanding-staff-advances list.

After 12.C ships:

- Every cash payout (`petty_cash`, `salary_advance`, `courier`, `other_payout`) and every float adjustment (`float_topup`, `float_pullout`) fires a balanced JE via Postgres triggers.
- Every EOD close with non-zero variance fires a "cash short / over" JE.
- Reception can't back-date payments into a locked date without admin reopening the close (audit-logged).
- Staff cash advances accumulate in a `staff_advances` receivable subledger that 12.6 (payroll) can later draw down via payslip deductions.

**Standard defaults (locked in brainstorm wrap-up):**

| ID | Decision |
|---|---|
| A | Variance threshold: **any non-zero variance requires a reason note.** No tolerance amount. |
| B | Method coverage: **v1 physically counts cash only.** GCash / Maybank / BPI / card balances are *displayed* on the cash-drawer page for situational awareness, but not reconciled. A future 12.D will own bank/wallet reconciliation. |
| C | Reopen-after-close: **closed records are DB-locked** (trigger blocks INSERT/UPDATE/DELETE on `payments`, `eod_cash_adjustments` targeting a closed `(business_date, shift_id)`). Admin reopens via `reopenEodCloseAction`; reopen flips `eod_close_records.status` to `'reopened'`, writes an audit row, and re-enables operational writes for that date. |

## 3. Scope

**In:**

- Five new tables + one settings row + two new CoA codes + two new `je_source_kind` enum values.
- Reception UI: `/staff/payments/cash-drawer` (read+write during the day), `/staff/payments/eod` (close).
- Admin UI: `/staff/admin/accounting/cash-routing` (map kinds → contra accounts, same pattern as `payment-routing`), `/staff/admin/reports/daily-revenue` (service × day pivot, read-only), `/staff/admin/reports/staff-advances` (outstanding balance per staff, read-only).
- GL bridge triggers on `eod_cash_adjustments` (INSERT / soft-void UPDATE) and `eod_close_records` (status flip to `closed`).
- Period-lock-aware close: a close-day attempt rejects if the business date sits in a CLOSED accounting period (P0002 surface).
- Audit-log entries on every write surface (create, void, close, reopen, mapping update, advance creation).
- Server Actions with Zod validation, all returning `{ ok: true, data } | { ok: false, error }`.
- New pg-error codes P0015–P0019 with `translatePgError` entries.
- Smoke test covering: opening float math, payout JE posting, salary advance receivable creation, close + variance JE, reopen path, period-lock interaction, idempotency.

**Out (deferred to later sub-projects):**

- Bank deposit / wallet reconciliation for non-cash methods → future `12.D Bank reconciliation`.
- Petty cash voucher PDF / printed receipt for payouts → future polish.
- Staff advance write-off UI → 12.6 ships this when payroll is built.
- Multi-shift handover-count workflow → schema already supports it; UI extension is post-v1.
- Bulk-edit / bulk-void of cash adjustments → unscoped; admin can void individually.
- Cash float movement between two same-day shifts (handover) → schema supports via two close rows for the date, but no UI for v1.

**Key properties (mirror the 12.2 invariants):**

- One operational event → exactly one posted JE (idempotency via the partial unique index from migration 0030 — predicate already permits 'cash_adjustment' and 'eod_close').
- Triggers fire inside the operational transaction; JE failure rolls back the operational write atomically.
- Reception's existing payment UI is untouched. The `payments` table is read by 12.C but not modified.
- Admin sidecars are pure-read views on top of existing tables — no analytics tables, no cron jobs.

### 5.3 `cash_adjustment_account_map` (new)

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `kind` | text UNIQUE NOT NULL CHECK (`kind in ('petty_cash','salary_advance','courier','other_payout','float_topup','float_pullout')`) | |
| `account_id` | uuid FK → `chart_of_accounts(id)` NOT NULL | The non-cash side of the JE. |
| `requires_user_choice` | boolean NOT NULL default `false` | When true, UI forces the user to pick a contra account at row creation; bridge expects `eod_cash_adjustments.contra_account_id` to be non-null. |
| `notes` | text | |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | `touch_updated_at` trigger. |

**Seed:**

| kind | account_id (code) | requires_user_choice | notes |
|---|---|---|---|
| `petty_cash` | `6400` (Office Supplies) | `true` | User picks the actual expense account; default if skipped. |
| `salary_advance` | `1130` (Staff Advances) | `false` | Always 1130; not user-overridable in v1. |
| `courier` | `6320` (Courier) | `false` | Dedicated courier expense account seeded in 0043. |
| `other_payout` | `9999` (Suspense) | `true` | User must pick; falls back to Suspense and audits. |

**RLS:** admin read+write only. (Reception writes go through Server Actions that read the map via the service-role admin client.)

### 5.4 `eod_cash_adjustments` (new, the main daily-flow table)

One row per movement of cash *out of* (or *into*, for `float_topup`) the till that isn't a `payments` row.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `business_date` | date NOT NULL | Manila-tz date when the cash moved. |
| `shift_id` | uuid FK → `cash_shifts(id)` NOT NULL | Defaults to the active shift if only one is active. |
| `kind` | text NOT NULL CHECK (matches `cash_adjustment_account_map.kind`) | |
| `amount_php` | numeric(14,2) NOT NULL CHECK (`amount_php > 0`) | Always positive; direction is implied by `kind`. |
| `payee` | text | Free-text label (vendor name, courier name, etc.). |
| `payee_staff_id` | uuid FK → `staff_profiles(id)` | Required iff `kind = 'salary_advance'`. CHECK enforces this. |
| `contra_account_id` | uuid FK → `chart_of_accounts(id)` | Required iff `cash_adjustment_account_map.requires_user_choice = true` for this kind. Falls back to mapped default + Suspense audit if NULL. |
| `notes` | text | |
| `recorded_by` | uuid FK → `staff_profiles(id)` NOT NULL | |
| `recorded_at` | timestamptz NOT NULL default `now()` | |
| `voided_at` | timestamptz | Soft-void, same pattern as 12.2 `payments`. |
| `voided_by` | uuid FK → `staff_profiles(id)` | |
| `void_reason` | text | |

**Indexes:**

- `idx_eod_cash_adjustments_date` on `(business_date, shift_id)`.
- `idx_eod_cash_adjustments_kind` on `(kind)`.
- `idx_eod_cash_adjustments_active` on `(business_date, shift_id) WHERE voided_at IS NULL` (drives the cash-drawer page's running totals).
- `idx_eod_cash_adjustments_staff` on `(payee_staff_id) WHERE payee_staff_id IS NOT NULL`.

**Constraints:**

- CHECK `(kind != 'salary_advance' OR payee_staff_id IS NOT NULL)`.
- CHECK `(amount_php > 0)`.
- CHECK `(voided_at IS NULL) = (voided_by IS NULL)`.

**RLS:** reception + admin read; reception + admin write (where the day isn't locked — trigger enforces).

### 5.5 `eod_close_records` (new)

One row per `(business_date, shift_id)` per close attempt. Most rows are `status='closed'`; admin reopens flip them to `'reopened'` and a *new* close row gets written on re-close.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `business_date` | date NOT NULL | |
| `shift_id` | uuid FK → `cash_shifts(id)` NOT NULL | |
| `status` | text NOT NULL CHECK (`status in ('closed','reopened')`) default `'closed'` | |
| `opening_float_php` | numeric(14,2) NOT NULL | Snapshot of effective opening (baseline + sum of float adjustments) at close time. |
| `cash_payments_php` | numeric(14,2) NOT NULL | Snapshot of `payments` total for the date where `method='cash'` and `voided_at IS NULL`. |
| `cash_payouts_php` | numeric(14,2) NOT NULL | Snapshot of `eod_cash_adjustments` total for the date where `kind in ('petty_cash','salary_advance','courier','other_payout')` and `voided_at IS NULL`. |
| `expected_cash_php` | numeric(14,2) NOT NULL | `opening_float + cash_payments_php - cash_payouts_php`. Computed at close time. |
| `counted_cash_php` | numeric(14,2) NOT NULL CHECK (`counted_cash_php >= 0`) | |
| `variance_php` | numeric(14,2) NOT NULL | `counted - expected`. Negative = short, positive = over. |
| `variance_reason` | text | NOT NULL when `variance_php != 0` (enforced via CHECK). |
| `closed_by` | uuid FK → `staff_profiles(id)` NOT NULL | Whichever receptionist clicked "Close day". |
| `closed_at` | timestamptz NOT NULL default `now()` | |
| `reopened_by` | uuid FK → `staff_profiles(id)` | Populated when admin reopens. |
| `reopened_at` | timestamptz | |
| `reopen_reason` | text | NOT NULL when `status='reopened'` (CHECK). |
| `created_at` | timestamptz default `now()` | |

**Indexes:**

- UNIQUE `(business_date, shift_id) WHERE status = 'closed'` — one *active* close per date per shift. Reopened rows linger for audit history.
- `idx_eod_close_records_date` on `(business_date DESC, shift_id)`.

**Constraints:**

- CHECK `(variance_php = counted_cash_php - expected_cash_php)`.
- CHECK `(variance_php = 0 OR variance_reason IS NOT NULL)`.
- CHECK `(status != 'reopened' OR reopen_reason IS NOT NULL)`.

**RLS:** reception + admin read; reception writes via `closeEodAction` only (RLS allows INSERT, audit'd); admin writes via `reopenEodCloseAction` only.

### 5.6 `staff_advances` (new, receivable subledger)

Mirrors `salary_advance` rows from `eod_cash_adjustments` so that 12.6 (payroll) can iterate outstanding balances. Maintained by trigger `staff_advance_sync` — never manually inserted by application code in 12.C.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `staff_id` | uuid FK → `staff_profiles(id)` NOT NULL | |
| `source_adjustment_id` | uuid FK → `eod_cash_adjustments(id)` UNIQUE NOT NULL | One advance row per cash payout. UNIQUE for idempotency. |
| `business_date` | date NOT NULL | Mirrors `eod_cash_adjustments.business_date`. |
| `original_amount_php` | numeric(14,2) NOT NULL CHECK (`original_amount_php > 0`) | |
| `outstanding_balance_php` | numeric(14,2) NOT NULL CHECK (`outstanding_balance_php >= 0`) | Starts at `original_amount_php`. 12.6 will UPDATE this down. Voiding the source adjustment sets to 0 (and posts the reversal JE). |
| `status` | text NOT NULL CHECK (`status in ('outstanding','settled','voided','written_off')`) default `'outstanding'` | |
| `notes` | text | |
| `created_at` | timestamptz NOT NULL default `now()` | |
| `updated_at` | timestamptz NOT NULL default `now()` | |

**Indexes:**

- UNIQUE `(source_adjustment_id)`.
- `idx_staff_advances_staff_outstanding` on `(staff_id) WHERE status='outstanding'`.

**RLS:** admin read+write; reception read (for `/staff/payments/cash-drawer` to show "this employee owes ₱X" hint when picking a payee for `salary_advance`).

### 5.9 Views and helpers

**`v_daily_revenue_by_service`** — drives `/staff/admin/reports/daily-revenue`. Read-only.

```sql
create view public.v_daily_revenue_by_service as
select
  (tr.released_at at time zone 'Asia/Manila')::date as business_date,
  s.id   as service_id,
  s.code as service_code,
  s.name as service_name,
  s.kind as service_kind,
  count(*)                                       as released_count,
  coalesce(sum(tr.final_price_php), 0)::numeric(14,2) as revenue_php,
  coalesce(sum(tr.discount_amount_php), 0)::numeric(14,2) as discount_php
from public.test_requests tr
join public.services s on s.id = tr.service_id
where tr.status = 'released'
group by business_date, s.id, s.code, s.name, s.kind;
```

**`v_staff_advances_outstanding`** — drives `/staff/admin/reports/staff-advances`. Read-only.

```sql
create view public.v_staff_advances_outstanding as
select
  sa.staff_id,
  sp.full_name,
  sp.role,
  count(*) filter (where sa.status = 'outstanding')                  as advance_count,
  coalesce(sum(sa.outstanding_balance_php), 0)::numeric(14,2)         as outstanding_php,
  min(sa.business_date) filter (where sa.status = 'outstanding')      as oldest_advance_date
from public.staff_advances sa
join public.staff_profiles sp on sp.id = sa.staff_id
group by sa.staff_id, sp.full_name, sp.role;
```

**`cash_drawer_state(p_business_date, p_shift_id)`** — SECURITY DEFINER function returning a single jsonb document. Drives `/staff/payments/cash-drawer` running totals; no live recompute in the client. (Function, not view, because it parameterises on date + shift.)

```sql
create or replace function public.cash_drawer_state(
  p_business_date date,
  p_shift_id      uuid
)
returns jsonb
language sql
stable
security definer
set search_path = public
as $$
  with
    baseline as (
      select coalesce(value_php, 0)::numeric(14,2) as v
        from public.accounting_settings
        where key = 'default_change_fund_php'
    ),
    floats as (
      select
        coalesce(sum(case when kind = 'float_topup'   then amount_php else 0 end), 0)::numeric(14,2) as topups,
        coalesce(sum(case when kind = 'float_pullout' then amount_php else 0 end), 0)::numeric(14,2) as pullouts
      from public.eod_cash_adjustments
      where business_date = p_business_date
        and shift_id      = p_shift_id
        and voided_at is null
    ),
    cash_in as (
      select coalesce(sum(amount_php), 0)::numeric(14,2) as v
        from public.payments p
        join public.visits v on v.id = p.visit_id
        where (p.received_at at time zone 'Asia/Manila')::date = p_business_date
          and p.method = 'cash'
          and p.voided_at is null
    ),
    payouts as (
      select coalesce(sum(amount_php), 0)::numeric(14,2) as v
        from public.eod_cash_adjustments
        where business_date = p_business_date
          and shift_id      = p_shift_id
          and kind in ('petty_cash','salary_advance','courier','other_payout')
          and voided_at is null
    ),
    closed as (
      select id, closed_at
        from public.eod_close_records
        where business_date = p_business_date
          and shift_id      = p_shift_id
          and status        = 'closed'
        limit 1
    )
  select jsonb_build_object(
    'baseline_float_php', baseline.v,
    'float_topups_php',   floats.topups,
    'float_pullouts_php', floats.pullouts,
    'opening_float_php',  (baseline.v + floats.topups - floats.pullouts),
    'cash_payments_php',  cash_in.v,
    'cash_payouts_php',   payouts.v,
    'expected_cash_php',  (baseline.v + floats.topups - floats.pullouts + cash_in.v - payouts.v),
    'closed_id',          (select id from closed),
    'closed_at',          (select closed_at from closed)
  )
  from baseline, floats, cash_in, payouts;
$$;
```

## 6. GL bridge — trigger specifications

### 6.1 `bridge_cash_adjustment_insert()`

Fires AFTER INSERT on `eod_cash_adjustments`. Constructs the JE.

**JE shape (by kind):**

| kind | Debit | Credit |
|---|---|---|
| `petty_cash` | `contra_account_id` (or `cash_adjustment_account_map.account_id` for the kind) | `1010` Cash on Hand |
| `salary_advance` | `1130` Staff Advances (fixed; map's `account_id`) | `1010` Cash on Hand |
| `courier` | `contra_account_id` (or mapped default) | `1010` Cash on Hand |
| `other_payout` | `contra_account_id` (or 9999 Suspense + audit) | `1010` Cash on Hand |
| `float_topup` | `1010` Cash on Hand | `contra_account_id` (e.g. 1020) |
| `float_pullout` | `contra_account_id` (e.g. 1020) | `1010` Cash on Hand |

Pattern (lifted from `bridge_payment_insert`):

1. SELECT idempotency check: skip if a posted JE already exists for `('cash_adjustment', NEW.id)`.
2. Resolve contra account:
   - If `NEW.contra_account_id IS NOT NULL`, use it.
   - Else look up `cash_adjustment_account_map.account_id` for `NEW.kind`. If `requires_user_choice = true` and we got here without a chosen account, fall back to `9999` and emit a `coa.suspense_post` audit row.
3. Insert `journal_entries` as `'draft'`, `source_kind='cash_adjustment'`, `source_id=NEW.id`, `posting_date=NEW.business_date`, `created_by=NEW.recorded_by`.
4. Insert two `journal_lines` (debit/credit per the kind matrix).
5. UPDATE the JE to `'posted'` (validates balance).

### 6.2 `bridge_cash_adjustment_void()`

Fires AFTER UPDATE on `eod_cash_adjustments` WHEN `OLD.voided_at IS NULL AND NEW.voided_at IS NOT NULL`. Same reversal pattern as `bridge_payment_void`:

1. Find original posted JE.
2. Insert reversal JE as `'draft'`, `source_kind='reversal'`, `reverses=<original>`, `created_by=NEW.voided_by`.
3. Mirror lines with debit/credit swapped.
4. UPDATE reversal to `'posted'`.
5. UPDATE original to `'reversed'`, `reversed_by=<reversal>`.

### 6.3 `bridge_eod_close()`

Fires AFTER INSERT on `eod_close_records` WHEN `NEW.status='closed' AND NEW.variance_php != 0`.

JE shape (cash short / over):

- `variance_php < 0` (counted < expected → till is short):
  - DR `6900` Cash Short / Over (abs(variance))
  - CR `1010` Cash on Hand (abs(variance))
- `variance_php > 0` (counted > expected → till is over):
  - DR `1010` Cash on Hand (variance)
  - CR `6900` Cash Short / Over (variance)

Posting date = `NEW.business_date`. `source_kind='eod_close'`, `source_id=NEW.id`.

### 6.4 `staff_advance_sync()`

Fires AFTER INSERT and AFTER UPDATE on `eod_cash_adjustments`:

- INSERT with `kind='salary_advance'`:
  - INSERT into `staff_advances` (`staff_id=NEW.payee_staff_id`, `source_adjustment_id=NEW.id`, `business_date=NEW.business_date`, `original_amount_php=NEW.amount_php`, `outstanding_balance_php=NEW.amount_php`, `status='outstanding'`).
- UPDATE flipping `voided_at` NULL → NOT NULL with `kind='salary_advance'`:
  - UPDATE `staff_advances` SET `outstanding_balance_php=0`, `status='voided'`, `updated_at=now()` WHERE `source_adjustment_id=NEW.id`.

### 6.5 Guard triggers

**P0015 — EOD lock (the keystone trigger).** Fires BEFORE INSERT/UPDATE/DELETE on `payments` AND `eod_cash_adjustments`. If the target row's date+shift has an *active* close (`eod_close_records.status='closed'`), reject:

```
EOD already closed for business_date <date> (shift <code>).
Ask an admin to reopen the close before recording further activity.
```

Implementation note: payment-side, the trigger infers `business_date` as `(NEW.received_at at time zone 'Asia/Manila')::date`. Shift is resolved via `SELECT id FROM cash_shifts WHERE is_active=true ORDER BY sort_order LIMIT 1` — correct for v1 (exactly one active shift). When multi-shift UI ships, `payments` will need its own `shift_id` column and the resolver will read it from `NEW.shift_id`; deferred to that work.

**P0017 — adjustment immutable after JE.** Fires BEFORE UPDATE on `eod_cash_adjustments`. If a posted JE exists for the row, reject mutations to `amount_php`, `kind`, `business_date`, `shift_id`, `payee_staff_id`, `contra_account_id`. Soft-void columns + `notes` remain editable.

**P0018 — staff advance overdraw.** Fires BEFORE UPDATE on `staff_advances`. Rejects `outstanding_balance_php < 0`. (Belt-and-suspenders; the column CHECK already enforces this, but the trigger emits a friendlier message for the 12.6 caller.)

**P0019 — cash account map references inactive account.** Fires BEFORE INSERT/UPDATE on `cash_adjustment_account_map`. Rejects if `account_id` points at `chart_of_accounts.is_active=false`.

### 6.6 Period interaction

The existing **P0002** period-lock trigger fires for free — if a user tries to record an adjustment dated in a closed accounting period, the bridge's JE insert trips P0002 and rolls the whole operational write back. The Server Action surfaces the canonical "That accounting period is closed." message from `translatePgError`.

## 7. RLS policies

**Reception + admin read-write** (subject to P0015 lock):

- `eod_cash_adjustments` — INSERT/UPDATE/DELETE require `has_role(array['reception','admin'])`.
- `eod_close_records` — INSERT requires `has_role(array['reception','admin'])`. UPDATE (reopen) requires `has_role(array['admin'])`.

**Admin-only:**

- `cash_shifts`, `accounting_settings`, `cash_adjustment_account_map` — full CRUD.
- `staff_advances` — read-write (only triggers actually INSERT here in v1; admin UPDATE will arrive in 12.6).

**Reception read-only:**

- `staff_advances` — SELECT only, for the cash-drawer "this staff member owes ₱X" hint.

**Audit-log:** standard pattern — all writes go through the service-role admin client which writes the audit row in the same transaction.

One schema per action under `src/lib/accounting/cash/schemas.ts`. Date validation uses `src/lib/dates/manila.ts` helpers (per the 12.3 follow-up FU-A — captured in `project_accounting_phase12_brainstorm.md`).

Specifically `recordCashAdjustmentAction` schema enforces:

- `business_date <= todayManilaISODate()`.
- `kind in (cash_adjustment_account_map.kind)` enum.

## 11. Error handling

New entries in `src/lib/accounting/pg-errors.ts`:

```ts
case "P0015":
  return err.message ?? "End of day is already closed for that date. Ask an admin to reopen first.";
case "P0017":
  return err.message ?? "Cannot edit this cash adjustment after its journal entry has posted. Void and re-create instead.";
case "P0018":
  return err.message ?? "Staff advance cannot go below zero.";
case "P0019":
  return err.message ?? "That account is inactive. Pick a different one.";
```

P0016 is reserved (we use a Zod refinement client-side instead — the CHECK constraint `variance_php = 0 OR variance_reason IS NOT NULL` doubles as a safety net but should never surface to the user).

## 15. Hard invariants 12.C must not violate

Carried forward from 12.1 / 12.2 / 12.3 / 12.A (all live):

1. Debits = Credits on every posted JE.
2. No JE posts to a closed accounting period.
3. One operational event → exactly one JE.
4. CoA is append-only (codes immutable; soft-disable via `is_active=false`).
5. Reception's existing UI is untouched. (12.C *adds* reception routes; it does NOT modify `/staff/visits` or any payment-recording surface.)
6. RLS is the source of truth for access. No `createAdminClient()` bypasses in code outside Server Actions / triggers.
7. Audit-log every write.
